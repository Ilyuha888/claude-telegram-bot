# Solutions for: Telegram attachments (images, PDFs, documents) not accessible to agent

Problem: prob-20260420-006

## Variants (3)

### V_content_blocks. SDK content blocks — inline image/PDF bytes in query prompt

Change photo.ts to base64-encode JPEG and pass {type:'image', source:{type:'base64',...}} content blocks. Change document.ts PDF branch to pass {type:'document', source:{type:'base64', media_type:'application/pdf',...}} blocks. Widen session.ts sendMessageStreaming to accept string | ContentBlock[] and use AsyncIterable<SDKUserMessage> path when blocks provided. SDK query() bundle confirms: session_id='' is correct, streamInput handles endInput after generator exhausts.

**Strengths:**
- Eliminates hallucination root cause (no model choice required)
- Drops poppler-utils dependency
- Fixes both PDFs and images in one change

**Novelty:** Removes the Read-tool round-trip entirely for both images and PDFs; model receives bytes directly

**Weakest link:** SDK AsyncIterable path has never been exercised in this codebase — behavior with canUseTool/permission-keyboard flow on isSingleUserTurn=false is unverified

**Risks:**
- isSingleUserTurn=false changes stdin-close behavior — requires testing
- Large base64 payload increases subprocess stdin memory pressure

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Once content-block path works, multi-modal documents (docx, xlsx previews) and send_file flows can use the same pipeline

### V_path_fixup. Install poppler + explicit Read-tool prompt directive

apt-get install poppler-utils on VM (fixes PDF extraction). Change photo prompt from '[Photo: /path]' to 'Use your Read tool to view the image at /path, then describe it.' Change SAFETY_PROMPT to add 'When sent an image, ALWAYS call the Read tool first.' Fix the brew→apt error message in document.ts.

**Strengths:**
- Zero SDK interface changes
- Simpler to test
- Immediate poppler fix unblocks PDFs today

**Novelty:** Preserves existing string-prompt architecture; only changes poppler install and prompt wording

**Weakest link:** Model still chooses whether to call Read — explicit prompt reduces but does not eliminate hallucination risk

**Risks:**
- Model hallucination still possible (less likely but not eliminated)
- poppler is a new system dependency

### V_hybrid. Content blocks for images, pdftotext for PDFs

Use SDK content blocks for images (vision quality fix). Keep pdftotext for PDFs (install poppler). This splits the risk: image blocks are well-tested in multimodal API; PDF blocks require document API support verification.

**Strengths:**
- Image hallucination eliminated
- PDF extraction familiar and debuggable

**Novelty:** De-risks by limiting content-block path to images only where vision quality is the clear bottleneck

**Weakest link:** PDF handling remains fragile (poppler system dep, extraction quality degrades for scanned PDFs)

**Risks:**
- Two different approaches to maintain
- poppler still a dep
