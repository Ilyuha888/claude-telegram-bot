# V1 — Full asyncio end-to-end

## 1. Problem Frame

**Signal:** ADR-002 specifies Python 3.12+ but is silent on sync vs async execution model. Every service boundary (Telegram handler, vault writer, scheduler, LLM client) depends on this choice. The current architecture diagrams in ARCHITECTURE.md mix patterns without committing.

**Constraints:**
- Must not introduce shell execution or subprocesses into the runtime LLM path (ADR-005)
- Must be compatible with chosen Telegram library (ADR-003)
- Must support single-user MVP scope without over-engineering for concurrency

## 2. Decision

**Selected:** V1 — Full asyncio end-to-end

PTB v22+ forces asyncio at the entry point — V1 is the only variant that extends this naturally rather than fighting it. One uniform concurrency pattern across all layers minimises cognitive overhead for a single-developer MVP. Dulwich is the sole sync library; run_in_executor wrapping is the only bridge needed, and it is explicit and localized. V1 dominates V2 on pattern consistency and cognitive overhead, and dominates V3 on all four optimizable dimensions.

**Invariants:**
- Telegram gateway is async (forced by PTB v22+). All handlers must be async def coroutines.
- Dulwich (sync) must always be called via asyncio.run_in_executor — never called directly from a coroutine.
- asyncio.run() inside a coroutine is forbidden.
- The DB driver must be async (psycopg3 async or asyncpg).

**Admissibility:**
- NOT: Blocking calls in the event loop without executor delegation
- NOT: asyncio.run() inside a coroutine
- NOT: Mixing thread-pool dispatch with asyncio handlers
- NOT: Introducing a sync HTTP client in the LLM call path

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| V1 — Full asyncio end-to-end | **Selected** | PTB v22+ forces asyncio at the entry point — V1 is the ... |
| V2 — Async edge, sync core | Rejected | Requires run_in_executor discipline at every handler boundary (same risk as V1's Dulwich seam) while adding a permanent second mental model and making testing harder. No meaningful benefit over V1 for this stack. |
| V3 — Thread-pool dispatch (sync-first) | Rejected | Trades async complexity for thread complexity that is harder to manage: exception marshalling across the thread/loop boundary, cancellation semantics, and thread-safety requirements even for a single-user MVP. Highest reversibility cost of the three options. |

**Weakest link:** Correct and consistent run_in_executor usage for every Dulwich call; a direct sync call from a coroutine blocks the event loop silently and is not caught at runtime

## 4. Consequences

**Refresh triggers:**
- Dulwich adds native async support
- A second concurrent user is added to the platform
- DB driver choice changes away from psycopg3/asyncpg
- PTB major version upgrade changes concurrency model

**Affected files:** ARCHITECTURE.md, docs/adr/ADR-002-python-version.md
