# V5 — Hybrid: slash-prefix router + /menu inline keyboard

## 1. Problem Frame

**Signal:** Current Telegram bot has one interaction mode — all user input enters the conversational LLM context regardless of whether the user intends knowledge work (Mode 1) or infrastructure management (Mode 2: spawning remote sessions, worktrees, repo navigation). No predictable surface cue distinguishes the two. Infrastructure commands pollute the knowledge-work context, and Mode-2 invocations require knowledge of hidden syntax. Directly related to prob-20260417-006 (skill invocation UX invisibility) and the vision captured in Agent_Obsidian_Vault/notes/2026-04-20-dual-mode-telegram-vision.md.

**Constraints:**
- Preserve Obsidian Sync mobile access (no vault path changes)
- Subscription-billing compatible — no API-token model (dec-20260416-003)
- Single-user scope (dec-20260320-001)
- No terminal passthrough from Telegram to VM shell
- Mode-2 must be stateless (menu only, no LLM reasoning)

**Acceptance:** User can predict active mode from bot surface cues; Mode-2 menu commands never produce LLM reasoning responses; mobile-to-desktop handoff preserves session context; pilot test with 10 diverse requests shows correct mode routing with zero bleed between modes.

## 2. Decision

**Selected:** V5 — Hybrid: slash-prefix router + /menu inline keyboard

**Selection policy:** Prefer minimum implementation effort among variants achieving mode_separation=3; break ties on ux_clarity. Single-user system — operational simplicity outweighs enterprise isolation.

**Why selected:** V5 achieves mode_separation=3 (same as V3) at 2 days effort vs V3's 4 days and second-bot operational overhead. /menu reuses existing askq: inline-keyboard callback infrastructure from dec-20260420-003. No second BOT_TOKEN, no second systemd service. V1 was dominated by V5 on ux_clarity (2 vs 3) at same separation — eliminated.


**Invariants:**
- Mode-2 slash commands (/work, /sessions, /attach, /close, /repos) MUST be intercepted BEFORE sendMessageStreaming() is called
- LLM MUST never receive Mode-2 command text as conversational input
- Menu inline keyboard buttons MUST use callback_data ≤64 bytes
- /menu MUST use InlineKeyboardMarkup (ephemeral), never ReplyKeyboardMarkup

**Pre-conditions:**
- [ ] BotFather /setcommands registered for /work, /sessions, /attach, /close, /repos, /menu
- [ ] Existing callbackQuery handler supports 'menu:' prefix in addition to 'askq:' prefix
- [ ] linuz90 bot version supports bot.callbackQuery()

**Post-conditions:**
- [ ] 10-request pilot passes: 5 Mode-1 KB queries (LLM responds), 5 Mode-2 commands (deterministic, 0 LLM tokens)
- [ ] journalctl shows zero sendMessageStreaming() calls triggered by /work or /sessions
- [ ] BotFather slash-autocomplete shows /work, /sessions, /attach, /close, /repos, /menu
- [ ] /menu responds with InlineKeyboardMarkup within 2s

**Admissibility:**
- NOT: Router MUST NOT call sendMessageStreaming() for Mode-2 commands under any code path
- NOT: Mode-2 response MUST NOT include LLM-generated text
- NOT: callback_data prefix 'menu:' MUST NOT collide with 'askq:' namespace

## 3. Rationale

**Counterargument:** V3 (dual-bot) provides absolute transport-level separation with zero LLM bleed risk. V5's router must be correct — a bug that leaks a /work argument into sendMessageStreaming() would route an infrastructure command to the LLM.

**Selected variant weakest link:** Router correctness — if /work or /sessions match the Mode-1 dispatch path, the LLM sees infrastructure commands as conversation messages.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| V5 — Hybrid: slash-prefix router + /menu inline keyboard | **Selected** | V5 achieves mode_separation=3 (same as V3) at 2 days effo... |
| V1 — Slash-prefix router (text-only) | Rejected | Dominated by V5 on ux_clarity (2 vs 3) at same mode_separation and negligible additional effort difference |
| V3 — Dual bot handle | Rejected | Achieves same mode_separation=3 at 4 days effort (2x V5), requires dec-20260416-003 refresh, doubles operational load (2nd BOT_TOKEN, 2nd systemd service). Overkill for single-user system. |

**Evidence requirements:**
- journalctl pilot log showing 10 commands processed, 0 SDK queries on Mode-2 path
- Telegram screenshot of /menu inline keyboard rendering
- BotFather /setcommands confirmation screenshot

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| Zero LLM tokens spent on Mode-2 commands | journalctl for claude-telegram-bot.service: no SDK query() call logged when /work, /sessions, /attach, /close, /repos are sent | 0 LLM calls in 10-command pilot |
| /menu inline keyboard renders correctly in Telegram iOS | Screenshot or user confirmation of inline buttons visible below bot message | All 4 buttons visible and tappable |
| Router adds <100ms latency to Mode-1 messages | Time from Telegram message receipt to sendMessageStreaming() call start (journalctl timestamps) | <100ms overhead compared to pre-router baseline |

## 4. Consequences

**Rollback plan:**
Triggers:
- Router bug routes Mode-2 commands to LLM
- Inline keyboard callback_data collisions with askq: prefix
- BotFather /setcommands registration fails persistently
Steps:
1. Revert bot to pre-V5 version via git revert on VM
2. Remove /setcommands registration via BotFather
3. No data migration needed — no persistent state added by V5
Blast radius: linuz90 bot command layer only; no VM state, no vault, no session store affected

**Refresh triggers:**
- Telegram Bot API deprecates callback_data or InlineKeyboardMarkup
- linuz90 upstream changes callbackQuery handler interface
- Mode-2 command volume exceeds 20% of total — revisit dual-bot option
- User reports /work or /sessions being routed to LLM (router bug)

**Affected files:** /home/assistant/claude-telegram-bot/src/bot.ts, /home/assistant/claude-telegram-bot/src/handlers/mode2.ts
