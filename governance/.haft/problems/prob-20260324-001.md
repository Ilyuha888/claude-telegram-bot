# No structured output strategy for orchestrator LLM calls

## Signal

The orchestrator architecture specifies typed tool contracts (Pydantic) for tool inputs/outputs, but has no policy for how the LLM produces structured reasoning or non-tool structured outputs — intent classification, session compaction, and review generation are all unaddressed. Schema-Guided Reasoning (SGR) is a candidate philosophy worth evaluating before the orchestrator scaffold is written.

## Constraints

- Must work with openai Python SDK targeting Gemini 2.5 Flash (dec-20260320-003)
- Must not add production dependencies without justification
- Must preserve provider-swappability via env-var swap
- Pydantic schemas are already the typed boundary standard — any new approach must compose with them

## Optimization Targets

- Reasoning auditability — can we inspect why the LLM chose an action?
- Output consistency — do structured outputs reliably conform to expected shapes?
- Implementation simplicity — minimal schema-design overhead per LLM call type

## Observation Indicators

- Latency impact of constrained decoding vs free-form + validation
- Schema maintenance burden over time as call types grow

## Acceptance

A clear policy on when/whether to use reasoning schemas vs plain tool calls vs free-form text, documented in ARCHITECTURE.md, with refresh trigger set for post-first-implementation review.

## Blast Radius

Every LLM call path in the orchestrator: intent resolution, session compaction, review generation, tool loop. Foundational — changes after scaffold is written will be expensive.

## Reversibility

low
