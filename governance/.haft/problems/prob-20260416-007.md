# .claudeignore enforcement boundary: scan-only vs hard Read block

## Signal

Phase A of dec-20260416-001 deployed .claudeignore to protect sensitive paths (~/.ssh/, .env files, credentials). The security claim assumes .claudeignore hard-blocks the Read tool on matched paths. But .claudeignore documentation and known behavior suggests it may only prevent files from being auto-included during context scans (project-wide file indexing, glob patterns). If enforcement is scan-only, a prompt-injected explicit Read("/root/.ssh/id_rsa") call would still succeed and return the private key. The actual enforcement boundary has not been verified on the VM's Claude Code session.

## Problem Type

diagnosis

## Constraints

- Must not weaken bot functionality or interrupt Telegram messaging
- Test must be run in a VM Claude Code session (not via SSH bash), since .claudeignore is a Claude Code mechanism, not an OS-level file permission

## Optimization Targets

- Determine definitively whether .claudeignore blocks explicit Read calls or only scan-based inclusion
- If scan-only: close the gap with a PreToolUse hook that rejects Read calls on sensitive paths

## Observation Indicators

- Existing workflows (vault read/write, git operations) must not be disrupted by any gap-closing mechanism

## Acceptance

We know the actual .claudeignore enforcement boundary from a live VM test. If it hard-blocks Read: Phase A security claim holds, no further action needed. If scan-only: a PreToolUse hook is deployed that blocks explicit reads of sensitive paths, and the hook is verified to fire correctly.

## Blast Radius

Security posture assessment only — no production changes until diagnosis is complete. If a gap-closing hook is needed, it touches /root/.claude/settings.json on the VM.

## Reversibility

high
