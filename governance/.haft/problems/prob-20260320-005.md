# No LLM provider abstraction decision recorded

## Signal

ARCHITECTURE.md references Z.ai and an LLMClient interface, but no ADR or quint decision exists for the LLM provider layer. Token budget management, retry strategy, fallback providers, streaming support, and cost controls are all undecided. The name "Z.ai" appears without explanation or rationale.

## Constraints

- Runtime LLM must not get arbitrary shell access (ADR-005)
- Provider interface must be placed behind an adapter (project convention)
- Must support single-user MVP token budget without complex billing infrastructure

## Optimization Targets

- Define a minimal, swappable LLMClient interface before writing any tool-call logic
- Decide retry/fallback policy explicitly rather than ad-hoc

## Observation Indicators

- Streaming support requirements
- Cost per 1M tokens for candidate providers
- Rate limit characteristics

## Blast Radius

High — the LLM client is the core intelligence boundary of the platform. Every tool call, vault write, and response generation passes through it. Provider lock-in, API contract, and error handling patterns set here will be hard to change later.

## Reversibility

low

## Characterization v1 (2026-03-20)

| Dimension | Role | Scale | Unit | Polarity | Measurement |
|-----------|------|-------|------|----------|-------------|
| Provider portability | target | ordinal | low/medium/high | higher_better | Can orchestrator swap provider without changing call logic? Changing only env vars = high; changing adapter class = medium; rewriting orchestrator = low. |
| Interface surface area | target | ratio | method count | lower_better | Count of public methods/entry points the orchestrator must learn to call the LLM layer. |
| Streaming support | target | nominal | yes/deferred/no | higher_better | Does the variant support async streaming without a rewrite? yes = native; deferred = possible but requires extra code; no = blocked. |
| Retry/fallback complexity | target | ordinal | low/medium/high | lower_better | How much custom retry/fallback code must be written? Watch for over-engineering; do not optimize in MVP. |
| Token budget visibility | target | nominal | none/partial/full | higher_better | Can the orchestrator observe token usage per call? Watch but do not build complex billing logic at MVP. |
| Z.ai web search coupling | target | nominal | optional/mandatory/unsupported | higher_better | Is Z.ai web search capability per-turn opt-in only (optional), always active (mandatory), or unavailable (unsupported)? Must be optional. |

**Parity rules:** All variants evaluated at MVP scope: single user, Python 3.12+, asyncio end-to-end (per dec-20260320-002), Z.ai as initial provider. Same tool-call contract. Same secrets model (one API key env var). No second provider exists yet.
