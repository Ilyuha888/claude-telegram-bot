# Solutions for: ADR-006 staging complexity unjustified for single-user MVP

Problem: prob-20260320-002

## Variants (3)

### V1. V1 — Keep ADR-006 as-is (worktrees + PR)

Pre-approval staging in an isolated Dulwich worktree per review request. On approval, replay change_manifest in a fresh worktree, commit, push review branch, create PR via forge adapter. Full 6+4 state machine. Branch naming with transliteration and slug fallback.

**Strengths:**
- No implementation changes needed
- Replay integrity is well-specified
- PR provides GitHub diff UI for review
- Step-local idempotent recovery is fully specified

**Weakest link:** Dulwich worktree support is unverified. If prob-003 resolves negatively, this entire variant requires full rework — highest R_eff risk.

**Risks:**
- Dulwich worktree support unverified (prob-003 open) — if worktrees are not supported, entire staging model breaks
- Two worktrees per review (staging + replay) doubles filesystem state
- Forge HTTP adapter required for PR creation adds surface area
- Branch naming transliteration adds non-trivial complexity for zero-value in single-user context

**Rollback:** No change to make; this is the status quo.

### V2. V2 — Manifest-only staging + PR on approval

Drop pre-approval worktrees entirely. Proposed file content stored in Postgres as change_manifest blobs. On approval: fetch origin/main, apply manifest in-memory via Dulwich object model (no worktree), create review branch, commit, push, create PR. States collapse: pending -> awaiting_approval -> committing -> done | conflicted | failed_recoverable | abandoned. Conflict detection via replay still present.

**Strengths:**
- Eliminates Dulwich worktree dependency entirely — closes prob-003
- Fewer states (6 total: 4 main + 2 side classes)
- No pre-approval filesystem management
- Manifest-in-Postgres is already the source of truth in ADR-006 — this just removes the redundant worktree layer
- PR adapter kept as stepping stone for future needs

**Weakest link:** Dulwich in-memory index manipulation without a checked-out worktree — must verify porcelain API supports this before committing to V2.

**Risks:**
- Dulwich in-memory commit (without working tree checkout) needs verification — Dulwich supports ObjectStore and index manipulation but this pattern must be tested
- Proposed file content stored in Postgres — large binary attachments may be impractical (mitigable: cap blob size, reject binary writes)
- Loss of pre-approval diff materialization on disk (user sees Telegram diff only before approving)

**Stepping stone:** yes — opens future possibilities

**Rollback:** If Dulwich in-memory commit is not viable, fall back to a minimal worktree (checkout only at commit time, not at staging time) — reduces worktree lifetime from review duration to seconds.

### V3. V3 — Manifest-only staging + direct push to main (no PR)

Same as V2 but skip PR creation entirely. On approval: apply manifest in-memory, commit directly to main, push. No forge adapter needed. States: pending -> awaiting_approval -> done | conflicted | failed_recoverable | abandoned. User reviews diff in Telegram only — no GitHub PR.

**Strengths:**
- Minimal implementation surface
- No forge HTTP adapter required
- Fewest states (4 total)
- No dependency on GitHub or any forge

**Weakest link:** User experience quality — Telegram diff rendering is the only review surface; no fallback for complex multi-file changes. Also shares V2 Dulwich in-memory risk.

**Risks:**
- No PR means no GitHub diff UI — user must trust the Telegram diff summary completely
- Pushing directly to main has no secondary review gate; a bad approval cannot be caught before merge
- Rollback requires a separate revert review request

**Rollback:** Revert by creating a new review request that undoes the committed changes.
