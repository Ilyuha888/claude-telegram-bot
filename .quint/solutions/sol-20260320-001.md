# Solutions for: Single-user scope is an implicit assumption, not a recorded constraint

Problem: prob-20260320-006

## Variants (3)

### V1. V1 — Quint decision record only

Create a DecisionRecord in .quint/ that declares the single-user scope as an explicit architectural constraint. No changes to AGENTS.md.

**Strengths:**
- Full rationale lineage with evidence chain
- Machine-searchable via quint_query
- No doc duplication risk

**Weakest link:** Agent compliance — depends on agents always running /q-status; not enforced by the instruction file itself

**Risks:**
- Agents who skip /q-status before work will never see the constraint
- Constraint is invisible in the primary agent instruction file

**Rollback:** Archive the DecisionRecord if multi-user is added; no AGENTS.md change needed

### V2. V2 — AGENTS.md Architecture Invariants only

Add 'single-user MVP' to the Architecture Invariants section of AGENTS.md. No quint artifact created.

**Strengths:**
- Maximum agent visibility — invariants are read first
- Zero extra tooling required

**Weakest link:** Rationale lineage — no way for a future agent to audit why or when the constraint was set

**Risks:**
- No rationale lineage — future agents cannot find why the constraint exists
- No valid_until or refresh trigger — constraint can silently stagnate
- Violates quint protocol: high-blast-radius decisions should have quint records

**Rollback:** Remove the bullet from AGENTS.md when multi-user is added

### V3. V3 — Quint decision record + AGENTS.md invariant (recommended)

Create a DecisionRecord in .quint/ AND add one bullet to the Architecture Invariants list in AGENTS.md. The quint record carries full rationale; AGENTS.md carries visibility.

**Strengths:**
- Maximum agent visibility from AGENTS.md
- Full rationale lineage from quint record
- valid_until and refresh triggers make staleness detectable
- Both surfaces reinforce each other

**Weakest link:** Two-source sync — if AGENTS.md is updated without archiving the quint record (or vice versa), they diverge; acceptable given low frequency of change

**Risks:**
- Two places to update when multi-user is intentionally added (acceptable — one-line AGENTS.md edit + quint archive)

**Stepping stone:** yes — opens future possibilities

**Rollback:** Remove AGENTS.md bullet + archive DecisionRecord when multi-user is added
