# V1 — openai SDK + Gemini 2.5 Flash

## 1. Problem Frame

**Signal:** ARCHITECTURE.md references Z.ai and an LLMClient interface, but no ADR or quint decision exists for the LLM provider layer. Token budget management, retry strategy, fallback providers, streaming support, and cost controls are all undecided. The name "Z.ai" appears without explanation or rationale.

**Constraints:**
- Runtime LLM must not get arbitrary shell access (ADR-005)
- Provider interface must be placed behind an adapter (project convention)
- Must support single-user MVP token budget without complex billing infrastructure

## 2. Decision

**Selected:** V1 — openai SDK + Gemini 2.5 Flash

V1 meets every constraint: openai SDK builds transferable expertise, Gemini 2.5 Flash is a high-quality proprietary model confirmed working (tool calls + streaming smoke-tested 2026-03-20), 1,500 req/day free tier is the most generous available, and provider portability is one env-var swap. V2 (Groq) is an equally valid SDK choice — identical orchestrator code — retained as documented fallback on model quality or availability grounds.

**Invariants:**
- All LLM calls must be async — await client.chat.completions.create(...); no sync SDK usage (consistent with dec-20260320-002)
- Provider is swappable by changing OPENAI_BASE_URL and OPENAI_API_KEY env vars only — zero orchestrator code changes required
- openai Python SDK is the only LLM-related production dependency at this layer
- Thinking token budget must be accounted for — set max_tokens generously (>=1000) or disable thinking explicitly when latency matters

**Pre-conditions:**
- [ ] Gemini API key provisioned and stored in secrets model (never in repository)
- [ ] openai Python package added to dependencies with explicit version pin
- [ ] OPENAI_BASE_URL=https://generativelanguage.googleapis.com/v1beta/openai/ confirmed in environment config
- [ ] Smoke test results for tool calls and streaming on record (2026-03-20)

**Post-conditions:**
- [ ] Orchestrator calls await client.chat.completions.create(model=..., messages=..., tools=...) for tool-enabled turns
- [ ] Streaming path uses async for chunk in await client.chat.completions.create(..., stream=True)
- [ ] No custom retry logic in MVP — rely on SDK defaults
- [ ] No custom HTTP client or JSON parsing of raw LLM responses anywhere in the runtime path

**Admissibility:**
- NOT: Hand-rolled HTTP client or manual JSON parsing of LLM responses
- NOT: Synchronous SDK usage inside async coroutines
- NOT: Anthropic SDK imported anywhere in the runtime LLM path
- NOT: Hardcoded base_url or api_key in source code
- NOT: Custom retry loops reimplementing what the SDK already provides

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| V1 — openai SDK + Gemini 2.5 Flash | **Selected** | V1 meets every constraint: openai SDK builds transferable... |
| V2 — openai SDK + Groq (Kimi K2 / Llama 4) | Rejected | Model quality ceiling below Gemini 2.5 Flash for complex assistant tasks (reasoning, writing, multi-step planning). Groq is an excellent fallback — zero code changes to switch — but V1 wins on quality and free tier headroom. |
| V3 — Anthropic SDK + Z.ai proxy | Rejected | Directly contradicts stated user goal of building openai SDK expertise. Portability is low — switching providers requires SDK rewrite. Tool calls and streaming on the Z.ai Anthropic proxy are unverified. Eliminated. |

**Weakest link:** Gemini OAI-compat shim (https://generativelanguage.googleapis.com/v1beta/openai/) is confirmed working as of 2026-03-20 but is not a first-class Google API commitment. Fallback to Groq (V2) requires zero code changes.

**Evidence requirements:**
- Smoke test tool calls and streaming in first implementation sprint — already done 2026-03-20
- Verify thinking token behavior under production-realistic max_tokens settings
- Confirm rate limit headroom after first 30 days of personal use

## 4. Consequences

**Rollback plan:**
Triggers:
- Gemini OAI-compat endpoint deprecated or breaks tool call schema
- Gemini free tier reduced below 500 req/day making it unviable
- Model quality proves insufficient for actual assistant workloads after implementation
Steps:
1. Change OPENAI_BASE_URL to https://api.groq.com/openai/v1
2. Change OPENAI_API_KEY to Groq API key
3. Update model name constant to kimi-k2 or llama-4-scout
4. Zero orchestrator code changes required
Blast radius: env-var change only — no code modifications

**Refresh triggers:**
- Gemini OAI-compat endpoint deprecated by Google
- A second LLM provider is added to the runtime (triggers re-evaluation of Protocol abstraction — V3 revisit)
- Token costs or rate limits require explicit budget enforcement in the client layer
- openai SDK major version bump with breaking changes to chat.completions interface

**Affected files:** ARCHITECTURE.md — update LLMClient and Z.ai references to reflect Gemini + openai SDK decision, requirements.txt / pyproject.toml — add openai package with version pin, docs referencing Z.ai as the LLM provider
