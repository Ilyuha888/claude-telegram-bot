# V3 — Quint decision record + AGENTS.md invariant

## 1. Problem Frame

**Signal:** The entire platform is designed for one user but no ADR or architecture document declares this as a hard constraint. Any future agent or contributor can inadvertently design for multi-user scenarios (shared vault, per-user session isolation, concurrent approval flows) without triggering a deliberate review. ADR-006's staging complexity is partly justified by multi-editor risk avoidance — yet the risk does not exist. The constraint needs to be explicit so that lifting it requires a conscious decision, not incremental drift.

**Constraints:**
- Recording the constraint must not lock out a future multi-user path — it must just make expanding that path explicit and deliberate
- The constraint must appear in a durable location checked by agents before significant work (AGENTS.md or a quint decision)

## 2. Decision

**Selected:** V3 — Quint decision record + AGENTS.md invariant

Achieves both goals simultaneously: AGENTS.md gives every agent immediate visibility without requiring /q-status; the quint record preserves rationale lineage, valid_until, and refresh triggers. The maintenance overhead (two places to update when multi-user is added) is one line in AGENTS.md plus one quint archive call — acceptable for a constraint expected to be stable for 12+ months.

**Invariants:**
- the platform is scoped to a single user
- multi-user support is out of scope until a quint decision explicitly supersedes this record

**Pre-conditions:**
- [ ] README already states multi-user is out of scope for V1
- [ ] No multi-tenant code paths exist in the codebase

**Post-conditions:**
- [ ] AGENTS.md Architecture Invariants contains single-user bullet
- [ ] quint DecisionRecord is searchable via quint_query(search, 'single-user')
- [ ] prob-20260320-006 is Addressed

**Evidence requirements:**

## 4. Consequences

**Rollback plan:**

- a quint decision supersedes this record
Steps:

Blast radius: documentation only — no runtime code affected
