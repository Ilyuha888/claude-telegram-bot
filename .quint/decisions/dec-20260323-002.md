# BitGN PAC agent built as independent project, not inside muscat

## 1. Problem Frame

**Signal:** The entire platform is designed for one user but no ADR or architecture document declares this as a hard constraint. Any future agent or contributor can inadvertently design for multi-user scenarios (shared vault, per-user session isolation, concurrent approval flows) without triggering a deliberate review. ADR-006's staging complexity is partly justified by multi-editor risk avoidance — yet the risk does not exist. The constraint needs to be explicit so that lifting it requires a conscious decision, not incremental drift.

**Constraints:**
- Recording the constraint must not lock out a future multi-user path — it must just make expanding that path explicit and deliberate
- The constraint must appear in a durable location checked by agents before significant work (AGENTS.md or a quint decision)

## 2. Decision

**Selected:** BitGN PAC agent built as independent project, not inside muscat

PAC challenge (April 11 2026) requires a stateless task-solving agent against BitGN's simulated tool API. Muscat has zero production code and its architecture (Telegram + Obsidian + Git approval + Postgres) is ~90% irrelevant to PAC. Only conceptual overlap: orchestrator pattern, tool validation, LLM client wrapper. Building separately avoids polluting muscat's architecture with PAC-specific concerns while allowing proven patterns to flow back after the competition.

**Invariants:**
- Muscat and PAC repos remain independent — no shared dependency or monorepo
- PAC agent is stateless per-task; no Postgres, no session persistence
- PAC agent does not adopt muscat's Telegram or Obsidian integrations

**Pre-conditions:**
- [ ] BitGN sandbox API available for testing
- [ ] PAC competition rules and scoring criteria reviewed

**Post-conditions:**
- [ ] PAC agent runs as a single process hitting BitGN API
- [ ] After PAC (post April 11), evaluate backflow: orchestrator loop, injection resistance, tool validation, LLM client wrapper, empirical model data

**Admissibility:**
- NOT: Coupling PAC-specific code into muscat's runtime
- NOT: Delaying PAC work to wait for muscat code that doesn't exist

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| BitGN PAC agent built as independent project, not inside muscat | **Selected** | PAC challenge (April 11 2026) requires a stateless task-s... |
| Build PAC agent inside muscat repo | Rejected | Muscat's architecture solves a fundamentally different problem. Telegram transport, Obsidian vault, Dulwich Git, Postgres persistence, session compaction, human-in-the-loop approval — all irrelevant or conflicting with PAC's stateless per-task model. Would add dead-weight abstractions and confuse muscat's architectural boundaries. |
| Extract shared library from muscat for both projects | Rejected | No code exists to extract. Premature abstraction over zero implementations. The shared patterns (orchestrator loop, tool validation, LLM client) are better discovered empirically in PAC and then ported back. |

**Weakest link:** Backflow value is hypothetical until PAC agent is actually built and tested. If PAC patterns turn out to be too competition-specific, muscat gains less than expected.

**Evidence requirements:**
- PAC agent successfully completes practice tasks in BitGN sandbox
- Identify which patterns/code from PAC are portable to muscat Phase 0
- Document injection resistance techniques tested under adversarial conditions

## 4. Consequences

**Rollback plan:**
Triggers:
- PAC cancelled or rules change fundamentally
- Discovery that muscat already needs the same orchestrator loop before April 11
Steps:
1. Merge any useful PAC code into muscat if consolidation becomes warranted
2. Archive PAC repo if no longer needed
Blast radius: Low — repos are independent, rollback is just a repo-management decision

**Refresh triggers:**
- PAC competition completed (after April 11 2026)
- Muscat Phase 0 implementation begins — evaluate backflow at that point
- BitGN releases SDK or tooling that changes the build approach
