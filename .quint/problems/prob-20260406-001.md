# No knowledge lifecycle model — vault is infrastructure without operational modes

## Signal

Three independent implementations (Karpathy's LLM wiki, llm_under_hood's Personal OS v2, muscat) converge on the same vault infrastructure — markdown + Git + Obsidian + typed tool contracts. But the two that ship both have explicit operational modes (ingest → compile → query → lint/maintenance) that turn infrastructure into compounding knowledge. Muscat has detailed CRUD tool contracts and a manifest-staging flow but no design for these higher-level workflows. The vault can store notes; there is no design for how raw inputs become cross-referenced, structured knowledge, or how knowledge is queried back with synthesis, or how the vault stays internally consistent over time.

## Constraints

- Phase 0 runtime scaffold has higher execution priority — this problem informs future phases and must not block Phase 0
- Vault architecture decisions are settled: two-root vault (User_Obsidian_Vault + Agent_Obsidian_Vault), manifest-only staging in Postgres, review-before-commit for user vault writes, in-memory Dulwich commit on approval (dec-20260320-005)
- Single-user scope is an explicit constraint (dec-20260320-001)
- Selective SGR for non-tool structured outputs is decided (dec-20260324-001) — lifecycle outputs (compiled pages, query syntheses) must conform
- No new vault write primitives — modes must orchestrate existing vault.* and git.* tool contracts

## Optimization Targets

- Operational mode definitions: ingest, query, lint/maintenance — with clear entry/exit conditions, input/output contracts, and mapping to existing vault.* tool calls
- Knowledge compilation pipeline concept: how raw inputs (messages, links, files) are transformed into cross-referenced, structured vault content rather than just filed notes
- Mode composability: modes share tool contracts and session context without introducing new runtime primitives

## Observation Indicators

- Whether the lifecycle model influences Phase 0 interface design (should inform, must not block)
- Compatibility of the compilation model with both User and Agent vault roots — user vault writes stay review-gated even during ingest mode
- Search backend choice (currently undecided) — lifecycle query mode depends on it but should not be conflated with it

## Acceptance

A documented knowledge lifecycle spec covering: (1) ingest mode — input sources, compilation steps, output note types, cross-reference update strategy; (2) query mode — search + synthesis + optional file-back flow; (3) lint/maintenance mode — consistency checks, orphan detection, cross-reference repair. Spec must map each mode to existing tool contracts (vault.*, git.*) and identify open questions (e.g. search backend) without resolving them. No code required.

## Blast Radius

Design spec only. No production code exists — wrong model gets revised cheaply. Risk is opportunity cost: a poorly shaped lifecycle model could misdirect Phase 1+ implementation effort.
