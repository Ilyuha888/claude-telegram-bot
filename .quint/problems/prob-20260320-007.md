# No testing strategy for LLM-dependent code paths

## Signal

No ADR or architecture document specifies how to test code paths that invoke the LLM. The options — live API calls, recorded cassettes, mock adapters, deterministic stubs — have meaningfully different trade-offs in cost, flakiness, and coverage fidelity. Without a declared strategy, test files will be written inconsistently: some will hit the live API, some will skip the LLM layer entirely. This must be decided before any tool-call or handler code is written, not retrofitted later.

## Constraints

- Must not require live API calls in CI (cost and flakiness)
- Must not require shell execution or external services in unit tests
- LLMClient adapter boundary must be testable in isolation

## Optimization Targets

- Define the test double strategy before the first tool-call handler is written
- Ensure the LLMClient interface is designed with testability as a first-class property

## Observation Indicators

- Test execution cost per CI run
- Flakiness rate of LLM-touching tests

## Blast Radius

High — the testing strategy shapes the LLMClient adapter interface design, CI pipeline requirements, local dev ergonomics, and cost controls. Retrofitting a test strategy after handlers and tools are written requires rewriting both production and test code.

## Reversibility

medium
