# Solutions for: No LLM provider abstraction decision recorded

Problem: prob-20260320-005

## Variants (3)

### V1. V1 — openai SDK + Gemini 2.5 Flash

Configure openai.AsyncOpenAI(base_url='https://generativelanguage.googleapis.com/v1beta/openai/', api_key=GEMINI_API_KEY). No custom client class. Orchestrator calls client.chat.completions.create(...) directly. Tool calls and streaming confirmed working via smoke test. Provider swap = one env-var change.

**Strengths:**
- Tool calls, streaming, and basic completion all smoke-tested and confirmed on 2026-03-20
- Gemini 2.5 Flash is a high-quality proprietary model — significantly smarter than open-source 70B alternatives for assistant-class tasks
- Free tier: 1,500 req/day — most generous of any provider at zero cost
- openai SDK skill is fully transferable to OpenAI proper, Groq, Together, any OAI-compatible provider
- Provider portability: swapping to any OAI-compatible provider requires changing only base_url + api_key env vars

**Weakest link:** Google's OAI-compat shim stability — confirmed working today but not a first-class commitment. If deprecated, migrating to native Gemini SDK or another OAI-native provider (Groq) is a one-day task.

**Risks:**
- Gemini 2.5 Flash thinking tokens count against max_tokens — must set generous limit or disable thinking per request
- OAI-compat endpoint is a Google shim, not the native Gemini SDK — Google could alter or deprecate it
- Google's API reliability and deprecation cadence historically faster than OpenAI

**Stepping stone:** yes — opens future possibilities

**Rollback:** If Gemini OAI-compat endpoint changes: switch base_url to Groq (V2) or OpenRouter in one env-var change. No orchestrator code changes.

### V2. V2 — openai SDK + Groq (Kimi K2 / Llama 4)

Configure openai.AsyncOpenAI(base_url='https://api.groq.com/openai/v1', api_key=GROQ_API_KEY). Groq's API is natively OpenAI-compatible by design. Primary model: Kimi K2 (frontier MoE, ~32B active params) or Llama 4 Scout. Free tier: 1,000 req/day on large models.

**Strengths:**
- Groq API is natively OAI-compatible — not a shim, purpose-built to match the OpenAI spec
- Kimi K2 is a frontier-class MoE model (1T total params) — competitive with GPT-4o class on reasoning
- Inference speed: 400–1,000 tokens/sec — lowest latency of any option
- Tool calls confirmed (parallel_tool_calls in API schema)
- Free tier: 1,000 req/day on large models; paid tier ~$0.59/M tokens (70B) — well under $2/month at personal scale

**Weakest link:** Model quality ceiling for complex assistant tasks — Kimi K2 is strong but Gemini 2.5 Flash is better for nuanced writing, planning, and multi-step reasoning. This is an observation, not a blocker, but matters for a personal assistant.

**Risks:**
- Model roster on free tier can change — Groq rotates available models
- Open-source model quality ceiling below Gemini 2.5 Flash for complex reasoning and instruction-following
- Rate limit (1K req/day) tighter than Gemini free tier (1,500 req/day)

**Stepping stone:** yes — opens future possibilities

**Rollback:** If Groq model quality proves insufficient: switch base_url to Gemini OAI-compat (V1) in one env-var change.

### V3. V3 — Anthropic SDK + Z.ai proxy

Use anthropic.AsyncAnthropic(base_url='https://api.z.ai/api/anthropic', auth_token=Z_AI_TOKEN). Currently working in Claude Code. GLM models mapped to Anthropic model tiers via env vars. No cost beyond existing Z.ai subscription.

**Strengths:**
- Already confirmed working — Claude Code uses this path today
- No additional cost beyond existing Z.ai subscription
- GLM 4.6 is a capable model for general assistant tasks

**Weakest link:** Directly contradicts the user's stated learning goal (build openai SDK expertise). Even if it works perfectly, it optimizes for the wrong outcome. Tool call compatibility on the Anthropic proxy path is also unverified.

**Risks:**
- Anthropic SDK skill is provider-specific — not transferable to OpenAI-compatible providers
- Tool calls and streaming on this proxy path are unverified — smoke test blocked by balance issue on the paas endpoint; proxy behavior unknown
- Z.ai subscription terms and GLM-to-Anthropic mapping could change
- Contradicts stated goal: user wants openai SDK expertise

**Rollback:** Switching from Anthropic SDK to openai SDK requires rewriting all LLM call sites in the orchestrator — not a one-line change.
