# Solutions for: Sync vs async runtime model undecided

Problem: prob-20260320-001

## Variants (3)

### V1. V1 — Full asyncio end-to-end

All layers use async/await: PTB v22+ async handlers (forced), async DB driver (psycopg3 async or asyncpg), async HTTP for LLM calls (httpx.AsyncClient), Dulwich wrapped in asyncio.run_in_executor for all vault Git ops. One consistent concurrency pattern across the entire codebase. Executor delegation for Dulwich is the only sync/async bridge, and it is explicit and localized.

**Strengths:**
- Uniform pattern — no mental model switching between layers
- No risk of accidentally blocking the event loop from service code
- PTB v22+ constraint is satisfied natively, not bridged
- asyncio.run_in_executor cleanly isolates the one sync library (Dulwich)

**Weakest link:** Correct and consistent asyncio.run_in_executor usage for every Dulwich call; a missed await or direct sync call blocks the event loop silently

**Risks:**
- Dulwich run_in_executor usage must be disciplined — forgetting it blocks the loop
- Async DB drivers (psycopg3 async / asyncpg) add a new dependency vs sync psycopg2

**Stepping stone:** yes — opens future possibilities

**Rollback:** Reversing to sync core requires rewriting all service layers and DB driver — high cost after first implementation

### V2. V2 — Async edge, sync core

PTB handlers are async (forced by PTB v22+), but immediately delegate to sync service methods via asyncio.run_in_executor or loop.run_in_executor. Internal services use sync psycopg2, sync requests/httpx for LLM, sync Dulwich. Two coexisting patterns: async edge layer and sync interior.

**Strengths:**
- Service layer looks familiar to developers who think in sync
- Sync psycopg2 is battle-tested, no new DB dependency

**Weakest link:** Discipline boundary between async edge and sync core; one blocking call inside an async handler that skips the executor stalls the event loop

**Risks:**
- Every async handler must correctly delegate to executor — easily missed
- Two mental models coexist permanently; contributors must know which layer they are in
- run_in_executor overhead applied uniformly even where it is unnecessary
- Testing is harder: must mock both the async wrapper and the sync service

**Rollback:** Forward migration to full async is possible but requires rewriting all service layers

### V3. V3 — Thread-pool dispatch (sync-first)

PTB handler receives update, immediately dispatches the entire request turn to a ThreadPoolExecutor. All actual work — DB, LLM, Dulwich, business logic — executes synchronously inside the thread. The event loop exists only at the PTB application level and is not used by any service code.

**Strengths:**
- Internal services are pure sync — simpler to reason about in isolation
- No async/await syntax inside service layer

**Weakest link:** Error propagation and timeout/cancellation semantics across the thread-pool boundary; exceptions from worker threads require explicit marshalling back to the PTB coroutine

**Risks:**
- Error propagation and cancellation across thread/event-loop boundary is complex
- Highest bridging complexity for MVP: futures, exception chaining, timeouts all need explicit handling
- Thread-safety requirements introduced even for single-user MVP (shared state must be thread-safe)
- Contradicts the asyncio-first direction PTB v22+ was designed for

**Rollback:** Would require full rewrite of handler layer and introduction of async service layer to adopt V1 or V2
