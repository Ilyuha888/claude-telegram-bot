# Solutions for: Telegram bridge architecture: official plugin vs community Agent SDK bot

Problem: prob-20260416-004

## Variants (4)

### V_plugin. Official Telegram channel plugin (status quo)

Keep the current Anthropic-provided Telegram channel plugin (research preview) as an MCP server in Claude Code. No code changes. Fire-and-forget relay with no extension points.

**Strengths:**
- Zero implementation effort
- Native subscription billing (no CLAUDE_CODE_OAUTH_TOKEN workaround)
- Maintained by Anthropic
- No new attack surface

**Novelty:** Baseline — represents the current state with zero implementation effort and native subscription billing

**Weakest link:** Architectural ceiling on UX — the plugin is a closed MCP server with no exposed hooks for formatting, queuing, voice, or multi-session. Capabilities can only improve if Anthropic ships new plugin versions.

**Risks:**
- Messages lost when session closes (no queuing)
- No MarkdownV2/formatting control
- No voice transcription
- No inline keyboards
- Single session only
- Plugin is research preview — could be deprecated

**Diversity role:** Null hypothesis / status quo

### V_linuz90. linuz90/claude-telegram-bot (Agent SDK, TypeScript/Bun)

Deploy linuz90/claude-telegram-bot. Uses @anthropic-ai/claude-agent-sdk with CLI auth (existing subscription). Features: voice via OpenAI Whisper, message queuing, streaming, inline keyboards via ask_user MCP tool, multi-session (/new, /resume), intent classification, rate limiting, audit logging. TypeScript/Bun runtime.

**Strengths:**
- All blocking UX problems addressed: voice, formatting, queuing, multi-session
- CLI auth preserves subscription billing
- Security layer: intent classification, path validation, rate limiting, audit logging
- Active development (425 stars, recent releases)
- ask_user MCP tool converts options to inline keyboards without custom Telegram code

**Novelty:** Only variant using Anthropic's official Agent SDK — the intended programmatic interface for Claude Code, not a CLI hack or closed plugin

**Weakest link:** Agent SDK subscription billing policy stability — relies on CLAUDE_CODE_OAUTH_TOKEN workaround (confirmed working per anthropics/claude-agent-sdk-python#559) that Anthropic could revoke or change without notice

**Risks:**
- CLAUDE_CODE_OAUTH_TOKEN workaround could break on Anthropic policy change
- Adds Bun runtime dependency to VM
- Needs OpenAI API key for voice (~$0.006/min Whisper)
- TypeScript stack separate from existing shell/Python familiarity

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Agent SDK is Anthropic's official programmatic interface and will gain features over time. Adopting it positions the setup on the right abstraction boundary; future multi-repo, scheduling, and proactive agent features are natural SDK extensions.

**Rollback:** Re-enable Telegram channel plugin in settings.local.json, stop linuz90 bot process.

### V_terranc. terranc/claude-telegram-bot-bridge (CLI subprocess, Python)

Deploy terranc/claude-telegram-bot-bridge. Spawns Claude CLI as a subprocess and manages SDK streams directly. Features: Whisper voice + Volcengine alt, up to 3 concurrent sessions per user, progressive draft streaming, auto-convert numbered options to inline keyboards, daemon mode with auto-restart and log rotation. Python. v0.9.5 released 2026-04-10.

**Strengths:**
- Whisper voice with Volcengine fallback
- 3 concurrent sessions per user
- Auto-inline-keyboard generation
- Daemon mode with auto-restart
- Python stack
- Recent active development

**Novelty:** CLI subprocess approach — wraps the Claude CLI process directly rather than using the Agent SDK, yielding similar feature parity via a lower-level interface

**Weakest link:** Linux support maturity — project is macOS-first; WSL/Ubuntu/Debian listed as 'partial support'. Ubuntu 24.04 on the Hetzner VM is unverified. Could be minor fix or systemic blocker.

**Risks:**
- macOS-first; Ubuntu 24.04 support unverified
- CLI subprocess is fragile — Claude CLI changes could silently break it
- Not on Anthropic's intended SDK path

**Rollback:** Stop terranc daemon, re-enable channel plugin.

### V_hanxiao. hanxiao/claudecode-telegram (tmux bridge, Python)

Deploy hanxiao/claudecode-telegram. Injects messages into the running Claude Code session via tmux send-keys. Claude's stop hook reads the transcript and sends the response to Telegram via webhook. Closest to current architecture. Features: /status, /clear, /resume, /stop, /loop, inline keyboard for session resume. 583 stars, Python.

**Strengths:**
- Closest to current architecture — minimal migration distance
- Highest community validation (583 stars)
- No SDK dependency or billing workaround needed
- Works with existing tmux session and systemd setup

**Novelty:** Tmux injection bridge — unique approach treating the running Claude Code terminal session as a message bus; no SDK required, works with any Claude Code version supporting hooks

**Weakest link:** tmux injection is architecturally fragile — depends on Claude Code's terminal output format and stop hook timing; any Claude Code update could break the bridge silently

**Risks:**
- tmux injection is fragile and not a stable interface
- Requires Cloudflare tunnel or public webhook endpoint (new infra dependency)
- Voice not supported
- No proper multi-session (only /resume)
- Silent breakage risk on Claude Code updates

**Rollback:** Stop webhook bridge, remove Cloudflare tunnel config, re-enable channel plugin.

No compensating advantage anywhere.

## Pareto Front Trade-Offs

- Official Telegram channel plugin (status quo) — Best implementation effort (0) and architecture alignment (3) — zero cost to stay here. Pays with zero UX quality, no queuing, no voice, no multi-session. Only rational choice if the other three are judged too risky to adopt.
- linuz90/claude-telegram-bot (Agent SDK, TypeScript/Bun) — Best UX (5), multi-session (3), and security delta (+1). Stepping stone on Anthropic's intended SDK path. Sacrifices top message-reliability score (3 vs terranc's 4) and carries the CLAUDE_CODE_OAUTH_TOKEN billing risk as its WLNK.
- terranc/claude-telegram-bot-bridge (CLI subprocess, Python) — Best message reliability (4) — priority queue with auto-restart. Narrow compensating win over V_linuz90 (4 vs 3), low-stakes for single-user. Pays with more implementation effort (3 vs 2), weaker UX ceiling, worse multi-session (2 vs 3), no security improvements, no stepping-stone value.
- hanxiao/claudecode-telegram (tmux bridge, Python) — Dominated by V_linuz90 across all dimensions. Included for completeness — closest to current architecture and highest star count, but no dimension where it wins once V_linuz90 is in the set.

**Selection policy:** Maximize UX quality and message reliability for single-user assistant; subscription billing hard constraint; implementation effort and community risk as tiebreakers; stepping-stone value as secondary preference on close calls.

**Recommendation (advisory):** linuz90/claude-telegram-bot (Agent SDK, TypeScript/Bun)

**Recommendation rationale:** V_linuz90 covers all four blocking UX problems (voice, formatting, queuing, multi-session), sits on Anthropic's intended SDK path (stepping stone), and adds a security layer the others lack. The one place V_terranc beats it — message reliability 4 vs 3 — is marginal and doesn't matter for a single-user setup. The billing risk (CLAUDE_CODE_OAUTH_TOKEN) is real but has an explicit fallback and is confirmed working as of 2026-04.
