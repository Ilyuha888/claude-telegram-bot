# Solutions for: No testing strategy for LLM-dependent code paths

Problem: prob-20260320-007

## Variants (3)

### V1. V1 — Protocol fake only

Define LLMClientProtocol (Python Protocol). FakeLLMClient returns scripted ChatCompletion responses. Unit tests inject the fake; no HTTP ever occurs. Contract enforcement relies entirely on discipline when updating fakes.

**Strengths:**
- Zero external dependencies in test suite
- Fast CI — pure in-process Python
- Full asyncio compatibility — fake is just an async def
- Low maintenance as long as API is stable

**Weakest link:** Fake silently drifts from real API wire format; tool call schema mutations go undetected until integration smoke tests

**Risks:**
- Fake silently drifts from real API wire format
- Tool call schema mutations go undetected until integration smoke tests
- High divergence risk over time

### V2. V2 — VCR cassette recording (pytest-recording / vcrpy)

Record real HTTP exchanges once against the live endpoint; replay in CI using vcrpy or pytest-recording. Highest wire-level fidelity; captures exact JSON shapes including headers and error envelopes.

**Strengths:**
- Highest fidelity — exact wire replay
- Catches JSON schema regressions deterministically
- No manual stub writing

**Weakest link:** Async HTTP support is fragile (httpx/aiohttp + VCR requires patching); cassettes go stale when API or prompt changes

**Risks:**
- Async HTTP support is fragile — httpx/aiohttp + VCR requires patching event loop
- Cassettes go stale when API or prompt changes
- Bootstrapping requires a live call outside CI
- cassette storage in git repo grows over time

**Rollback:** Delete cassette files and fall back to fake; no production code change needed

### V3. V3 — Layered: typed stub + Pydantic schema contract tier

Same as V1 at unit-test layer (StubLLMClient with scripted responses). Pydantic schemas for tool-call request/response payloads are the shared contract — both prod code and stubs validate against the same models. A separate tests/contract/ tier (excluded from default CI, marked @pytest.mark.live) hits the live API and asserts real responses satisfy those same Pydantic schemas.

**Strengths:**
- Fast unit CI — no HTTP
- Schema validation catches drift between stub and real API during contract runs
- Full asyncio compatibility
- Contract tier is a pure pre-merge gate — not default CI cost
- Stepping stone: Protocol + Pydantic contract enables future cassette replay (V2) if needed

**Weakest link:** Stub fixture scripts need updating when tool schemas evolve; schema validation only catches mismatches when contract tier is actually run

**Risks:**
- Stub fixture scripts need updating when tool schemas evolve
- Contract tier requires OPENAI_API_KEY secret in CI for pre-merge gate

**Stepping stone:** yes — opens future possibilities

**Rollback:** If contract tier proves too expensive to maintain, demote to V1 (pure fake) without changing LLMClientProtocol interface or production code
