# Solutions for: Agent cannot deliver images or files to the user via Telegram — outbound is text-only

Problem: prob-20260422-001

## Variants (2)

### V1. Fire-and-forget MCP drop-file with server-side enforcement

A dedicated send-file MCP subprocess validates path (ALLOWED_PATHS env injection + forbidden-segment/filename deny list), applies per-kind size limits (10MB photo / 50MB document), writes a /tmp/send-file-*.json request, and returns immediately. The bot polls for the file at the result event (after all tools have run) and delivers via ctx.replyWithPhoto/Video/Audio/Document based on send_kind written by the server. Supports send_as_document flag for full-quality image delivery.

**Strengths:**
- Server-side enforcement isolates security logic from bot
- Fire-and-forget: agent continues generating immediately
- Audit log on every attempt (delivered/failed)
- send_as_document enables full-quality image delivery

**Novelty:** Combines server-side path/kind enforcement with bot-side delivery, decoupling security from Telegram API

**Weakest link:** Poll at result event fires once; if delivery fails, no automatic retry — agent sees the error on next turn only

**Risks:**
- Poll timing: must fire after MCP has run (fixed by polling at result event)
- Drop-file race if two send_file calls in one turn

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** send_kind and drop-file schema can be extended for chunked upload, progress callbacks, or multi-file batching

**Rollback:** Remove send-file entry from mcp-config.ts to disable; existing drop-files are cleaned up on each poll cycle

### V2. Native canUseTool intercept — no subprocess

Intercept file-send requests inside the bot's canUseTool callback or a dedicated bot tool (not MCP). The bot performs path validation and calls ctx.replyWithPhoto directly inside the permission flow, before returning allow to the SDK.

**Strengths:**
- No subprocess overhead
- Delivery is synchronous with tool approval

**Novelty:** Avoids subprocess entirely; all logic in-process

**Weakest link:** canUseTool callback does not have reliable access to Telegram ctx at tool-approval time in the SDK V2 event model

**Risks:**
- Couples security logic to canUseTool callback, making it harder to extend
- canUseTool runs before the assistant event — no ctx.reply available at that point without careful threading
