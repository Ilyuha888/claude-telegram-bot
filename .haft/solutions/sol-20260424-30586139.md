# Solutions for: No routine model for KB usage — when and how to interact

Problem: prob-20260417-005

## Variants (2)

### V_lean. Lean status-surface routines

Each routine does one focused read and returns a concise status list. Daily (09:00 Moscow): read 20-projects/tasks.md, extract In Progress rows, list them with tags. Read 00-inbox/ file list, count unprocessed captures. Output: bullet list, ≤10 items. Weekly (Sun 20:00): read Agent_Obsidian_Vault/drafts/ filenames + frontmatter status; count stale (>30d no edit) and orphan candidates; list top 2 promotion candidates by name only. No MOC analysis. Monthly/quarterly: count active projects in 20-projects/ by status field; count inbox items outstanding. No deep vault traversal.

**Strengths:**
- Low token budget per fire — fast, cheap, predictable
- Low false-positive risk — reads only current-status fields, not full note content
- Simple prompts are easy to audit and adjust
- Weekly report is quick to produce and quick to read

**Novelty:** Purely informational — reads the minimum set of files and surfaces raw status; no interpretation, no recommendations. The routine is a structured grep, not an analysis.

**Weakest link:** The daily digest has no 'focus suggestion' — it lists everything In Progress equally, which is often 6-10 items with no priority signal. The user still has to decide what to do today from a flat list.

**Risks:**
- Daily list provides no priority or focus signal — user must interpret
- Weekly report misses MOC gap analysis (one of three V-A curation axes)
- Monthly routine is thin — may feel like overhead for the value it provides

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** The focused-read pattern (specific files, specific frontmatter fields) is easy to extend: adding a 'focus suggestion' to the daily prompt or adding MOC analysis to the weekly prompt are incremental prompt changes, not architectural ones.

### V_rich. Analytical routines with specific proposals

Each routine reads more deeply and interprets the data. Daily (09:00 Moscow): read 20-projects/tasks.md (In Progress rows), read each active project file for context, read 00-inbox/ files to count and characterize unprocessed captures. Output: ranked task list with 'one focus for today' suggestion based on project context and task age. Weekly (Sun 20:00): full V-A curation pass — (1) read Agent_Obsidian_Vault/drafts/, identify top 3 promotion candidates with specific vault paths and one-line promotion rationale; (2) find stale notes in 10-notes/ (>30d since edit, no outbound links); (3) identify MOC files in 60-mocs/ with no recent updates and suggest which note cluster needs a new or updated MOC. Monthly: active project audit — read each project file, flag projects with no recent activity, suggest close or archive. Quarterly: full lifecycle review across all folders.

**Strengths:**
- Daily output is actionable: named focus reduces decision overhead
- Weekly curator covers all three V-A axes (draft promotion + stale/orphan + MOC gaps)
- Monthly audit surfaces projects that have silently stalled
- Promotion proposals include specific vault paths — user can act directly

**Novelty:** Analytical and prescriptive — reads full project context, interprets staleness and priority signals, and names specific notes and paths in its proposals. The routine is a mini-curator, not a grep.

**Weakest link:** Token budget is significantly higher — reading each active project file for the daily digest could push 4-8K tokens per fire depending on project file length. The 'focus suggestion' is also a judgment call that Claude may get wrong if project context is ambiguous.

**Risks:**
- Higher token budget per fire — daily digest reads all active project files
- False-positive risk on 'focus suggestion' — Claude may prioritize the wrong thing
- Weekly report is long to read — may feel like work to process

## Pareto Front Trade-Offs

- Lean status-surface routines — Low token budget, low false-positive risk, focused vault read. Loses on actionability (flat list, no focus signal) and misses MOC gap analysis. Rational if you prefer fast raw-status surface and decide priority yourself.
- Analytical routines with specific proposals — Full V-A curation coverage, specific promotion proposals with vault paths, named daily focus. Higher token budget (~6-12K) and medium false-positive risk on the focus suggestion. Rational when the routine must reduce decision overhead, not just report status.

**Selection policy:** Maximize output_actionability first; minimize false_positive_risk second; weekly_curator_completeness at least partial. Token budget and vault_read_depth are soft — acceptable up to ~12K/fire on subscription billing.

**Recommendation (advisory):** Analytical routines with specific proposals

**Recommendation rationale:** The acceptance criterion requires the weekly report to be worth acting on within 24h. V_lean's flat list and partial curator coverage means the user still manually identifies focus and promotion candidates — the routine saves time reading but not deciding. V_rich's proposals with specific vault paths and a daily focus suggestion are what close that gap. Token cost is acceptable on subscription; the false-positive risk on focus suggestion is low-stakes (user ignores a wrong suggestion and moves on).
