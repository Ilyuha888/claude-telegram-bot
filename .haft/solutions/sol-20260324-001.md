# Solutions for: No structured output strategy for orchestrator LLM calls

Problem: prob-20260324-001

## Variants (3)

### V1. V1 — Tool-calls-only (status quo trajectory)

All structured output — including non-action needs like intent classification and session compaction — goes through the openai SDK tool-call mechanism. These are modeled as 'internal tools' (tools that return structured data but have no side effects). Reasoning is implicit in chain-of-thought or thinking tokens only.

**Strengths:**
- Single output mechanism — no dual paradigm to learn or maintain
- Tool-call format is natively supported by Gemini 2.5 Flash via openai-compat endpoint
- No additional schema scaffolding beyond existing tool contracts

**Weakest link:** No auditability of reasoning steps; intent classification and compaction lack inspectable intermediate state

**Risks:**
- Reasoning steps are invisible — no way to audit why the LLM chose a particular intent or compaction
- Internal tools are a semantic misuse of the tool-call abstraction
- Output conformance depends entirely on provider tool-call enforcement

### V2. V2 — SGR-everywhere

Every LLM call that needs structured output uses response_format with a JSON schema that bakes reasoning checkpoints into the schema itself (e.g., a 'reasoning' field before 'decision'). Pydantic models define both reasoning steps and final output. All outputs are auditable.

**Strengths:**
- Full reasoning auditability on every call
- Pydantic schema is the single source of truth for both reasoning and output shape
- Consistent mechanism regardless of call type

**Weakest link:** Gemini 2.5 Flash structured output support via openai-compat response_format is unverified; schema overhead may be disproportionate for simple calls

**Risks:**
- Schema design overhead per call type is high — every call needs a reasoning-aware schema
- response_format JSON schema support via Gemini openai-compat endpoint is not verified for all call types
- May conflict with tool-call mechanism — can't use both in the same API request reliably
- Adds mandatory fields that may not add value for simple calls

### V3. V3 — Selective SGR (hybrid policy)

Tool calls use the existing tool-call mechanism unchanged. Non-tool structured outputs (intent resolution, session compaction, review generation) use response_format with Pydantic-backed schemas that include a short reasoning field. SGR is adopted only where auditability or consistency is a measured need — not universally.

**Strengths:**
- Tool-call paths unchanged — zero regression risk for action execution
- Auditability where it matters most (intent resolution, compaction decisions)
- Pydantic schemas compose naturally — response_format schema derived from same models
- Incremental — can start with one call type and expand based on evidence

**Weakest link:** Policy clarity — without a documented rule for when to add reasoning schemas, the hybrid drifts toward either extreme over time

**Risks:**
- Two output paradigms to maintain and document
- Clear policy boundary required to prevent drift — when do you add a reasoning field?
- response_format Gemini compatibility still needs a spike

**Stepping stone:** yes — opens future possibilities

## Non-Dominated Set

**Pareto front:** V3 — Selective SGR (hybrid policy)

**Selection policy:** V1 fails reasoning_auditability (none). V2 dominates on auditability but has unverified provider_compatibility (a hard constraint) and disproportionate schema overhead. V3 is the only non-dominated variant: it satisfies the provider constraint, delivers auditability where it matters, and keeps overhead proportional. The weakest link (policy clarity) is addressable by documenting the rule in ARCHITECTURE.md.

**Recommended:** V3 — Selective SGR (hybrid policy)
