# Solutions for: Bot conflates knowledge-work conversation with infrastructure commands

Problem: prob-20260420-001

## Variants (5)

### V1. Slash-prefix router (text-only)

Bot-side dispatcher intercepts messages beginning with /work, /sessions, /attach, /close, /repos BEFORE calling SDK query(). No LLM tokens consumed for Mode-2 commands. All other text enters Mode-1 LLM path unchanged. Discoverability via BotFather /setcommands autocomplete.

**Strengths:**
- Smallest implementation surface (~1 day)
- Zero modal confusion risk
- Zero context bleed guaranteed — Mode-2 never reaches canUseTool
- Reuses existing Telegram slash-command convention

**Novelty:** Purely stateless — no toggle, no keyboard, no modal state. Mode identity is implicit in message prefix.

**Weakest link:** Discoverability: depends on users noticing Telegram's native slash-autocomplete; prob-20260417-006 (skill discovery UX) must be resolved alongside to make the commands findable on first use.

**Risks:**
- Users unfamiliar with slash commands may never discover /work

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** V5 is V1 + inline keyboard overlay; shipping V1 first is a direct stepping stone to V5 if discoverability proves insufficient.

### V2. Persistent reply-keyboard [💬 KB] / [🛠 Ops]

Every Mode-1 bot response attaches a one-row ReplyKeyboardMarkup with [💬 KB][🛠 Ops] buttons. Tapping Ops swaps to a Mode-2 sub-menu (Work, Sessions, Attach, Close). Tapping KB returns to bare text mode. LLM never sees keyboard state.

**Strengths:**
- High UX clarity — mode always visible
- No slash-command discovery needed

**Novelty:** Always-visible persistent UI element — mode cue is visual and permanent, not behavioral.

**Weakest link:** Reply keyboards crowd the mobile input area and can be permanently dismissed by users; re-attaching on every outbound message is noisy and may feel aggressive.

**Risks:**
- Reply keyboard dismissed = silent degradation to V1-equivalent without discoverability
- State machine adds complexity

### V3. Dual bot handle (@assistant_kb + @assistant_ops)

Two separate bot tokens, two systemd services. Mode 1 runs in the current linuz90 bot; Mode 2 is a minimal second bot that only handles /work, /sessions, /attach, /close. Context bleed is structurally impossible — different Telegram conversations. Requires a shared SQLite backend for session metadata.

**Strengths:**
- Strongest possible mode_separation score
- Zero implementation coupling between modes
- Can evolve independently

**Novelty:** Transport-level separation — the only variant that eliminates mode-bleed by making it a different chat channel, not a routing rule.

**Weakest link:** Operator complexity doubles: second BOT_TOKEN, second .env, second systemd unit, second BotFather registration. Violates the spirit of dec-20260416-003 (single bridge) — would require a refresh/supersede decision before implementation.

**Risks:**
- dec-20260416-003 needs refresh before starting
- Doubles operational burden for a single-user system

### V4. Explicit /ops modal toggle

User types /ops to enter Mode-2 state (mode flag per chat_id); bot responds with a fixed command menu and refuses LLM invocation. /chat or /exit returns to Mode 1. Each Mode-2 response is prefixed [OPS] as visual state indicator.

**Strengths:**
- Simple to implement (~1.5 days)

**Novelty:** Explicit stateful toggle — the only variant that uses persistent per-chat mode state as the routing mechanism.

**Weakest link:** Modal state is invisible when user returns after a gap; users will type questions in Ops mode and receive terse 'unknown command' rejections, creating confusion.

**Risks:**
- Modal confusion well-documented UX failure pattern
- Requires clear 'you are in OPS mode' reminder on every response to be usable

### V5. Hybrid: V1 router + ephemeral /menu inline keyboard

Pure slash-command router (V1 core) plus a /menu command that sends an ephemeral inline keyboard with buttons (callback_data) that trigger the same Mode-2 handlers. Power users use slashes directly; newcomers tap /menu. No modal state, no persistent reply keyboard. Reuses the existing askq: callback scaffolding from dec-20260420-003.

**Strengths:**
- ux_clarity score 3 — menu available on demand
- mode_separation score 3 — LLM never invoked for Mode-2
- ~2 days, reuses askq: callback scaffolding
- No modal state, no persistent keyboard noise

**Novelty:** Dual-surface: combines behavioral routing (V1) with a discoverable menu overlay that reuses the existing inline-keyboard callback infrastructure.

**Weakest link:** Still depends on /setcommands and prob-20260417-006 for initial discoverability of /menu itself; the inline keyboard is only helpful once the user knows /menu exists.

**Risks:**
- Two surfaces to document for users

## Dominated Variant Elimination

- Slash-prefix router (text-only) — dominated by Hybrid: V1 router + ephemeral /menu inline keyboard. V5 scores ux_clarity=3 vs V1's 2, identical on all other dimensions. V1 is a stepping stone to V5 but strictly dominated by it on the scored surface. Ship V1 only if 1-day constraint exists.
- Persistent reply-keyboard [💬 KB] / [🛠 Ops] — dominated by Hybrid: V1 router + ephemeral /menu inline keyboard. V2's reply-keyboard can be dismissed by users, capping effective clarity at 2. V5 achieves clarity=3 at the same effort (2d) with no persistent keyboard noise.
- Explicit /ops modal toggle — dominated by Hybrid: V1 router + ephemeral /menu inline keyboard. V4 scores ux_clarity=1 and mode_separation=2 — worst on both primary targets — at 1.5d effort. Modal state is a known UX failure pattern with no compensating advantage.

## Pareto Front Trade-Offs

- Hybrid: V1 router + ephemeral /menu inline keyboard — Clarity=3, separation=3, ~2d. Reuses existing askq: inline-keyboard callback scaffolding. Preferred under the declared policy (min effort among separation=3 options). Advisory recommendation.
- Dual bot handle (@assistant_kb + @assistant_ops) — Also clarity=3, separation=3 on scored dimensions, but 4d effort and requires dec-20260416-003 refresh/supersede. Warranted only if transport-level isolation becomes a safety requirement.

**Selection policy:** Prefer minimum impl_effort among variants with mode_separation=3; break ties on ux_clarity. Single-user system — operational simplicity outweighs enterprise isolation unless safety demands it.

**Recommendation (advisory):** Hybrid: V1 router + ephemeral /menu inline keyboard

**Recommendation rationale:** V5 achieves mode_separation=3 and ux_clarity=3 in ~2d by extending existing askq: callback path. V3 scores identically on all five dimensions but costs twice the effort and requires reopening dec-20260416-003.
