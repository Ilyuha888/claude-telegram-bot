# Solutions for: VM security hardening: root user, no network controls, no .claudeignore

Problem: prob-20260416-005

## Variants (3)

### V1. Layered rollout — config + user migration, skip outbound egress

Phase A: deploy .claudeignore (exclude ~/.ssh/, *.pem, .env*, credentials), add --disallowedTools to settings.local.json, enable UFW deny-all-inbound + allow SSH. Phase B: create dedicated `claude-assistant` user, migrate ~/.claude/ config + vault clone + GitHub PAT + claude login auth, update systemd unit (User=claude-assistant), update start-assistant.sh paths. Outbound egress filtering explicitly deferred — single-user threat model doesn't justify proxy maintenance.

**Strengths:**
- Eliminates root privilege escalation risk
- Blocks credential reads via .claudeignore
- Restricts dangerous tool access
- No ongoing service to maintain
- Inbound firewall as baseline

**Novelty:** Only variant that includes non-root user migration while keeping operational overhead near-zero by skipping egress proxy

**Weakest link:** User migration complexity — auth tokens (claude login), GitHub PAT, and MCP server paths are all rooted under /root. Migration requires re-authentication and path updates across ~5 config files. If any auth step fails, the assistant is down until SSH-fixed.

**Risks:**
- claude login may not work cleanly for non-root users — needs testing
- tmux session ownership changes
- post-commit hook in vault needs re-permissioning
- MCP server binaries (haft, etc.) may need path updates
- Downtime during migration

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Non-root user is the prerequisite for future privilege separation (e.g., seccomp, cgroups, container isolation). Also enables egress filtering per-user via iptables owner match if needed later.

**Rollback:** Revert systemd unit to root, restore /root/.claude config. Messy but possible within 30 min SSH session.

### V2. Full hardening — all four components including outbound proxy

Everything in V1 plus outbound egress filtering via tinyproxy running on localhost. Proxy allowlist: api.anthropic.com, api.telegram.org, github.com, registry.npmjs.org, context7 endpoints. iptables rule forces all outbound HTTP/HTTPS from claude-assistant user through the proxy. Direct outbound blocked. Proxy runs as a separate systemd service.

**Strengths:**
- Maximum attack surface reduction
- Blocks data exfiltration even via prompt injection
- Hostname-based filtering (not brittle IP lists)
- Audit log of all outbound connections via proxy logs

**Novelty:** Only variant that addresses outbound data exfiltration risk via hostname-level egress control

**Weakest link:** Proxy maintenance burden — tinyproxy config must track every legitimate endpoint Claude Code contacts. New MCP servers, plugin updates, or Anthropic infrastructure changes break the assistant silently until the allowlist is updated.

**Risks:**
- tinyproxy is another service to monitor and restart
- Incomplete allowlist causes silent failures — Claude just can't reach an endpoint
- Proxy adds latency to every API call
- Debugging connectivity issues becomes harder
- Must discover ALL legitimate endpoints before deployment

**Rollback:** Remove iptables rules, stop tinyproxy, revert to V1 state. Clean rollback in ~15 min.

### V3. Config-only hardening — .claudeignore + disallowedTools + inbound UFW

Deploy .claudeignore at /root/.claudeignore excluding ~/.ssh/, *.pem, .env*, /etc/shadow, credential files. Add --disallowedTools to settings.local.json to block dangerous tool patterns. Enable UFW: deny all inbound except SSH (22/tcp). No user migration. No outbound filtering. Claude Code remains root but with reduced tool surface and protected sensitive paths.

**Strengths:**
- 5-minute deployment via SSH
- Zero downtime — hot-reload config
- Zero risk of breaking auth, tmux, or systemd
- Inbound firewall blocks port scanning
- Immediately reversible

**Novelty:** Only variant that delivers meaningful hardening with zero system administration changes — pure Claude Code configuration

**Weakest link:** Still running as root — .claudeignore and --disallowedTools are advisory guardrails enforced by Claude Code itself, not OS-level isolation. A sufficiently sophisticated prompt injection could potentially bypass them via Bash tool if it's not in the disallowed list.

**Risks:**
- .claudeignore bypass if Bash tool reads files directly
- Root privileges remain — blast radius of any escape is total
- No protection against outbound exfiltration

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Natural first phase before V1. Proves the config layer works, builds confidence for the more disruptive user migration. Can be deployed today as a quick win.

**Rollback:** Delete .claudeignore, revert settings.local.json, ufw disable. 2-minute rollback.

## Non-Dominated Set

## Pareto Front Trade-Offs

- Layered rollout — config + user migration, skip outbound egress — Best security-to-overhead ratio. Eliminates root (biggest single risk factor) with all config-layer protections, no ongoing services. One-time migration cost. Weakest link: auth token portability during user migration.
- Full hardening — all four components including outbound proxy — Maximum security ceiling — only variant addressing outbound exfiltration. But proxy adds ongoing maintenance: allowlist drift causes silent failures, latency on every API call, another service to monitor. Marginal security gain over V1 is disproportionate to single-user threat model.
- Config-only hardening — .claudeignore + disallowedTools + inbound UFW — Best effort/risk ratio and natural Phase A of V1. Near-zero implementation risk, instant deployment, immediate rollback. But leaves root privileges intact — advisory guardrails only, no OS-level isolation.

**Selection policy:** Maximize security improvement per unit of operational overhead, constrained by single-user threat model. Prefer variants with low ongoing maintenance burden.

**Recommendation (advisory):** Layered rollout — config + user migration, skip outbound egress

**Recommendation rationale:** V1 is the sweet spot: eliminates root (biggest single risk factor) and deploys all config-layer protections without adding ongoing maintenance. Implementation effort is a one-time cost during a planned SSH session. Recommended execution: deploy V3 first as Phase A (quick win, proves config layer works), then complete V1 with user migration as Phase B. V2 remains available if the threat model changes, but the proxy's maintenance burden is unjustified for a single-user personal assistant.
