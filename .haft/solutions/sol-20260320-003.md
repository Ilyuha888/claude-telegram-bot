# Solutions for: No LLM provider abstraction decision recorded

Problem: prob-20260320-005

## Variants (3)

### V1. V1 — openai SDK as LLMClient (configure, don't wrap)

Configure openai.AsyncOpenAI(base_url=Z_AI_BASE_URL, api_key=...) directly. No custom client class. The orchestrator calls client.chat.completions.create(...) directly. Web search enabled via model parameter per turn. Provider is swapped by changing one env var.

**Strengths:**
- Zero custom client code — no adapter maintenance burden
- Full OpenAI SDK compatibility: streaming, tool calls, retries via tenacity or httpx built-in
- Portability across all OpenAI-compatible providers (Groq, Together, OpenAI proper) by changing one env var
- Streaming is natively supported via async for chunk in client.chat.completions.create(..., stream=True)

**Weakest link:** Z.ai tool-call and streaming compatibility with the openai SDK is assumed, not yet verified — must be smoke-tested in first implementation sprint

**Risks:**
- Orchestrator is coupled to the openai SDK interface — not a generic Protocol
- If Z.ai diverges from OpenAI SDK contract, shim code is needed

**Stepping stone:** yes — opens future possibilities

**Rollback:** If Z.ai diverges from openai SDK contract, wrap in a minimal shim class (<=50 lines) normalizing the response — V2-lite, not full rewrite

### V2. V2 — Custom thin async adapter (hand-rolled httpx class)

Write a minimal LLMClient class with async def complete(messages, tools, web_search) -> LLMResponse. Internally calls Z.ai via httpx. Defines a project-owned response schema via Pydantic. No dependency on the openai SDK.

**Strengths:**
- Full control over request shape and response parsing
- Explicit Pydantic boundary — typed schema owned by the project
- No dependency on openai SDK version pinning

**Weakest link:** Maintenance cost of manually reimplementing SDK features (streaming, tool-call parsing, retries) is systematically underestimated at the start of a project

**Risks:**
- Reimplements what the openai SDK already provides: streaming, tool-call parsing, retry logic
- Maintenance burden grows as Z.ai API evolves
- Switching providers requires rewriting the adapter, not just changing env vars

**Rollback:** No clean rollback path — switching providers means rewriting the adapter

### V3. V3 — Python Protocol + Z.ai implementation

Define LLMClientProtocol(Protocol) with typed async methods. ZAIClient implements it using the openai SDK internally. Future providers implement the same protocol. Orchestrator depends only on the Protocol, enabling mock implementations in tests.

**Strengths:**
- Maximum decoupling — orchestrator depends on a Protocol, not a concrete class
- Enables mock/stub implementations in tests without real API calls
- Explicit typed contract for all LLM interactions

**Weakest link:** Premature abstraction — the Protocol's method signature and return types can only be validated by contact with a second real provider; until then it may be shaped around Z.ai quirks and silently wrong

**Risks:**
- Over-engineering for a single-provider MVP — Protocol shape can only be validated when a second real provider is added
- Protocol conformance not enforced at runtime (structural subtyping only)
- Adds abstraction cost and extra file before a real need exists

**Stepping stone:** yes — opens future possibilities

**Rollback:** Protocol can be removed and orchestrator updated to use concrete class directly if it proves to be wrong shape
