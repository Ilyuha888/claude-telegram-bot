# Solutions for: Telegram attachments land ephemerally in /tmp — can't be persisted into the vault alongside a note

Problem: prob-20260423-84db08b4

## Variants (2)

### V1. buildMessageContext hint — surface /tmp paths in user turn

Extend buildMessageContext to prepend a machine-readable [Attachments on disk: ...] block with MIME type and size to each turn that includes downloaded files. Agent uses its existing Read/Write/Edit/Bash toolset to copy bytes into a sibling files/ directory, add an Obsidian wikilink, and commit. No new MCP tool. Contract documented in CLAUDE.md.

**Strengths:**
- Zero new infrastructure — reuses Read/Write/Bash the agent already has
- Additive to inline content blocks (vision still works)
- Composes cleanly with buildMessageContext Forwarded/Quoting prefixes
- Agent can choose filename, target note, and link style autonomously

**Novelty:** Uses the text turn itself as the signaling channel rather than a new tool or event

**Weakest link:** Quality of persistence (filename, wikilink placement, commit message) depends entirely on agent judgment, not enforced by code

**Risks:**
- Agent must interpret the contract correctly on every turn — relies on CLAUDE.md instruction quality
- Files stay in /tmp until the turn ends; if bot crashes mid-turn, bytes are lost

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** The /tmp path hint is sufficient for a future save_attachment_to_vault MCP tool to read from — upgrading to structured tooling later doesn't require a schema change

### V2. Dedicated save_attachment_to_vault MCP tool

New MCP server exposes a save_to_vault tool that takes the /tmp path, target note path, and optional filename. The server copies the file into the note's files/ directory, generates the wikilink, and patches the note atomically.

**Strengths:**
- Deterministic naming and wikilink format — not agent-judgment-dependent
- Can enforce file naming convention in code

**Novelty:** Moves copy+link+commit logic out of agent reasoning into a deterministic server

**Weakest link:** Agent still needs to decide which note to target — the hard part of the problem is not addressed by the tool

**Risks:**
- New MCP subprocess to maintain
- Target note path still requires agent judgment — doesn't fully eliminate the natural language gap
- More surface area to secure (write access to vault from subprocess)
