# Solutions for: No scheduled task execution from Telegram when using ephemeral bot sessions

Problem: prob-20260416-006

## Variants (4)

### V_cron_cli. VM cron + claude -p per-fire (stateless subprocess)

A crontab or systemd timer on the Hetzner VM fires each scheduled routine. The cron job invokes `claude -p "<prompt>"` (headless CLI) with the vault path in scope, subscription-billed via existing CLI auth. A thin shell wrapper calls the Telegram Bot API directly via curl to deliver the result to the configured chat ID. No RC sessions or bot process involvement.

**Strengths:**
- Simplest substrate — VM cron survives any bot crash
- Reboot-survival is automatic via crontab or systemd timer
- Zero new dependencies beyond existing claude CLI
- Subscription-billed, matches billing constraint

**Novelty:** Fully stateless per-fire — treats Claude CLI as a Unix filter (stdin prompt, stdout response); no persistent agent sessions involved at scheduling time.

**Weakest link:** The Telegram side-channel delivery path — a separate curl-to-BotAPI path outside the bot process, no access to grammY formatting or session context; if BOT_TOKEN rotates the cron script breaks silently.

**Risks:**
- Parallel Telegram send path outside grammY — no formatting, no commit-confirm, no session context
- claude -p output may include tool-call artifacts without agent retry logic
- No catch-up for missed fires — cron skips windows that passed while VM was down

### V_cron_rc. VM cron + tmux send-keys into dedicated scheduler RC session

A crontab or systemd timer fires and uses `tmux send-keys` to inject a pre-formatted prompt into a long-running dedicated RC session named `scheduler`. The RC session processes the prompt through its full agent loop (vault tools, MCP, commit-confirm available) and uses a Telegram-send MCP tool to deliver the result. The scheduler session is kept alive by the same boot-resume logic in sessions.json used for Mode-2 /work sessions.

**Strengths:**
- Full agent context in scheduled session — vault tools, MCP, commit-confirm available
- Reuses existing session lifecycle (sessions.json + reaper + boot-resume)
- Delivery via same Telegram-send MCP tool used conversationally
- VM cron precision: ≤1 min jitter

**Novelty:** Reuses the verified tmux+RC session substrate in a scheduler role — no new invocation primitive; the novelty is the dispatch pattern (cron fires into a long-running RC session) rather than session architecture.

**Weakest link:** tmux send-keys injection is architecturally fragile — output capture from RC session requires polling or expect-style matching; any Claude Code update that changes terminal output format can break it silently.

**Risks:**
- tmux send-keys output capture is fragile; no clean stdout pipe from RC session
- Persistent scheduler session adds resource overhead
- RC session expiry at claude.ai/code could orphan the scheduler after extended downtime

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Natural extension: Mode-2 Telegram commands (/schedule, /remind) write to a scheduler-session dispatch queue, enabling user-defined schedules from Telegram without touching crontab.

### V_inprocess. In-process grammY scheduler (node-cron) + direct SDK call

Add node-cron to the bot process. On each fire, call the existing sendMessageStreaming() path for a designated scheduler user ID; result delivered via standard Telegram pipeline. Schedule config persisted in schedules.json (alongside sessions.json); on restart, re-register all active schedules. Missed-fire catch-up: if last_fired timestamp for a daily routine is >23h behind, fire immediately on startup.

**Strengths:**
- Full grammY context for delivery — formatting, inline keyboards, session state
- Missed-fire catch-up implementable with last_fired in schedules.json
- TZ-aware cron expressions (Europe/Moscow) supported by node-cron
- No VM cron or systemd timer required

**Novelty:** Bot owns its own clock — scheduling, dispatch, and Telegram delivery all inside the single Bun/grammY process; no OS-level scheduler or external session; the process is self-sufficient.

**Weakest link:** Bot crash correlation — if the bot crashes at 08:59 and restarts at 09:05, the daily reminder fires 5 min late; missed-fire catch-up logic depends on accurate last_fired timestamps surviving bot restart via atomic JSON writes.

**Risks:**
- Bot crash during a fire window loses that fire unless catch-up logic runs on restart
- schedules.json must be written atomically to avoid corruption
- Tighter coupling: scheduling logic adds complexity to an already multi-concern bot process

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** schedules.json + in-process scheduler is the natural substrate for 'remind me at X from Telegram' UX — user says it conversationally, bot writes a schedule entry, scheduler fires it. Zero VM cron management by the user.

### V_gcal. Google Calendar MCP polling + bot dispatcher

A polling loop (in-process or sidecar) calls Google Calendar MCP to fetch upcoming events tagged with a Claude action (e.g. CLAUDE_ACTION: daily-digest in event description). When an event fires in the polling window, the dispatcher runs the corresponding Claude action via existing SDK path and delivers to Telegram. User manages routine cadences in Google Calendar on their phone — no Telegram commands needed for schedule management.

**Strengths:**
- User-facing schedule management in Google Calendar — change timing on phone without touching bot config
- Bidirectional: agent can create calendar events for future reviews
- GCal free tier sufficient for 1-min polling volume
- Natural fit for 'what do I have today' part of the daily briefing

**Novelty:** Calendar-native scheduling — user manages routine cadences in Google Calendar (mobile-friendly, visual, supports ad-hoc overrides and vacation mode) rather than in Telegram commands or config files; agent can also write events back.

**Weakest link:** OAuth token lifecycle — GCal API requires a persistent refresh token; token rotation, revocation, or scope changes silently break all schedules. Polling jitter also adds imprecision vs cron.

**Risks:**
- OAuth refresh token must be stored securely and rotated correctly
- GCal API is a new external dependency and failure mode outside bot control
- Highest implementation effort — OAuth flow, MCP tool wiring, event-format convention
- Polling interval adds latency jitter on top of GCal's own propagation delay

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Once calendar integration is live, the agent can bi-directionally manage calendar events — opens 'schedule from Telegram by dictating a calendar event' and 'what do I have today' briefings without a separate query.
