# Solutions for: AbortError in SDK write path crashes bot on /stop and permission timeout

Problem: prob-20260422-002

## Variants (3)

### V1. Two targeted fixes in session.ts

Fix Bug 1: wrap abortController.abort() in try-catch inside stop() so SDK abort listeners can't crash the method. Fix Bug 2: remove the (queryCompleted || askUserTriggered || stopRequested) gate from the catch block — any abort/cancel error is always a lifecycle event, never a real failure. Optionally: harden config.ts TEMP_DIR setup with mkdirSync + try-catch instead of Bun.write.

**Strengths:**
- Smallest possible diff — easy to review and revert
- Directly maps each crash to its root cause
- No new concepts or patterns introduced

**Novelty:** Minimal invasive: two 3-line changes, no new abstractions, touches only the two exact lines that failed.

**Weakest link:** Still relies on string-matching ('abort'/'cancel') to classify errors as safe to suppress — a future SDK error mentioning 'abort' in its message would be silently swallowed.

**Risks:**
- Error-string matching is brittle if SDK error messages change

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** The softer catch condition is a prerequisite for any future structured abort-reason propagation.

### V2. Global process-level uncaughtException/unhandledRejection guard

Add process.on('uncaughtException') and process.on('unhandledRejection') handlers that catch AbortErrors before they crash the process. All other errors re-thrown or logged. This catches any escape path — not just the two known ones.

**Strengths:**
- Covers unknown future escape paths
- One addition rather than multiple edits

**Novelty:** Defensive at the process boundary rather than at the throw site — shifts responsibility from the SDK interaction to the process harness.

**Weakest link:** Global handlers can mask real bugs; harder to reason about in a crash post-mortem. Swallows AbortErrors even from code paths we haven't audited.

**Risks:**
- Over-broad: may hide real crashes that coincidentally mention 'abort'
- Anti-pattern: process-level exception handlers discouraged in production Node/Bun apps
- Harder to trace which code path produced the swallowed error

### V3. Semantic: set stopRequested=true when interrupt:true returned

In the canUseTool callback, after awaitPermission resolves, check if the result has interrupt: true and set this.stopRequested = true before returning. This makes the existing catch condition work correctly for the timeout case without loosening it. Also fix the stop() try-catch for Bug 1.

**Strengths:**
- Maintains original defensive posture of the catch condition
- Semantically correct: permission timeout IS a stop
- Easier to reason about in future audits

**Novelty:** Preserves the original intent of the suppression condition by correctly propagating interrupt intent to the stopRequested flag, rather than removing the guard.

**Weakest link:** Only fixes the permission-timeout trigger; does not handle other future sources of SDK-driven abort that don't go through canUseTool (e.g., a future SDK lifecycle event).

**Risks:**
- Partial fix: only covers the canUseTool interrupt path; still vulnerable to other SDK abort triggers
- Requires access to 'this' inside the arrow function (already available — it's a method closure)
