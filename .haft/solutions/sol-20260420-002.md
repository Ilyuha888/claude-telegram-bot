# Solutions for: Bot prompts for every tool call — no allowlist in canUseTool

Problem: prob-20260420-010

## Variants (3)

### V_allowlist. Static allowlist in canUseTool

Add an if-block at the top of canUseTool that auto-approves a fixed set of tool+pattern combinations before the prompt path: Write(*), Edit(*), MultiEdit(*), Bash(mv|mkdir|cp|find|touch) with path validated via existing isPathAllowed(). Still logs every auto-approved call. Destructive patterns (rm, git push, git commit) remain on prompt path. ~30 lines in session.ts.

**Strengths:**
- Minimal code change (~30 lines)
- No new state files or persistence
- Immediate fix for the batch-write case
- Audit log easy to add alongside

**Novelty:** Operates purely at tool-name+pattern level; no session state, no task file reads.

**Weakest link:** Pattern matching is static — any Bash command not in the allowlist still prompts; edge cases like quoted paths or subshell expansions can slip through the safety regex.

**Risks:**
- Pattern gaps mean some batch ops still prompt (e.g. git mv, sed, python scripts)
- All-or-nothing per pattern — no per-session scope concept

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Establishes the auto-approve path in canUseTool that V_batch_mode can build session-scoped policy on top of.

**Rollback:** Revert the if-block in session.ts. One-line change.

### V_batch_mode. Bot-level batch mode flag (task-file state)

Bot reads a lightweight state file (e.g. /tmp/claude-bot-batch.json) before deciding in canUseTool. User sends a Telegram command like /batch-start 'migrate vault' to set batch mode; bot auto-approves write-class tools scoped to the declared directory until /batch-stop or commit. Commit/push/destructive rm always gates regardless of batch mode. Summary shown at /batch-stop.

**Strengths:**
- Scope-bounded autonomy — user controls the envelope
- Explicit start/stop gives user mental model of what's auto-approved
- Resumable if state persisted to disk

**Novelty:** Adds explicit user-controlled session state to the permission layer — scope intent is declared by user, not inferred from tool calls.

**Weakest link:** Defining 'scope match' robustly — user sets /batch-start path=/vault but Claude may write to adjacent paths; false positives possible. Also, bot restart clears in-memory batch state unless persisted to disk.

**Risks:**
- New Telegram commands and state file management
- Scope ambiguity (partial path matches)
- Adds ~150 lines + new handler

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** State file pattern is the foundation for persistent grants (V_grants) without needing grants.yaml schema yet.

**Rollback:** Remove the batch-mode branch in canUseTool and the state file. Bot falls back to prompt-every-call.

### V_sdk_permissions. Use SDK permissionMode + allowedTools instead of canUseTool

Remove custom canUseTool callback entirely. Configure the Claude Code SDK with permissionMode: 'acceptEdits' (or equivalent) and pass an allowedTools list. Let the SDK handle auto-approval for the allow-listed set; canUseTool is only called for tools outside the list. AskUserQuestion interception moves to a separate SDK hook.

**Strengths:**
- No custom allowlist regex to maintain
- SDK-level guarantee (not bot-level guess)
- Cleaner separation: SDK owns permissions, bot owns UX

**Novelty:** Delegates approval policy to the SDK layer rather than implementing it in bot code — removes the maintenance surface entirely if the SDK supports it.

**Weakest link:** SDK permissionMode semantics are not well-documented for SubprocessCLITransport — 'acceptEdits' may not exist or may behave differently than expected; requires SDK research before committing.

**Risks:**
- SDK permissionMode options for Agent SDK may not support fine-grained tool allow-listing
- Losing canUseTool means losing AskUserQuestion interception (must find alternative hook)

**Rollback:** Re-add canUseTool block. SDK falls back to default permission behavior.
