# Solutions for: No mechanism to spawn per-repo remote Claude Code sessions from Telegram

Problem: prob-20260420-002

## Variants (5)

### A. tmux + git-worktree + SQLite session store

Bot runs: git worktree add ~/repos/<repo>/.worktrees/<name> <branch> (create or verify), then tmux new-session -d -s work-<slug> -c <target-path> 'claude'. Metadata in /home/assistant/bot-data/sessions.db (SQLite): id, repo, path, worktree, branch, tmux_name, created_at, last_activity_at, closed_at. /sessions = SQL query cross-checked with tmux ls. /attach returns SSH string + Cloud Code Remote cwd hint. /close = tmux kill-session + mark closed; git worktree untouched. Idle reaper: systemd timer (hourly) kills sessions where last_activity_at + 7d < now.

**Strengths:**
- Simplest attach path for Cloud Code Remote (ssh + tmux attach -t work-<slug>)
- ~2s spawn latency
- No new service type — tmux already common on the VM
- SQLite is lightweight and embeds naturally in bot process

**Novelty:** tmux as the session supervisor: sessions are persistent PTYs that any SSH client (including Cloud Code Remote) can attach to by name without reinvoking claude.

**Weakest link:** Activity proxy is imperfect: last_activity_at updated by which event? tty mtime, transcript file mtime, or explicit message-received timestamp each give different results. Must choose one and document the semantic.

**Risks:**
- Activity tracking accuracy
- tmux not crash-supervised — if tmux server crashes, all sessions vanish

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Variant B adds systemd transient units on top of the same tmux+worktree foundation; migrating A→B is additive, not destructive.

**Rollback:** Kill all work-* tmux sessions + drop sessions.db. No git state touched.

### B. systemd --user transient units per session

systemd-run --user --unit=work-<slug> --working-directory=<target> claude. List via systemctl --user list-units 'work-*' enriched with SQLite metadata. Logs to journald (persistent, greppable). /close = systemctl --user stop work-<slug>; worktree untouched. loginctl enable-linger assistant already active. Native idle timers possible via systemd OnIdleSec or wrapper.

**Strengths:**
- Native restart-on-failure policy
- Free log retention via journald
- cgroup resource accounting prevents runaway sessions
- loginctl enable-linger already active

**Novelty:** systemd as process supervisor: cgroup isolation, native crash-recovery policy, journald audit trail — production-grade ops hygiene without custom monitoring.

**Weakest link:** Transient units don't expose an interactive PTY; Cloud Code Remote attach requires ssh + cd <target> + claude --resume <session-id> rather than a simple tmux attach. Changes the UX of /attach.

**Risks:**
- Attach UX is 'reconnect by reinvoking claude' not 'attach to running PTY'
- 4 days impl vs 3 for A

**Rollback:** systemctl --user stop/disable all work-* units + drop sessions.db. Worktrees preserved.

### C. Claude Code native resume-only (no process supervisor)

No tmux, no systemd. Bot stores (session_id, cwd, worktree, branch) in JSON. /attach returns ssh+cwd+claude-resume-id. User invokes claude --resume <id> from Cloud Code Remote. No long-running process between attaches.

**Strengths:**
- ~1 day implementation
- Zero process management overhead

**Novelty:** Minimum-footprint approach: no supervisor, no persistent process, pure session-resume semantics.

**Weakest link:** Violates the acceptance criterion 'session survives Telegram disconnect / network interruptions' — there is no running process to survive. This is a bookmark system, not a session system.

**Risks:**
- Does not satisfy 'survives disconnect' requirement — ELIMINATED on constraint

### D. tmux + worktree + vault-committed markdown session ledger

Same tmux+worktree core as A, but session metadata stored as Agent_Obsidian_Vault/sessions/<id>.md YAML files auto-pushed via existing vault post-commit hook. Mirrors the session-state.md example in the task spec.

**Strengths:**
- Session metadata viewable from Obsidian on any device

**Novelty:** Vault-as-state-store: session metadata is git-synced, visible from any device that has the vault, no separate database.

**Weakest link:** Vault is a curated human knowledge surface, not an operational ledger; lifecycle churn (create/update/close) produces ~100 commits/week of noise. Post-commit push adds 2-4s latency to every /work spawn.

**Risks:**
- Vault noise
- Push latency on every lifecycle event
- Dominated by A on every perf dimension

### E. Cloud Code Remote native session API (deferred probe)

If Cloud Code Remote exposes a programmatic session-creation API on the VM agent, /work reduces to a lightweight API call; the bot owns zero process supervision state. Reattach is native to the Cloud Code Remote client.

**Strengths:**
- Zero process management in bot
- Native desktop UX for attach/resume if API exists

**Novelty:** Delegate session lifecycle entirely to Cloud Code Remote's native session management — bot is a thin API client, not a supervisor.

**Weakest link:** API surface unverified — Cloud Code Remote may only support desktop-as-creator and not programmatic spawning. Feasibility probe required before this variant can be scored.

**Risks:**
- Feasibility unknown — 2h probe required to verify API surface
- May be infeasible entirely
