# CTB Open-Source Release: Packaging, Licensing & KB Scaffold

## Signal

Bot is mature and useful in private use, but has no public release: no licensing audit (fork origin unknown), no Docker setup, personal KB data mixed with structure, and no clear separation into ctb + ctb-vault repos. Anyone wanting to adopt it has no path.

## Problem Type

synthesis

## Constraints

- No personal data ships in the public repo
- License must be legally clean relative to any fork origin
- Claude CLI dependency must be explicit — no pretending portability to other LLMs in v1
- Google MCP auth is user-session-scoped and cannot ship — document setup only

## Optimization Targets

- Time-to-running for a net-new user (target: clone → working bot in under 30 min)
- KB scaffold usefulness out of the box — structure + methodology ships, zero personal content
- Legal cleanliness — no GPL surprises post-release

## Observation Indicators

- Repo star growth (vanity metric — don't optimize for it)
- Issue volume (signals friction, not value)
- Fork count

## Acceptance

A person who has never seen the bot can read one README, authenticate Claude CLI with their own token, and have a working personal assistant running. All tracks in ctb-open-source-release.md are resolved: licensing audited, two-repo structure (ctb + ctb-vault) published, Docker config working, optional extensions documented.

## Blast Radius

Public users trying to adopt the bot; depends on license of original fork; affects what KB structure can ship publicly

## Reversibility

high
