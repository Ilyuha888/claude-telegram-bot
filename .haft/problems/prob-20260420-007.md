# Bot auto-resume after process restart — session orphaned on first post-restart message

## Signal

After claude-telegram-bot process restarts (systemctl restart, crash + auto-restart, deploy), the next user message creates a fresh Claude Code session instead of resuming the most-recent persisted one. /tmp/claude-telegram-session.json already stores an array of past sessions (session_id, saved_at, working_dir, title) and session.ts:639 has a resumeLast() method — but neither is auto-invoked on process start. this.sessionId in memory is null after restart, so query() is sent without resume:, orphaning the prior session. Diagnosed from 2026-04-17 journalctl: 3 fresh STARTING events on the same day, each following a systemd restart, never a RESUMING. Root cause of the original prob-20260420-004 misframing (which assumed token-limit pause).

## Problem Type

diagnosis

## Constraints

- Do not break existing /new and /resume UX — user may still want explicit session switching
- Must not auto-resume stale sessions — require a TTL guard (default 24h)
- working_dir of the resumed session must match current WORKING_DIR — skip if mismatch to avoid resuming stale-path sessions
- No schema change to /tmp/claude-telegram-session.json — use existing fields only

## Optimization Targets

- user_intervention_required — minimize prompting needed to get context-continuity after restart (target: 0 prompts, fully automatic)
- resume_latency — time from post-restart user message to resumed-session response (target: same as normal message latency)

## Observation Indicators

- False-positive auto-resume rate — user wanted /new but got /resume silently (watch for user issuing /new immediately after auto-resume notice)
- Session staleness — auto-resumed session is too old to be useful (TTL covers this but watch edge cases)

## Acceptance

After bot restarts, user sends a normal message (no /resume). If getSessionList()[0].saved_at is within 24h and working_dir matches WORKING_DIR, bot resumes silently and prefixes reply with a one-line notice ('↩︎ resumed: <title>'). If session is older than 24h or dir mismatches, falls back to /new as today. Existing /new and /resume commands continue to work exactly as before.

## Blast Radius

src/session.ts only; no schema changes; no systemd changes; bot restart required to deploy

## Reversibility

high — revert the tryAutoResume() call to restore original behavior
