# Telegram attachments land ephemerally in /tmp — can't be persisted into the vault alongside a note

## Signal

When a user sends a photo/document via Telegram, the bot downloads the file to /tmp/telegram-bot/ and inlines its bytes as an SDK content block (dec-20260420-001) so the agent can see it for the current turn. But the agent cannot copy the file INTO the vault: the inbound path is ephemeral, /tmp/telegram-bot/ is outside ALLOWED_PATHS, and no handler persists the bytes alongside a note's files/ directory. Concrete recent miss: user wanted BLS visa payment receipt photos attached to 2026-06-06-check-portugal-visa-status.md — had to manually transcribe the data instead. Distinct from prob-20260422-001 (outbound agent→user) and dec-20260420-001 (inbound-ephemeral); this is inbound persistence. Task spec: Agent_Obsidian_Vault/tasks/2026-04-28-bot-image-file-attachment.md.

## Problem Type

synthesis

## Constraints

- Path validation must hold — every write path checked by isPathAllowed() from dec-20260420-003
- Subscription billing untouched (dec-20260416-003 admissibility; dec-20260414-001)
- No new egress — download via existing api.telegram.org allowlist path (dec-20260416-001)
- Credential-protection posture untouched — settings.local.json chmod 600, .env, guard-sensitive-reads.py scope all preserved
- No shell escape / raw subprocess in the runtime LLM path (AGENTS.md §Safety Rules)
- Denylist model preserved — no relaxation of BASH_DENY_RE from dec-20260420-003
- Writable surface scoped — only the vault's files/ subdirectories become writable, not new top-level ALLOWED_PATHS
- Single-user scope (dec-20260320-001) — no multi-recipient fan-out
- Commit-confirm flow for User_Obsidian_Vault writes stays intact — large attachment commits go through the same gate
- Content scope v1: photos (sendPhoto/Telegram photo type) and documents (sendDocument) — audio/video deferred

## Optimization Targets

- Attachment persistence reliability — photo/document saved to vault when user requests it
- Note-attachment linkage clarity — note markdown includes a working reference to the saved file
- Multi-file batch handling — one Telegram message or media group produces one batch write and one commit

## Observation Indicators

- Vault repo disk growth per attachment cohort (watch, don't optimize)
- Per-attachment download latency from api.telegram.org
- Attachment format coverage — which image/document types succeed vs fail silently

## Acceptance

User sends one or more photos/documents via Telegram with an instruction naming a target note (e.g., "attach these receipts to the visa status note"). Bot (a) downloads each file from Telegram to a vault-writable path, (b) places it under the target note's files/ sibling directory using a deterministic stable filename, (c) updates the target note with a markdown image reference or wikilink to each attached file, (d) commits the vault change through the existing commit-confirm flow, (e) confirms in Telegram with the list of saved paths. Multi-file messages (Telegram media groups) handled as a single batch. Zero regressions in inbound content-block flow, text path, or voice path.

## Blast Radius

linuz90 bot handlers (src/handlers/photo.ts, src/handlers/document.ts), vault filesystem (new files under Agent_Obsidian_Vault/**/files/ and User_Obsidian_Vault/**/files/). No MCP, no egress posture changes, no runtime LLM path. Zero impact on inbound content-block flow (dec-20260420-001) or outbound text path.
