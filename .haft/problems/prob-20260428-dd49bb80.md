# Mode 2 Remote Session Spawner: /work Command Implementation

## Signal

Mode 2 (/work, /sessions, /attach, /close) is fully designed in dual-mode-telegram-bot.md but not implemented. The mobile-to-desktop handoff workflow — core value proposition for deep repo work — is unavailable. Mode 1 works; Mode 2 is a stub.

## Problem Type

search

## Constraints

- Mode 2 commands must NOT spawn LLM sessions — session lifecycle is mechanical (parse, validate, create worktree, return metadata)
- Must not pollute Mode 1 conversational state — no shared session file
- Worktree names must be validated before creation to prevent typo-spawned branches
- /close must be explicit — no auto-cleanup of worktrees the user didn't ask to delete

## Optimization Targets

- Correctness of worktree lifecycle (create → attach → close without state drift)
- Latency of /work command response (parse + SSH + worktree + return metadata)
- Clarity of /sessions output — user should know exactly how to connect from MacBook

## Observation Indicators

- Number of accidental worktrees created (signals UX confusion)
- Mode 1 regression rate after Mode 2 is added

## Acceptance

From Telegram mobile: /work data-style path wt-x branch → session created on VM. /sessions lists it. /attach returns a usable Cloud Code Remote connection string. User opens MacBook, connects, and has the correct repo/branch context. /close shuts it down cleanly. Mode 1 has zero regressions.

## Blast Radius

Only additive — Mode 1 is unchanged. Affects how worktrees are named, what bot-data files grow, and how Cloud Code Remote sessions are tracked.

## Reversibility

medium
