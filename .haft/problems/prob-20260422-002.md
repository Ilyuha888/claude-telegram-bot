# AbortError in SDK write path crashes bot on /stop and permission timeout

## Signal

Two distinct triggers both crash the bot via the same unhandled AbortError in @anthropic-ai/claude-agent-sdk handleControlRequest → write(): (1) /stop command with a response in-flight (Apr 18 14:35), (2) 900s permission timeout auto-deny sending an interrupt (Apr 20 17:05). In both cases the abort controller fires, the stream is marked aborted, but a subsequent write() call throws AbortError which propagates to top-level and kills the process. Any unanswered permission prompt left for 15+ minutes will crash the bot.

## Problem Type

diagnosis

## Constraints

- Do not modify sdk.mjs (third-party vendored code)
- Fix must not suppress legitimate abort signals — abort should still stop the Claude session cleanly
- Bot must restart cleanly via systemd Restart=on-failure after the fix (or not crash at all)
- No new production dependencies

## Optimization Targets

- Eliminate bot crash on permission timeout and /stop-with-in-flight-response
- Preserve clean abort/stop semantics (session terminates, user notified)

## Observation Indicators

- Journal crash rate post-fix (target: zero AbortError exits)
- Session resume correctness after /stop (should not orphan)

## Acceptance

Zero AbortError-caused exits in journal over 7 days post-deploy. /stop and permission timeout still terminate the session cleanly without crashing the process.

## Blast Radius

Bot source only — handlers/commands.ts or session.ts. systemd unit unchanged. No vault or Claude Code config impact.

## Reversibility

high — targeted TypeScript change, revert via git
