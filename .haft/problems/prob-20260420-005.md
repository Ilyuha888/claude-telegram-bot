# Telegram message delivery semantics and idempotency unverified under forward/edit/rapid scenarios

## Signal

The bot uses a tg-in:<bot_id>:<update_id> idempotency model (referenced in yet_another_personal_assistant spec) but behavior under three scenarios is undocumented: (1) forward+comment arrives as one update or two, (2) user edits a sent message — treated as new message or ignored, (3) rapid-succession messages — delivered in-order or coalesced. If edits are not isolated by idempotency key, a malicious or mistaken edit could bypass deduplication and trigger second processing of a modified message. This is a probe-first problem — comparison basis is incomplete without test evidence. Detailed investigation spec in Agent_Obsidian_Vault/tasks/2026-04-20-investigate-telegram-message-batching.md.

## Problem Type

diagnosis

## Constraints

- Investigation only — no code changes until evidence is gathered
- Must reuse existing Telegram webhook handler rather than replacing it
- Must not require bot restart to activate any idempotency key changes

## Optimization Targets

- idempotency_coverage — all delivery scenarios confirmed handled
- edit_attack_surface — zero edit-based bypass paths through idempotency layer

## Observation Indicators

- Telegram API rate limits during test (don't saturate during investigation)
- Bot uptime during test scenarios (don't disrupt live service)

## Acceptance

Documented delivery semantics for all three scenarios (forward+comment, post-send edit, rapid succession); test harness verifying idempotency key holds under each; explicit decision on whether edited_message updates should be processed as new messages or ignored; security analysis confirming edit-as-attack-vector is blocked.

## Blast Radius

## Reversibility

high — probe phase; any code changes from findings will be tactical and isolated to webhook handler
