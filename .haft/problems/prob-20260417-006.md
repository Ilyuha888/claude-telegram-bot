# No skill invocation UX in Telegram — all skills invisible to user

## Signal

Claude Code has a rich skill library (haft reasoning: /h-status, /h-reason, /h-frame; engineering: /commit, /design-solution, /pr-creation, /ai-engineer, /execute-plan; PKM: /h-note, /h-verify; and more). All work when typed verbatim in Telegram, but Telegram has no knowledge of any of them. No autocomplete, no slash command menu, no affordance. User must know exact syntax. Discovery is zero — the entire skill surface is invisible unless already known.

## Problem Type

synthesis

## Constraints

- Must not break existing free-text Telegram message flow
- Telegram slash commands registered via BotFather are a static list — not dynamic or runtime-queryable
- Skills are prompt-level text injection into Claude Code subprocess — no separate process to hook
- linuz90/claude-telegram-bot is the bridge — changes must be confined to it or BotFather registration
- Non-priority — backlog only

## Optimization Targets

- User can discover and invoke any skill from Telegram with slash command autocomplete or a browsable menu — not just haft, but commit, design-solution, pr-creation, ai-engineer, execute-plan, and the full skill surface
- No new runtime dependency required on the VM

## Observation Indicators

- Free-text message flow must not degrade — skill shortcuts are additive, not replacing natural language

## Acceptance

User types / in Telegram and sees a browsable list of available skills with one-line descriptions. Tapping one sends the correct invocation to Claude Code. Full skill surface discoverable — not just haft subset.

## Blast Radius

Telegram bridge config (BotFather command registration) + optionally linuz90/claude-telegram-bot message handler. Vault and Claude Code runtime untouched.

## Reversibility

high
