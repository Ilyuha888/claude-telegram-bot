# After token-limit pause, user must manually re-supply context to resume work

## Signal

When Claude Code hits its context limit during a long task and pauses (or when a Telegram bot session is interrupted), the returning user has no auto-resume mechanism. They must manually re-orient the assistant by typing 'what were we doing?' or providing context. Continuation notes referenced in vault (e.g., 2026-04-17-vault-migration-plan.md) were managed manually. This is orthogonal to scheduled task execution (prob-20260416-006 covers deferred/cron-style execution; this problem is about in-conversation continuity). Session state schema in this problem should be compatible with Mode-2 spawner session metadata to avoid schema divergence. Detailed spec in Agent_Obsidian_Vault/tasks/bot-improvement/2026-04-18-improve-session-persistence.md.

## Problem Type

optimization

## Constraints

- One active task in session state at a time (no multi-task session state)
- .session-state.md YAML schema must be compatible with Mode-2 spawner session metadata (ADR-008 field naming baseline)
- Must not require root access on VM
- State stored in Agent_Obsidian_Vault/ (git-synced, cross-device)

## Optimization Targets

- resume_latency — time from user return message to auto-resume offer (target: immediate)
- user_intervention_required — minimize prompting needed to trigger resume detection

## Observation Indicators

- Session state file accumulation (watch for un-archived stale states)
- Resume offer false positive rate (don't over-trigger on clean sessions with dirty git trees)

## Acceptance

User hits context limit, returns 3+ hours later, bot auto-detects paused work from .session-state.md and/or git dirty tree, offers resume summary without user prompt. Works cross-device via git sync. State expires and is archived after 7 days of inactivity.

## Blast Radius

CLAUDE.md auto-load logic, new Agent_Obsidian_Vault/.session-state.md file, optional vault CLAUDE.md update; no bot infrastructure changes required for phase 1

## Reversibility

high — new file addition plus CLAUDE.md update; fully reversible by removing the file and reverting CLAUDE.md

## Characterization v1 (2026-04-20)

| Dimension | Role | Scale | Unit | Polarity | Measurement |
|-----------|------|-------|------|----------|-------------|
| resume_latency | target | ratio | seconds from user return message to auto-resume offer displayed | lower_better | Test: pause mid-task, return after 3h, measure time from first message to resume offer (3 trials median) |
| user_intervention_required | target | ordinal | 0–3 (0=full manual context re-supply, 1=user must ask 'resume?', 2=auto-offer but user confirms, 3=fully automatic offer without any prompt) | higher_better | Test: return after pause without any context-supplying message; verify bot auto-detects and offers resume |
| cross_device_sync | target | ordinal | 0–3 (0=session state not synced, 1=manual git push required, 2=synced via post-commit hook, 3=auto-synced on state write) | higher_better | Write .session-state.md on VM, verify it appears on local dev machine after git sync |
| state_accuracy | target | ordinal | 0–3 (0=stale or wrong state, 1=task name accurate but steps stale, 2=task + next steps accurate, 3=task + steps + git dirty tree correlation accurate) | higher_better | After pause, compare resumed task description to actual git status; verify .session-state.md correctly reflects last checkpoint |
| state_accumulation_risk | target | ordinal | 0–3 (0=unbounded files, 1=manual cleanup, 2=7-day TTL on state file, 3=auto-archive on task completion + TTL) | higher_better | After 2 weeks of usage, count stale un-archived state files in Agent_Obsidian_Vault/ |

**Parity rules:** All variants must be evaluated with real token-limit pauses (not simulated). Cross-device sync test must use the actual VM-to-local git sync path, not a local-only test. State accuracy measured against actual git status, not the state file's self-report.
