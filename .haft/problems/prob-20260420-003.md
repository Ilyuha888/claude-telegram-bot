# Single model tier wastes subscription budget or degrades planning quality

## Signal

The personal assistant runs all interactions on one model tier (Sonnet 4.6 as of 2026-04-17). Running Opus for vault lookup questions wastes the corporate subscription budget. Running Sonnet-only for multi-file refactoring or architectural planning degrades plan quality compared to Opus 4.7. No mechanism exists to automatically route to the appropriate tier (Opus plan / Sonnet execute / Haiku cheap queries) nor to signal which phase of work is active. Detailed spec in Agent_Obsidian_Vault/tasks/2026-04-20-opus-plan-mode-remote.md. Gated on corporate subscription switch (tasks/2026-04-20-cloud-code-subscription-switch.md).

## Problem Type

optimization

## Constraints

- Corporate plan must expose Opus 4.7 + Sonnet 4.6 + Haiku 4.5 (prerequisite: cloud-code-subscription-switch.md)
- No manual tier selection required for common flows — auto-routing required
- Model switching must be transparent to user (no interruption for model handoff)
- Cost tracking must be auditable on request

## Optimization Targets

- cost_efficiency — correct model tier assigned per task type
- planning_quality — Opus engaged for all architectural and multi-file reasoning tasks

## Observation Indicators

- Opus token spend per week (watch — don't let routing optimization game the metric)
- Plan mode false positive rate (don't optimize at cost of spurious plan triggers on simple tasks)

## Acceptance

Complex tasks auto-enter Opus plan mode via EnterPlanMode heuristic (keywords + file count); post-approval execution runs on Sonnet; simple vault reads and single-file queries use Haiku; token usage logs confirm Opus confined to plan phase; weekly budget does not exceed personal-plan baseline at comparable workload.

## Blast Radius

CLAUDE.md routing heuristics, /execute-plan skill, Claude Code settings on VM (dec-20260414-001); does not affect vault contents, bot infrastructure, or SSH access surface

## Reversibility

high — config-only changes, fully reversible by restoring previous settings.json

| Dimension | Role | Scale | Unit | Polarity | Measurement |
|-----------|------|-------|------|----------|-------------|
| cost_efficiency | target | ordinal | 0–3 (0=single-tier all tasks, 1=manual tier selection, 2=semi-auto heuristic, 3=fully auto with correct assignment) | higher_better | Token log analysis: what fraction of Opus tokens were spent on plan-phase tasks vs trivial queries over 1 week |
| planning_quality | target | ordinal | 0–3 (0=no plan mode, 1=manual plan trigger only, 2=auto-trigger on keywords, 3=auto-trigger on complexity + keyword + file count) | higher_better | Test: 5 complex multi-file tasks, verify each auto-enters plan mode; 5 simple KB queries, verify none trigger plan mode |

**Parity rules:** All variants must be evaluated on the same 10-task test set (5 complex + 5 simple) under corporate subscription. Auto-switching heuristics must be evaluated against cold sessions (no prior context bias). Budget visibility measured independently of routing quality.
