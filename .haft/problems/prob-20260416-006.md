# No scheduled task execution from Telegram when using ephemeral bot sessions

## Signal

V_linuz90 (and any Agent SDK bot) spawns ephemeral Claude subprocesses per conversation — they terminate after the response. There is no persistent Claude Code session owned by the bot. This means: (1) "remind me at X" requests via Telegram have no persistent state to fire the reminder; (2) Claude Code's CronCreate/RemoteTrigger features are only accessible in the main tmux session, not from bot-spawned subprocesses; (3) scheduled vault summaries, daily briefings, or calendar-driven actions cannot be set up conversationally and executed later without a separate mechanism.

## Problem Type

search

## Constraints

- Must use Claude subscription billing — no API pay-per-token
- Scheduled task execution must survive VM reboot
- Must not require a permanently running interactive Claude Code session (that would negate the V_linuz90 migration benefit)
- ~$10/mo budget for non-Claude API costs

## Optimization Targets

- Ability to schedule tasks conversationally from Telegram (e.g. 'every morning summarize my vault')
- Calendar-driven triggers (Google Calendar event → Claude action)

## Observation Indicators

- Implementation complexity — cron-spawned sessions vs calendar MCP integration vs hybrid
- Latency tolerance — is near-real-time trigger needed or is cron-level (1-min) precision sufficient

## Acceptance

A working scheduled task that: (a) is set up conversationally from Telegram OR via calendar event, (b) executes a Claude action at the right time, (c) sends the result back to Telegram, (d) survives VM reboot.

## Blast Radius

start-assistant.sh, systemd service, potentially a new cron daemon or Google Calendar MCP integration

## Reversibility

medium
