# Long agent messages split mid-HTML-tag, breaking Telegram rendering

## Signal

sendChunkedMessages() in src/handlers/streaming.ts splits content at a raw character offset (TELEGRAM_SAFE_LIMIT = 4000) with no awareness of HTML tag boundaries. If a split lands inside a tag — <b>, <pre>, <tg-spoiler>, etc. — Telegram's HTML parser rejects the chunk and falls back to plain text, losing all formatting for that segment. The <thinking>-block spoiler added today makes this worse: long thinking blocks now appear in the text stream and are prime candidates for mid-tag splits. No prior fix exists in git history.

## Problem Type

diagnosis

## Constraints

- Each chunk must be ≤ 4096 chars (Telegram hard limit)
- Short-message path must remain unchanged
- Must not introduce noticeable latency

## Optimization Targets

- Split at paragraph or line boundary ≤ 4000 chars
- Close all open HTML tags before each chunk boundary

## Observation Indicators

- Chunk count per long message
- Plain-text fallback rate (Telegram parse errors in logs)

## Acceptance

Messages of any length render with correct HTML formatting in all segments. No Telegram parse errors from split tags. Plain-text fallback rate is zero for well-formed content.

## Blast Radius

src/handlers/streaming.ts sendChunkedMessages(). All long responses: scheduler notifications, Curator output, heavy tool traces, thinking spoilers.

## Reversibility

high
