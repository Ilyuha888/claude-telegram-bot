# VM security hardening: root user, no network controls, no .claudeignore

## Signal

Claude Code runs as root on the Hetzner VM with no network egress controls, no .claudeignore file, and no --disallowedTools restrictions. Research doc identifies this as a meaningful risk for a self-hosted personal assistant: root means any tool execution has full filesystem and network privileges; no outbound restrictions means tools could exfiltrate data to arbitrary endpoints; no .claudeignore means Claude can read ~/.ssh/id_rsa, .env files, credentials; no --disallowedTools means all built-in tools are available including destructive ones. Threat model: self-hosted single-user VM, not multi-tenant — but prompt injection via vault content or dependency READMEs is a realistic attack surface.

## Problem Type

optimization

## Constraints

- Must not break existing assistant functionality
- Single-user scope — no need for multi-user access controls
- Changes must survive systemd restart
- Git push credentials (GitHub PAT) must remain functional

## Optimization Targets

- Run Claude Code as a dedicated non-root user with minimal privileges
- Restrict outbound network to known endpoints (api.anthropic.com, api.telegram.org, github.com)
- Add .claudeignore for sensitive paths and --disallowedTools for dangerous patterns

## Observation Indicators

- Whether any legitimate workflow breaks after hardening

## Acceptance

Claude Code runs as a dedicated non-root user; .claudeignore exists and excludes ~/.ssh, .env, credential files; --disallowedTools is configured; UFW denies all inbound except SSH. All existing workflows (vault read/write, git push, Telegram messaging) still function.

## Blast Radius

VM system configuration, start-assistant.sh, systemd unit file, .claudeignore (new file on VM), settings.local.json on VM
