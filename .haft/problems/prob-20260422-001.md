# Agent cannot deliver images or files to the user via Telegram — outbound is text-only

## Signal

The Telegram bridge (linuz90/claude-telegram-bot, dec-20260416-003) relays Claude Code's final output as MarkdownV2 text and intercepts AskUserQuestion via canUseTool, but the reply surface is text-only. When the user asks for a vault image, a rendered diagram, an exported document, or a haft artifact as a file, the agent can only: (a) paste a filesystem path the user cannot open from Telegram, (b) read and inline-quote the content as text (loses images entirely, mangles structured docs), or (c) abandon the request. Concretely: user asks "send me the architecture diagram" → agent replies with a vault path string. User asks "give me that PDF" → no pathway exists. The asymmetry is notable: inbound images and PDFs already land as SDK content blocks (dec-20260420-001), so the user can send media to the agent but not receive it. send_file was cited as a V_linuz90 selection reason in dec-20260416-003 line 49, but line 82 confirms it reflects TS Agent SDK capability, not an active wired-up feature.

## Problem Type

synthesis

## Constraints

- Subscription billing untouched — do not route through ANTHROPIC_API_KEY (dec-20260416-003 admissibility; dec-20260414-001)
- No shell escape — no subprocess / raw shell added to the runtime LLM path (AGENTS.md §Safety Rules; ARCHITECTURE.md invariant)
- Path validation enforced — every file read for outbound must pass isPathAllowed() (ALLOWED_PATHS) before the bot sends it; isPathAllowed() invariant from dec-20260420-003 must not be bypassed or weakened
- Denylist model preserved — no relaxation of the Bash denylist in dec-20260420-003 (sudo|git push|commit|reset|rebase|clean)
- Writable surface unchanged — do not widen ALLOWED_PATHS, do not grant new write capabilities (AGENTS.md §Safety Rules)
- Egress posture — api.telegram.org is already on the implicit outbound allowlist (dec-20260416-001 admissibility); fetching external URLs to attach is out of scope; WebFetch/WebSearch remain disallowed
- Two-way Telegram operational at all times — existing text + AskUserQuestion paths must not regress (dec-20260416-003 invariant)
- Credential protection — settings.local.json (chmod 600), .env, and paths covered by guard-sensitive-reads.py must not become reachable via the outbound file path even when they reside in an otherwise-allowed directory
- Single-user — no multi-recipient fan-out (dec-20260320-001)
- Content scope (user-confirmed 2026-04-22): images (sendPhoto) and documents (sendDocument) only; audio/video deferred; external URL fetch-to-attach deferred
- Sources (user-confirmed 2026-04-22): vault files/ sibling dirs, arbitrary paths passing isPathAllowed(), agent-generated mid-session artifacts written to an allowed path
- Initiator (user-confirmed 2026-04-22): explicit user request AND agent-proactive — agent may attach when it judges attachment is the right reply shape

## Optimization Targets

- Delivery succeeds at all — user receives a renderable image or properly-named downloadable document in Telegram, not a path string
- Round-trip latency — perceived latency from user request to file arrival comparable to current text replies; typing indicator used during send
- Intent fidelity — agent-proactive sends attach the right file (the one the response is about), not a wrong or stale artifact
- Safety legibility — every send is auditable: tool name, resolved absolute path, file size, isPathAllowed() verdict, Telegram chat id logged

## Observation Indicators

- Send volume / file count — more files sent is not better; do not reward the agent for attaching things
- File size — do not minimize at the cost of information (e.g., compressing a diagram to illegibility)
- Tool invocation count — raw send_file calls per session is a throughput proxy, not a quality signal

## Acceptance

All of the following hold, verified with evidence: (1) User can ask (in Telegram) for a named vault image or document and receive it inline in chat — image preview / downloadable file with original filename and extension preserved. (2) Agent can attach an image or document proactively when it is the correct reply shape, and the attached file matches the message content — manually verified on ≥3 proactive sends. (3) Every successful outbound file send logs: tool name, resolved absolute path, file size, isPathAllowed() verdict, Telegram chat id. (4) Negative test: an attempt to send a path outside ALLOWED_PATHS (or a sensitive path like .claude/) is refused before the Telegram API call, and the refusal surfaces back to the agent as a tool error — not a silent drop. (5) Existing text-reply and AskUserQuestion paths continue to work — no regression in a 1-week live window. (6) File size limits handled gracefully: >10MB for sendPhoto and >50MB for sendDocument produce a clear error, not a crash.

## Blast Radius

linuz90 bot codebase only: src/session.ts (canUseTool branch), src/handlers/*.ts (new handler), src/config.ts (allowlist entry for new tool name). Potentially settings.local.json on VM if a new tool name must be declared. No VM write expansion, no systemd change, no vault repo change, no ALLOWED_PATHS widening.
