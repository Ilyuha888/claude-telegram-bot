# Bot conflates knowledge-work conversation with infrastructure commands

## Signal

Current Telegram bot has one interaction mode — all user input enters the conversational LLM context regardless of whether the user intends knowledge work (Mode 1) or infrastructure management (Mode 2: spawning remote sessions, worktrees, repo navigation). No predictable surface cue distinguishes the two. Infrastructure commands pollute the knowledge-work context, and Mode-2 invocations require knowledge of hidden syntax. Directly related to prob-20260417-006 (skill invocation UX invisibility) and the vision captured in Agent_Obsidian_Vault/notes/2026-04-20-dual-mode-telegram-vision.md.

## Problem Type

synthesis

## Constraints

- Preserve Obsidian Sync mobile access (no vault path changes)
- Subscription-billing compatible — no API-token model (dec-20260416-003)
- Single-user scope (dec-20260320-001)
- No terminal passthrough from Telegram to VM shell
- Mode-2 must be stateless (menu only, no LLM reasoning)

## Optimization Targets

- ux_clarity — user ability to predict active mode without reading docs
- mode_separation — zero context bleed between Mode 1 and Mode 2

## Observation Indicators

- Telegram session memory consumption per mode (watch, not optimize)
- Mode-2 command dispatch latency (don't sacrifice correctness for speed)

## Acceptance

User can predict active mode from bot surface cues; Mode-2 menu commands never produce LLM reasoning responses; mobile-to-desktop handoff preserves session context; pilot test with 10 diverse requests shows correct mode routing with zero bleed between modes.

## Blast Radius

linuz90/claude-telegram-bot TypeScript codebase and CLAUDE.md routing logic; does not touch Obsidian vault contents or VM security surface
