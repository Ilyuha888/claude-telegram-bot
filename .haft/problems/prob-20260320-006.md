# Single-user scope is an implicit assumption, not a recorded constraint

## Signal

The entire platform is designed for one user but no ADR or architecture document declares this as a hard constraint. Any future agent or contributor can inadvertently design for multi-user scenarios (shared vault, per-user session isolation, concurrent approval flows) without triggering a deliberate review. ADR-006's staging complexity is partly justified by multi-editor risk avoidance — yet the risk does not exist. The constraint needs to be explicit so that lifting it requires a conscious decision, not incremental drift.

## Constraints

- Recording the constraint must not lock out a future multi-user path — it must just make expanding that path explicit and deliberate
- The constraint must appear in a durable location checked by agents before significant work (AGENTS.md or a quint decision)

## Optimization Targets

- Make the single-user constraint discoverable and machine-checkable
- Ensure any decision that implicitly relies on it cites it

## Observation Indicators

- Number of ADRs/decisions that would need revision if multi-user support were added

## Blast Radius

High — an unrecorded constraint is an invisible one. Every service boundary decision (session isolation, vault write policy, scheduler, Git approval flow) is implicitly shaped by single-user scope. If this assumption silently breaks, all those decisions are invalidated simultaneously.

## Reversibility

high
