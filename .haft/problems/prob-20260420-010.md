# Bot prompts for every tool call — no allowlist in canUseTool

## Signal

Telegram bot's canUseTool callback routes every tool invocation (Write, Edit, Bash, Read, MultiEdit) to a Telegram Allow/Deny prompt. No allowlist is implemented. Multi-file operations (vault migrations, bulk renames, refactors) generate one prompt per file/operation — a 100-file migration produces ~100 sequential taps. User flow breaks. An earlier dec-20260416-003 measurement described Write/Edit/MultiEdit auto-approval and expanded Bash patterns as "implemented", but the current bot code has no such logic — the claim was never committed.

## Problem Type

synthesis

## Constraints

- Security invariants of dec-20260416-001 must hold: reversibility gate (commit/push/destructive rm always gated per-op), no secrets in repo, non-root service
- ALLOWED_PATHS path restriction in security.ts must remain enforced — auto-approval of a tool does not bypass the path check
- Irreversible operations (git push, rm -rf, destructive delete) must NEVER be auto-approved
- No persistent grant state that silently persists across sessions without user-visible audit trail

## Optimization Targets

- Reduce per-batch prompt count to ≤ 2 (scope intent + commit approval) for ≥50-file operations
- Zero unintended auto-approvals on destructive operations over any 2-week measurement window

## Observation Indicators

- Total permission prompts per day — watch for creep after allowlist lands (Goodhart risk: optimizing for fewer prompts by silently expanding scope)
- Audit log completeness — every auto-approved tool call should still be logged

## Acceptance

Measured on a ≥50-file vault migration replay: total Telegram prompts ≤ 2. Audit log shows every auto-approved tool call. Zero destructive operations auto-approved over 2-week observation window.

## Blast Radius

Bot-local: src/session.ts, src/handlers/permission.ts. No VM system config touched. Rollback = revert session.ts canUseTool block.

## Reversibility

high
