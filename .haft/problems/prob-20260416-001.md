# No multi-session or multi-repo support from Telegram

## Signal

The VM runs a single Claude Code session tied to one repo (the vault). The user cannot spawn new sessions from Telegram, switch between sessions, or choose which repo a session starts in. As more repos are added (side projects, work code), Telegram becomes a bottleneck — all messages go to one context. Research evidence: --add-dir enables multi-repo context within a session; Agent SDK query() supports per-project sessions with different cwd; community bots (RichardAtCT) implement multi-project topic threads. All approaches require leaving the official Telegram plugin. Subscription billing constraint satisfied: Agent SDK supports subscription via CLAUDE_CODE_OAUTH_TOKEN (GH anthropics/claude-agent-sdk-python#559, confirmed working); CLI-subprocess approaches use subscription natively.

## Constraints

- Single-user scope (dec-20260320-001)
- Telegram remains the primary access channel
- Each session must have its own working directory and context
- Must not break existing vault assistant workflow
- Must stay on Claude subscription billing (not API pay-per-token)

## Optimization Targets

- Ability to spawn independent Claude sessions per repo from Telegram
- Ability to switch active session from Telegram without SSH

## Observation Indicators

- Session startup latency
- VM resource usage with multiple concurrent sessions

## Acceptance

User can message the bot to list active sessions, spawn a new session in a chosen repo, and switch which session receives subsequent messages — all from Telegram.

## Blast Radius

start-assistant.sh, Telegram plugin config, possibly a thin session manager script or systemd template units
