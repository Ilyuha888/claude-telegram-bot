# edited_message updates silently dropped — no handler registered, edit-attack surface is zero

## Rationale

Source audit of /home/assistant/claude-telegram-bot/src/index.ts shows bot registers handlers for message:text, message:voice, message:photo, message:document, message:audio, message:video, message:video_note, callback_query:data, and /commands. No bot.on("edited_message:text") or equivalent. grammY delivers edited_message as a distinct update type — without a matching handler it is silently dropped. Confirmed by grep: zero matches for edited_message, editedMessage in all src/ files. Consequence: a user who edits a previously-sent message gets no bot response and Claude never sees the edit. The security concern from prob-20260420-005 (edit bypasses idempotency, mutated instruction re-executed) is moot — there is no re-execution path at all. UX gap: user has no feedback that the edit was ignored.

## Evidence

index.ts handler registration (2026-04-20 read): message:text, message:voice, message:photo, message:document, message:audio, message:video, message:video_note, callback_query:data, commands. No edited_message entry. grep -r edited_message /src: empty output.
