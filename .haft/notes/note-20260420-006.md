# Edit policy: silently ignore edited_message — deliberate, secure, UX gap noted

## Rationale

prob-20260420-005 acceptance requires explicit decision on whether edited_message should be processed as new instructions or ignored. Current behavior is: ignore (no handler registered). This is the correct policy for three reasons: (1) Telegram edit semantics are ambiguous — the user may be fixing a typo, not retrying an instruction; (2) re-executing a mutated instruction without user confirmation would create an implicit "edit to change my order" pattern that bypasses the normal message flow; (3) single-user scope means the attack-surface argument is weak, but the design simplicity argument is strong. UX gap: user gets no feedback when edit is dropped. Optional improvement: register a bot.on("edited_message:text") handler that replies "Your edit was noted but not processed — send a new message to continue." Not blocking; log as potential UX issue.

## Evidence

src/index.ts 2026-04-20: no edited_message handler. Confirmed by grep across all src/ files. This is the status quo and the recommended ongoing policy.
