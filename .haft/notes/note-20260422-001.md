# prob-20260420-008 finding: restart source classified — operator churn dominant, two AbortError crash bugs identified

## Rationale

Investigation of 7-day journal (Apr 16–22) for claude-telegram-bot.service:

**Restart distribution (by category):**
| Date | Total starts | Crash-induced | Operator/clean |
|------|-------------|---------------|----------------|
| Apr 16 | 3 | 0 | 3 |
| Apr 17 | 13 | 2 (auto) + 1 (manual after) | 10 |
| Apr 18 | 1 | 1 | 0 |
| Apr 19 | 0 | — | — |
| Apr 20 | 13 | 1 | 12 |
| Apr 21 | 2 | 0 | 2 |
| Apr 22 | 4 (so far) | 0 | 4 |
| **Total** | **36** | **~5 (14%)** | **~31 (86%)** |

**Outlier assessment:** Apr 17 is NOT an outlier — Apr 20 also had 13 restarts. Both are heavy dev days. The 13-per-session cadence is the baseline for active development, driven entirely by `systemctl restart` deployments.

**Dominant source: operator restarts (86%)** — clean Stopping→Deactivated→Started triples from `systemctl restart` during dev workflow. Will naturally decline as bot development stabilizes.

**Crash RCA — 3 crash events:**

1. Apr 17 07:55 (2 auto-restart cycles) — `EACCES: permission denied, open '/tmp/telegram-bot/.keep'` (`config.ts:230`). Root-owned `/tmp/telegram-bot/` dir from pre-Phase-B setup; `assistant` could not write into it on first post-migration restart. Manually fixed by root between the 2nd crash (07:55:11) and 3rd start (07:55:20). Current state: dir is `assistant`-owned (Born 2026-04-17 07:55:21), has survived since. Risk: recurs if VM reboots and the dir is recreated as root (unlikely but not guarded). **Fix: add `mkdir -p /tmp/telegram-bot` to `ExecStartPre` in systemd unit.**

2. Apr 18 14:35 — `error: Operation aborted` in `@anthropic-ai/claude-agent-sdk/sdk.mjs:13256` — `write()` called on an already-aborted stream during `/stop` command cleanup race. Stack: `handleStop → stop → onAbortListener2 → handleControlRequest → write → AbortError`. Unhandled, crashes process. **Bug: AbortError in write path not caught.**

3. Apr 20 17:05 — same AbortError from same code path, triggered by 900s permission timeout auto-deny with interrupt. Log: "Permission request timed out after 900s — auto-denying with interrupt." This is a real production risk: any permission request left unanswered for 15 minutes will crash the bot. **Same fix as #2.**

**Audit log EACCES (Track B):** `EACCES: permission denied, open '/tmp/claude-telegram-audit.log'` — 4 failures on Apr 17 only, caused by root-owned audit log file (pre-Phase-B). Self-resolved: current file is `assistant:assistant` (224KB, active). Zero failures since Apr 17. No action needed.

**Recommendation:** Accept restart churn as dev noise. Fix the two AbortError crashes — the permission-timeout variant is a latent production stability risk (any unanswered permission prompt for 15min crashes the bot).
