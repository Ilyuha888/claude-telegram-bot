# Spec idempotency model (tg-in key) is pre-pivot spec-only — no implementation in live bot

## Rationale

prob-20260420-005 assumed tg-in:<bot_id>:<update_id> idempotency key is enforced by the live bot. Source audit (2026-04-20) shows it exists only in docs/DATA_MODEL.md:410 and docs/API_SURFACES.md:77-83 — artifacts from the abandoned Postgres-backed custom runtime (ADR-003). The live system is linuz90/claude-telegram-bot + Claude Code SDK. No idempotency_records table, no dedup logic, no seenUpdates set anywhere in src/. The "idempotency" question reduces to: what does grammY runner do with duplicate updates (answer: Telegram guarantees at-most-once delivery per webhook, grammY runner adds no extra dedup). For a single-user bot with no webhook retries, no dedup layer is needed.

## Evidence

grep -r across /home/assistant/claude-telegram-bot/src/: zero matches for update_id, seenUpdates, idempotency, edited_message. docs/DATA_MODEL.md:375-411 and docs/API_SURFACES.md:60-90 confirmed as pre-pivot spec. ADR-003 confirms Postgres runtime was the intended host for idempotency_records but was never built after pivot.
