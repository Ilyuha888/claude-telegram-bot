# reply_to_message and message.quote fields ignored — trust boundary is text-only, no expansion risk

## Rationale

src/handlers/text.ts reads only ctx.message.text and passes the bare string to session.sendMessageStreaming(). Neither ctx.message.reply_to_message nor ctx.message.quote (Bot API 7.0+ quotation field) are accessed anywhere in the handler or session layer. Rapid succession is serialized per chat_id via grammY sequentialize middleware — messages queue in arrival order, no race conditions, no coalescing. Photo/media handlers pass caption + image bytes; caption is trusted user input (no reply_to or quote fields parsed there either). Conclusions: (1) reply-to a bot message does NOT re-feed prior bot output to Claude — only new text is sent; (2) quoted fragments from third-party messages do NOT flow to Claude; (3) rapid messages serialize cleanly. All three remaining scenarios from prob-20260420-005 (rapid succession, reply-to, quotation) are safe by construction.

## Evidence

text.ts line: `let message = ctx.message?.text`. No destructuring of reply_to_message or quote. session.ts sendMessageStreaming receives string or UserContentBlock[]; no reply-to metadata passed. index.ts sequentialize: serializes on ctx.chat?.id.toString() for non-command, non-!, non-callback messages. grep -r reply_to_message /src: empty output. grep -r message.quote /src: empty output.
