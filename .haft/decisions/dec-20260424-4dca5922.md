# Fire-and-forget MCP drop-file with server-side enforcement

## 1. Problem Frame

**Signal:** The Telegram bridge (linuz90/claude-telegram-bot, dec-20260416-003) relays Claude Code's final output as MarkdownV2 text and intercepts AskUserQuestion via canUseTool, but the reply surface is text-only. When the user asks for a vault image, a rendered diagram, an exported document, or a haft artifact as a file, the agent can only: (a) paste a filesystem path the user cannot open from Telegram, (b) read and inline-quote the content as text (loses images entirely, mangles structured docs), or (c) abandon the request. Concretely: user asks "send me the architecture diagram" → agent replies with a vault path string. User asks "give me that PDF" → no pathway exists. The asymmetry is notable: inbound images and PDFs already land as SDK content blocks (dec-20260420-001), so the user can send media to the agent but not receive it. send_file was cited as a V_linuz90 selection reason in dec-20260416-003 line 49, but line 82 confirms it reflects TS Agent SDK capability, not an active wired-up feature.

**Constraints:**
- Subscription billing untouched — do not route through ANTHROPIC_API_KEY (dec-20260416-003 admissibility; dec-20260414-001)
- No shell escape — no subprocess / raw shell added to the runtime LLM path (AGENTS.md §Safety Rules; ARCHITECTURE.md invariant)
- Path validation enforced — every file read for outbound must pass isPathAllowed() (ALLOWED_PATHS) before the bot sends it; isPathAllowed() invariant from dec-20260420-003 must not be bypassed or weakened
- Denylist model preserved — no relaxation of the Bash denylist in dec-20260420-003 (sudo|git push|commit|reset|rebase|clean)
- Writable surface unchanged — do not widen ALLOWED_PATHS, do not grant new write capabilities (AGENTS.md §Safety Rules)
- Egress posture — api.telegram.org is already on the implicit outbound allowlist (dec-20260416-001 admissibility); fetching external URLs to attach is out of scope; WebFetch/WebSearch remain disallowed
- Two-way Telegram operational at all times — existing text + AskUserQuestion paths must not regress (dec-20260416-003 invariant)
- Credential protection — settings.local.json (chmod 600), .env, and paths covered by guard-sensitive-reads.py must not become reachable via the outbound file path even when they reside in an otherwise-allowed directory
- Single-user — no multi-recipient fan-out (dec-20260320-001)
- Content scope (user-confirmed 2026-04-22): images (sendPhoto) and documents (sendDocument) only; audio/video deferred; external URL fetch-to-attach deferred
- Sources (user-confirmed 2026-04-22): vault files/ sibling dirs, arbitrary paths passing isPathAllowed(), agent-generated mid-session artifacts written to an allowed path
- Initiator (user-confirmed 2026-04-22): explicit user request AND agent-proactive — agent may attach when it judges attachment is the right reply shape

**Acceptance:** All of the following hold, verified with evidence: (1) User can ask (in Telegram) for a named vault image or document and receive it inline in chat — image preview / downloadable file with original filename and extension preserved. (2) Agent can attach an image or document proactively when it is the correct reply shape, and the attached file matches the message content — manually verified on ≥3 proactive sends. (3) Every successful outbound file send logs: tool name, resolved absolute path, file size, isPathAllowed() verdict, Telegram chat id. (4) Negative test: an attempt to send a path outside ALLOWED_PATHS (or a sensitive path like .claude/) is refused before the Telegram API call, and the refusal surfaces back to the agent as a tool error — not a silent drop. (5) Existing text-reply and AskUserQuestion paths continue to work — no regression in a 1-week live window. (6) File size limits handled gracefully: >10MB for sendPhoto and >50MB for sendDocument produce a clear error, not a crash.

## 2. Decision

**Selected:** Fire-and-forget MCP drop-file with server-side enforcement

**Selection policy:** Minimize blast radius: prefer the option that isolates security enforcement and limits failure scope to file delivery only, without coupling to bot core logic.

**Why selected:** V1 keeps security enforcement in the MCP subprocess (isolated from bot logic), lets the agent continue generating immediately, and audit-logs every delivery attempt. The drop-file + result-event poll pattern cleanly separates tool execution from Telegram delivery without coupling to canUseTool timing.


**Invariants:**
- Forbidden segments (.git, .claude, .ssh, .aws, .gnupg) are always refused
- Forbidden filenames (.env, id_rsa, credentials, .netrc) are always refused
- Paths outside ALLOWED_PATHS and /tmp are always refused
- Every delivery attempt — success or failure — is audit-logged

**Admissibility:**
- NOT: No delivery for paths outside ALLOWED_PATHS
- NOT: No silent failures — all errors must surface to the agent as isError
- NOT: No delivery without an audit log entry

## 3. Rationale

**Counterargument:** The drop-file poll fires once at result-event; if delivery fails (Telegram API error), the agent only learns on the next turn and must retry explicitly.

**Selected variant weakest link:** Single poll at result event — no automatic retry on Telegram API failure.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|

| V2 — native canUseTool intercept | Rejected | canUseTool fires before the assistant event in SDK V2; ctx.reply is not reliably available at that point, making synchronous delivery fragile. |

**Evidence requirements:**
- Positive: at least one photo and one document successfully delivered in Telegram
- Negative: .ssh/id_rsa and .env rejected with isError, no Telegram API call
- Size cap: >10MB image rejected as photo, passes as document with send_as_document=true; >50MB rejected unconditionally
- Regression: plain text replies and AskUserQuestion keyboard unaffected

## 4. Consequences

**Rollback plan:**
Triggers:
- send_file delivery causes bot crash or hang
- allowlist bypass discovered in production
Steps:
1. Remove 'send-file' entry from mcp-config.ts to stop registering the tool
2. Restart bot — agent loses access to send_file immediately
3. Existing drop-files cleaned on next poll cycle
Blast radius: Outbound file delivery only — text replies and inbound attachment hint unaffected

**Refresh triggers:**
- SDK major version bump changes assistant-event / tool-execution ordering
- Telegram API changes file size limits
- MCP server subprocess stability issues observed over 2-week window

**Affected files:** send_file_mcp/server.ts, src/config.ts, src/handlers/streaming.ts, src/session.ts, mcp-config.ts
