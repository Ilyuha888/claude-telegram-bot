# Layered rollout — config + user migration, skip outbound egress

## 1. Problem Frame

**Signal:** Claude Code runs as root on the Hetzner VM with no network egress controls, no .claudeignore file, and no --disallowedTools restrictions. Research doc identifies this as a meaningful risk for a self-hosted personal assistant: root means any tool execution has full filesystem and network privileges; no outbound restrictions means tools could exfiltrate data to arbitrary endpoints; no .claudeignore means Claude can read ~/.ssh/id_rsa, .env files, credentials; no --disallowedTools means all built-in tools are available including destructive ones. Threat model: self-hosted single-user VM, not multi-tenant — but prompt injection via vault content or dependency READMEs is a realistic attack surface.

**Constraints:**
- Must not break existing assistant functionality
- Single-user scope — no need for multi-user access controls
- Changes must survive systemd restart
- Git push credentials (GitHub PAT) must remain functional

**Acceptance:** Claude Code runs as a dedicated non-root user; .claudeignore exists and excludes ~/.ssh, .env, credential files; --disallowedTools is configured; UFW denies all inbound except SSH. All existing workflows (vault read/write, git push, Telegram messaging) still function.

## 2. Decision

**Selected:** Layered rollout — config + user migration, skip outbound egress

**Selection policy:** Maximize security improvement per unit of operational overhead, constrained by single-user threat model. Prefer variants with low ongoing maintenance burden.

**Why selected:** Best security-to-overhead ratio. Eliminates root (biggest single risk factor) and deploys all config-layer protections without adding ongoing maintenance. Executed as V3→V1: Phase A deploys .claudeignore + --disallowedTools + inbound UFW as a quick win; Phase B completes user migration in a planned SSH session.


**Invariants:**
- Claude Code must remain operational via Telegram after each phase
- Vault read/write and git push must work at all times
- SSH root access must remain available as emergency fallback
- No secrets stored in any Git repository

**Pre-conditions:**
- [ ] SSH access to VM confirmed
- [ ] Current settings.local.json backed up
- [ ] Current start-assistant.sh backed up

**Post-conditions:**
- [ ] Phase A: .claudeignore exists and blocks ~/.ssh/ reads; UFW denies inbound except SSH; --disallowedTools configured
- [ ] Phase B: Claude Code runs as non-root user; systemd unit updated; all workflows functional

**Admissibility:**
- NOT: Do NOT remove SSH root access — it is the emergency recovery path
- NOT: Do NOT block outbound to api.anthropic.com, api.telegram.org, or github.com
- NOT: Do NOT deploy Phase B without first verifying Phase A is stable

## 3. Rationale

**Counterargument:** The user migration is the riskiest single step — if claude login auth tokens aren't portable, the assistant goes down until manually fixed via SSH. Sticking with V3 (config-only) would be safer and still blocks the most likely attack vector (credential reads via .claudeignore).

**Selected variant weakest link:** Auth token portability during user migration — claude login credentials and GitHub PAT are rooted under /root. Migration requires re-authentication and path updates across ~5 config files.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| Layered rollout — config + user migration, skip outbound egress | **Selected** | Best security-to-overhead ratio. Eliminates root (biggest... |
| Full hardening — all four components including outbound proxy | Rejected | Proxy adds ongoing maintenance: allowlist drift causes silent failures, latency on every API call, another service to monitor. Marginal security gain over V1 (outbound egress filtering) is disproportionate to single-user threat model. Available as future upgrade from V1 if threat model changes. |
| Config-only hardening — .claudeignore + disallowedTools + inbound UFW | Rejected | Good as Phase A stepping stone but insufficient as final state. Leaves root privileges intact — .claudeignore and --disallowedTools are advisory guardrails enforced by Claude Code, not OS-level isolation. A prompt injection bypassing the Bash tool could still access everything. |

**Evidence requirements:**
- Verify .claudeignore blocks sensitive file reads via Claude Code
- Verify UFW status shows deny incoming except SSH
- Verify full Telegram → vault → git push lifecycle after each phase
- Verify systemd service starts correctly as non-root user (Phase B)

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| .claudeignore blocks credential reads | Claude Code Read tool on ~/.ssh/id_rsa after Phase A | Read is denied or returns empty |
| Non-root user does not break Telegram flow | Send and receive Telegram message after Phase B | Message delivered within 30 seconds |
| UFW blocks unauthorized inbound | nmap scan from external host after Phase A | Only port 22 open |

## 4. Consequences

**Rollback plan:**
Triggers:
- Telegram messaging broken after phase deploy
- Vault git push fails
- Claude Code fails to start after systemd change
Steps:
1. Phase A rollback: delete .claudeignore, revert settings.local.json, ufw disable (2 min)
2. Phase B rollback: revert systemd unit to root user, restart service (30 min)
Blast radius: VM system configuration: systemd unit, start-assistant.sh, .claude config dir, UFW rules

**Refresh triggers:**
- Threat model changes (e.g., multi-user access, untrusted input sources)
- Claude Code permission model changes in a future release
- New MCP server added that requires root-level paths

**Affected files:** /root/.claudeignore, /root/.claude/settings.local.json, /root/start-assistant.sh, /etc/systemd/system/claude-assistant.service
