# Claude Code on VM as the personal assistant runtime

## 1. Problem Frame

**Signal:** Three independent implementations (Karpathy's LLM wiki, llm_under_hood's Personal OS v2, muscat) converge on the same vault infrastructure — markdown + Git + Obsidian + typed tool contracts. But the two that ship both have explicit operational modes (ingest → compile → query → lint/maintenance) that turn infrastructure into compounding knowledge. Muscat has detailed CRUD tool contracts and a manifest-staging flow but no design for these higher-level workflows. The vault can store notes; there is no design for how raw inputs become cross-referenced, structured knowledge, or how knowledge is queried back with synthesis, or how the vault stays internally consistent over time.

**Constraints:**
- Phase 0 runtime scaffold has higher execution priority — this problem informs future phases and must not block Phase 0
- Vault architecture decisions are settled: two-root vault (User_Obsidian_Vault + Agent_Obsidian_Vault), manifest-only staging in Postgres, review-before-commit for user vault writes, in-memory Dulwich commit on approval (dec-20260320-005)
- Single-user scope is an explicit constraint (dec-20260320-001)
- Selective SGR for non-tool structured outputs is decided (dec-20260324-001) — lifecycle outputs (compiled pages, query syntheses) must conform
- No new vault write primitives — modes must orchestrate existing vault.* and git.* tool contracts

**Acceptance:** A documented knowledge lifecycle spec covering: (1) ingest mode — input sources, compilation steps, output note types, cross-reference update strategy; (2) query mode — search + synthesis + optional file-back flow; (3) lint/maintenance mode — consistency checks, orphan detection, cross-reference repair. Spec must map each mode to existing tool contracts (vault.*, git.*) and identify open questions (e.g. search backend) without resolving them. No code required.

## 2. Decision

**Selected:** Claude Code on VM as the personal assistant runtime

Claude Code already implements everything muscat was going to build — skills, MCP servers, persistent memory, file tools, Git operations, session management, and a Telegram channel plugin. After 9 ADRs, 7+ quint decisions, and zero source code shipped, building a custom orchestrator adds no value when Claude Code does it better. The pivot eliminates months of planned implementation work and delivers a working assistant in ~2 hours of setup.

**Invariants:**
- Knowledge vault remains a separate Git repository — Claude Code writes to it but does not own it
- Single-user scope constraint (dec-20260320-001) still applies — Claude Code session is single-user by design
- No arbitrary shell execution granted to runtime LLM path — Claude Code permission model enforces this via CLAUDE.md and settings.json
- All vault mutations go through Git — Claude Code commits changes, no direct file writes that bypass history

**Pre-conditions:**
- [ ] claude.ai Pro or Max subscription active (required for Channels + Remote Control features)
- [ ] VM provisioned: Oracle Cloud free tier A1.Flex (preferred) or Hetzner CPX22 (~$9.50/mo)
- [ ] Obsidian knowledge base accessible as a Git repository (clone URL available)

**Post-conditions:**
- [ ] Claude Code running on VM in tmux, persisting across terminal disconnects
- [ ] Telegram channel plugin configured and two-way messaging confirmed
- [ ] Remote Control accessible via claude.ai/code from browser and mobile
- [ ] Obsidian vault cloned on VM and navigable by Claude Code file tools
- [ ] CLAUDE.md written at vault root with navigation conventions
- [ ] muscat repo archived on GitHub with this decision recorded
- [ ] All muscat-specific quint decisions marked as superseded

**Admissibility:**
- NOT: Do NOT build a custom Telegram bot from scratch — use the channel plugin first; only fall back to Agent SDK bridge if the plugin proves unworkable
- NOT: Do NOT re-introduce a custom Python orchestrator — if Claude Code lacks a capability, extend it via MCP server or skill, not a parallel runtime
- NOT: Do NOT store secrets (bot tokens, API keys) in the knowledge vault repo
- NOT: Do NOT grant Claude Code unrestricted shell access on the VM — maintain explicit tool permissions

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| Claude Code on VM as the personal assistant runtime | **Selected** | Claude Code already implements everything muscat was goin... |
| Continue muscat development | Rejected | Custom Python orchestrator is a worse, unmaintained version of Claude Code. Every component planned — LLM calls, structured output, tool policy, session schema, vault Git flow — has a better equivalent in Claude Code. The opportunity cost is indefinite: muscat would remain in design phase while Claude Code already ships the capability. |
| Hybrid (muscat + Claude Code) | Rejected | Unnecessary complexity. Two systems to keep synchronized with no clear boundary. Claude Code's Telegram channel plugin satisfies the Telegram UX requirement directly without a custom bridge. |

**Weakest link:** Telegram channel plugin stability — it is in research preview as of 2026-04. If it breaks or is discontinued, Telegram-first interaction fails. Remote Control remains as robust fallback but changes the UX.

**Evidence requirements:**
- Telegram two-way messaging confirmed working on the VM
- Remote Control accessible from browser and mobile
- Claude successfully reads vault notes and follows wikilinks
- Claude commits to vault repo via Git
- Session survives tmux detach + reattach cycle

## 4. Consequences

**Rollback plan:**
Triggers:
- Telegram channel plugin is discontinued without replacement
- Anthropic pricing for Claude Code makes always-on VM use economically unviable
- A hard requirement emerges that Claude Code cannot satisfy (e.g., multi-user, real-time data feeds, custom approval gate)
Steps:
1. Claude Code sessions store conversation history in ~/.claude/projects/ — export before rollback
2. Knowledge vault remains intact as a standalone Git repo — no migration needed
3. If rolling back to custom orchestrator: muscat ADRs and quint decisions are archived, not deleted — reopen via quint_refresh(action='reopen')
Blast radius: High — abandons Claude Code configuration investment. Knowledge vault data is safe (separate repo). No code to migrate.

**Refresh triggers:**
- Telegram channel plugin exits research preview with breaking API changes
- Claude Code remote sessions pricing changes significantly (currently included in Pro/Max subscription)
- New requirement emerges that Claude Code cannot satisfy without a custom runtime
- Claude Code Channels feature is deprecated or moved behind a higher tier

**Affected files:** ARCHITECTURE.md, README.md, .quint/
