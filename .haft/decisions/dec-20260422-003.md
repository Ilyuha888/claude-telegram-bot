# E-revised — Cloud Code Remote native session (tmux + --remote-control + --continue)

## 1. Problem Frame

**Signal:** The linuz90/claude-telegram-bot (dec-20260416-003) added /new and /resume for multi-session conversation within the knowledge-base repo, but cannot spawn Claude Code sessions bound to arbitrary repos or git worktrees on the VM. This is the unresolved multi-repo axis of prob-20260416-001, which was superseded by dec-20260416-003 only on the single-repo multi-session axis. The acceptance criterion from prob-20260416-001 — 'spawn a new session in a chosen repo, switch which session receives subsequent messages' — remains unmet. Mobile developers cannot initiate a focused coding session in e.g. data-style/junk_mart from Telegram and hand off to Cloud Code Remote on desktop. Detailed spec in Agent_Obsidian_Vault/tasks/2026-04-20-telegram-multi-repo-remote-session.md.

**Constraints:**
- VM SSH access required at dispatch time (repos under ~/repos/)
- Corporate subscription tier must be active before Opus-plan-mode sessions can be used
- Worktree must never be deleted by /close — only detached
- Failed dispatch must clean up any partially-created worktrees
- No LLM reasoning in Mode-2 command layer (menu-only)

**Acceptance:** /work <repo> [path] [worktree] [branch] spawns a remote Claude Code session on VM and returns a session id; Cloud Code Remote can attach; /sessions lists active sessions; /attach <id> routes subsequent messages; /close <id> detaches without deleting git worktree; 7-day idle triggers auto-close while preserving worktree state; zero dangling git worktrees after any edge-case disconnection.

## 2. Decision

**Selected:** E-revised — Cloud Code Remote native session (tmux + --remote-control + --continue)

**Selection policy:** Prefer lower implementation_effort among variants satisfying the session-persistence constraint (session survives Telegram disconnect); break ties on handoff_latency (lower better). Fall back to variant A if VM smoke test fails to confirm remote-control attach semantics.

**Why selected:** E-revised eliminates the custom SQLite session store required by variant A. Sessions are natively visible at claude.ai/code — zero custom attach UX needed. `claude --remote-control <name> --continue` gives conversation persistence across restarts without a separate supervisor. Handoff is native (claude.ai/code session list) rather than SSH terminal passthrough. Implementation effort is lower than A (no SQLite, no reaper cron — RC server manages session lifecycle).


**Invariants:**
- Git worktrees MUST NOT be deleted by /close — only detach or leave attached, never git worktree remove
- Failed /work dispatch MUST clean up any partially-created worktree before returning error to user
- Mode-2 handlers MUST NEVER call sendMessageStreaming() or ClaudeSession.saveSession()
- sessions.json MUST be written atomically to prevent corruption on concurrent /work commands
- 7-day idle auto-close MUST preserve worktree on disk

**Pre-conditions:**
- [ ] VM smoke test passes: `claude --remote-control 'test-probe' --continue` starts in tmux session, visible at claude.ai/code within 10s
- [ ] Bot user `assistant` has write access to /home/assistant/bot-data/
- [ ] All target repos exist under ~/repos/ before any /work command is issued
- [ ] git worktree add succeeds in at least one test repo on VM

**Post-conditions:**
- [ ] /work data-style creates tmux session + RC session visible at claude.ai/code
- [ ] /work with worktree argument creates worktree at correct path before spawning session
- [ ] /sessions lists active sessions from sessions.json with correct repo/path/worktree metadata
- [ ] /attach <id> returns claude.ai/code session name for desktop connection
- [ ] /close <id> kills tmux session, marks closed in sessions.json, worktree preserved on disk
- [ ] Bot restart with sessions.json present restarts dead sessions via --continue --remote-control
- [ ] 7-day idle reaper closes tmux session without touching worktree

**Admissibility:**
- NOT: Mode-2 handlers MUST NOT call sendMessageStreaming() under any code path
- NOT: sessions.json MUST NOT store conversation content — only metadata (slug, tmux_name, worktree_path, session_name, created_at, closed)
- NOT: git worktree remove MUST NOT be called by any Mode-2 handler

## 3. Rationale

**Counterargument:** Variant A (tmux + SQLite) is proven infrastructure; E-revised depends on claude remote-control session API behavior that has not been smoke-tested on the live VM. If RC sessions don't survive tmux detach or bot restart, fallback to A is required.

**Selected variant weakest link:** claude --remote-control session persistence under tmux detach — unverified that RC server keeps the session alive when the interactive process is detached rather than killed.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| E-revised — Cloud Code Remote native session (tmux + --remote-control + --continue) | **Selected** | E-revised eliminates the custom SQLite session store requ... |
| A — tmux + git-worktree + SQLite session store | Rejected | E-revised subsumes A's tmux+worktree core while eliminating the SQLite store and custom attach instructions. A remains the explicit fallback if E-revised smoke test fails. |
| C — Claude Code native resume-only, no supervisor | Rejected | Constraint-eliminated: violates session-persistence requirement. No running process survives Telegram disconnect. A bookmark system is not a session system. |

**Evidence requirements:**
- VM smoke test result: RC session visible at claude.ai/code
- End-to-end /work → /attach → desktop handoff test log
- journalctl showing zero sendMessageStreaming() for Mode-2 commands
- git worktree list output after /close confirming worktree preservation
- Idle reaper test with 1h threshold confirming clean close behavior

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| RC session visible at claude.ai/code within 10s of /work command | claude.ai/code session list shows new session with correct name after /work send | <10s from /work send to session visible |
| Conversation context restored after bot restart via --continue | After systemctl restart claude-telegram-bot, sessions.json triggers --continue on each saved session; journalctl shows RESUMING (not STARTING) for each session | 100% of sessions in sessions.json resumed on restart |
| Zero orphaned worktrees after /close | git worktree list in repo after /close shows worktree still present; tmux ls shows session gone | Worktree exists, tmux session does not exist |
| Idle reaper closes sessions without worktree deletion | After simulated 7-day threshold (1h test threshold), tmux session gone, worktree intact, sessions.json entry closed=true | Worktree on disk, tmux session removed, entry marked closed |

## 4. Consequences

**Rollback plan:**
Triggers:
- VM smoke test fails — RC sessions not visible at claude.ai/code
- Bot restart fails to resume sessions via --continue
- sessions.json corruption causes /work to error persistently
Steps:
1. Fall back to variant A: replace sessions.json store with SQLite, replace RC spawn with plain tmux+claude (no --remote-control), replace /attach with SSH terminal instructions
2. Reset sessions.json to empty array: echo '[]' > /home/assistant/bot-data/sessions.json
3. Run git worktree list in each repo to audit orphans; manually prune with git worktree prune
Blast radius: linuz90 bot Mode-2 handler and sessions.json only; Mode-1 session (tryAutoResume) unaffected; vault and worktrees untouched

**Refresh triggers:**
- claude remote-control API changes in Claude Code update
- VM smoke test fails at any point — trigger immediate fallback to variant A
- sessions.json corruption incident — evaluate SQLite migration (variant A)
- Claude Code removes --remote-control or --continue flag
- Worktree count exceeds 50 (storage pressure review)

**Affected files:** /home/assistant/claude-telegram-bot/src/handlers/mode2.ts, /home/assistant/bot-data/sessions.json, /home/assistant/claude-telegram-bot/src/bot.ts
