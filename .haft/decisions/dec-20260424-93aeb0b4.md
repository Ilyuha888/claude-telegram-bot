# V_rich — Analytical routines with specific proposals

## 1. Problem Frame

## 2. Decision

**Selected:** V_rich — Analytical routines with specific proposals

**Selection policy:** Maximize output_actionability first — routine must reduce decision overhead, not just report status. Weekly curator completeness must be full (all three V-A axes). Token budget is a soft target on subscription billing.

**Why selected:** The routine's purpose is to replace manual curation decisions, not just surface data. V_lean's flat task list and partial weekly coverage still requires the user to prioritize and identify promotion candidates manually — the routine saves reading time but not deciding time. V_rich's 'one focus for today' and promotion proposals with specific vault paths are what make the routine worth the daily context switch cost.


**Invariants:**
- Daily prompt MUST use focused reads (grep/status-field extraction) — MUST NOT read tasks.md or project files in full; 44KB tasks.md in full context is a budget violation
- Each routine prompt MUST label the focus suggestion as advisory: 'suggested focus (you decide)'
- Weekly report MUST name specific vault paths for each promotion candidate — generic 'this draft looks good' is not acceptable
- All routines fire via the ephemeral scheduler session (dec-20260424-aa4e9e2d) — MUST NOT write to SESSION_FILE or pollute the user's working context
- Prompts MUST be stored in source control (src/scheduler-prompts.ts), not embedded in schedules.json — prompts are code, not config

**Pre-conditions:**
- [ ] dec-20260424-aa4e9e2d scheduler substrate implemented and verified
- [ ] Vault CLAUDE.md documents task status conventions (In Progress / Done / Frozen) and folder layout so prompt engineering can rely on stable structure
- [ ] Each of the 4 prompts tested in an interactive Claude session before scheduler deployment — verify output format and token budget empirically

**Post-conditions:**
- [ ] Daily 09:00 soft notification title includes task count + inbox count; [Show] reveals ranked task list + focus suggestion
- [ ] Weekly Sun 20:00 [Show] reveals: (1) top 3 draft promotion candidates with Agent_Obsidian_Vault/drafts/ paths + one-line rationale, (2) stale notes list from 10-notes/ with days-since-edit, (3) at least one MOC gap observation from 60-mocs/
- [ ] Monthly [Show] reveals: active project count, list of projects with no activity in >30d flagged for close/archive consideration
- [ ] Daily token budget stays ≤12K per fire (verified by prompt testing pre-deployment)
- [ ] Focus suggestion is labeled advisory in every daily output

**Admissibility:**
- NOT: NOT: prompts MUST NOT read tasks.md or any project file in full — grep or Read with offset only
- NOT: NOT: weekly report MUST NOT include generic 'review your drafts' — must name specific files
- NOT: NOT: routines MUST NOT write any vault file — read-only; vault writes go through commit-confirm in interactive sessions only

## 3. Rationale

**Counterargument:** tasks.md is 44KB — reading it in full every day as part of V_rich's 'reads each active project file' would push the daily fire well above the ~12K token estimate, toward 20-40K. V_lean avoids this by extracting only In Progress rows. V_rich's token budget claim is only valid if the daily prompt uses focused reads (grep for status fields, not full file body reads). If prompt engineering fails to scope the read, V_rich degrades to an expensive V_lean with worse false-positive risk.

**Selected variant weakest link:** Prompt engineering quality — V_rich's value is entirely dependent on the prompts correctly scoping their reads (no full tasks.md), correctly identifying In Progress vs Done, and producing a focus suggestion that reflects actual project state. If the prompt underspecifies the read scope, the token budget blows out and false-positive risk rises toward V_lean levels with higher cost.

| Variant | Verdict | Reason |

The user would need to manually prioritize and identify promotions, defeating the routine's purpose. Acceptable only if decision overhead is not a concern; it is. |

**Evidence requirements:**
- Pre-deployment prompt test: run each of the 4 prompts interactively; record token usage and output quality
- First 4 weekly curator reports reviewed and rated (useful / not useful / mostly noise)
- 2-week daily focus suggestion acceptance log

## 4. Consequences

**Rollback plan:**
Triggers:
- Daily token budget consistently exceeds 12K despite focused reads
- Weekly reports consistently surface wrong/stale promotion candidates
- Prompt quality degrades after vault restructure
Steps:
1. Switch daily routine to V_lean prompt (grep In Progress rows only, no project file reads) — edit src/scheduler-prompts.ts, no scheduler change needed
2. Drop MOC gap analysis from weekly prompt if that axis is consistently wrong — partial coverage is still acceptable
3. Full rollback: disable all routines in schedules.json (enabled: false)
Blast radius: src/scheduler-prompts.ts and schedules.json only. Vault content, scheduler substrate, and notification system are untouched.

**Refresh triggers:**
- Daily token budget exceeds 12K consistently — revert to V_lean for the daily routine
- Weekly MOC gap analysis is consistently wrong or unhelpful — drop that axis, reduce to partial
- tasks.md structure changes (e.g. migrated to separate per-project files) — prompts need update
- Vault exceeds 500 notes — re-evaluate read scope and prompt design
