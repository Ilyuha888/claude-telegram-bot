# Two targeted fixes in session.ts

## 1. Problem Frame

**Signal:** Two distinct triggers both crash the bot via the same unhandled AbortError in @anthropic-ai/claude-agent-sdk handleControlRequest → write(): (1) /stop command with a response in-flight (Apr 18 14:35), (2) 900s permission timeout auto-deny sending an interrupt (Apr 20 17:05). In both cases the abort controller fires, the stream is marked aborted, but a subsequent write() call throws AbortError which propagates to top-level and kills the process. Any unanswered permission prompt left for 15+ minutes will crash the bot.

**Constraints:**
- Do not modify sdk.mjs (third-party vendored code)
- Fix must not suppress legitimate abort signals — abort should still stop the Claude session cleanly
- Bot must restart cleanly via systemd Restart=on-failure after the fix (or not crash at all)
- No new production dependencies

**Acceptance:** Zero AbortError-caused exits in journal over 7 days post-deploy. /stop and permission timeout still terminate the session cleanly without crashing the process.

## 2. Decision

**Selected:** Two targeted fixes in session.ts

**Selection policy:** Minimum viable fix covering all known crash triggers with lowest masking risk and smallest diff. Global guards and partial-path fixes both rejected.

**Why selected:** Non-dominated variant: covers both known crash triggers (stop() synchronous throw + catch condition too restrictive) with minimal diff and low masking risk. V2 anti-pattern, V3 partial.


**Invariants:**
- AbortError and cancel errors from the SDK must never crash the process
- Session terminates cleanly after abort — isQueryRunning reset, abortController nulled (already in finally block)
- Real non-abort errors must still propagate and be logged

**Pre-conditions:**
- [ ] Bot is running on VM (assistant user, systemd service)
- [ ] session.ts and config.ts are readable/writable on VM

**Post-conditions:**
- [ ] Zero AbortError exits in journal for 7 days post-deploy
- [ ] /stop terminates the session cleanly without crashing the process
- [ ] Permission timeout auto-deny does not crash the process
- [ ] config.ts startup TEMP_DIR creation is resilient to root-owned dir

**Admissibility:**
- NOT: Must not suppress non-abort errors (errors not containing 'abort' or 'cancel')
- NOT: Must not change the SDK or node_modules
- NOT: Must not disable the abort mechanism itself

## 3. Rationale

**Counterargument:** Removing the (queryCompleted || stopRequested || askUserTriggered) gate means any future SDK error whose message happens to include 'abort' or 'cancel' will be silently swallowed. The original gate was there precisely to prevent this. A more conservative approach would be V3 (set stopRequested on interrupt:true) which preserves the gate — but it leaves non-canUseTool abort paths unguarded.

**Selected variant weakest link:** String-match on 'abort'/'cancel' to classify errors as safe — brittle if SDK changes error messages

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| Two targeted fixes in session.ts | **Selected** | Non-dominated variant: covers both known crash triggers (... |
| V2 — global process handler | Rejected | Anti-pattern; high masking risk for future real crashes |
| V3 — set stopRequested on interrupt:true | Rejected | Partial fix — only covers canUseTool abort path, not other SDK abort sources |

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| No AbortError-caused exits in journal | grep 'Operation aborted' /tmp/ctb-7d.log on next 7-day pull | 0 occurrences after deploy date |
| /stop command no longer crashes bot | systemctl is-active claude-telegram-bot.service after /stop issued during active query | active (not crashed) |

## 4. Consequences

**Rollback plan:**
Triggers:
- Bot fails to start after deploy
- Non-abort errors being silently swallowed (check journal for missing error logs)
Steps:
1. git revert the commit on VM
2. systemctl restart claude-telegram-bot
Blast radius: Bot source only — no vault, no Claude Code config, no systemd unit changes
