# V3 — Selective SGR (hybrid policy)

## 1. Problem Frame

**Signal:** The orchestrator architecture specifies typed tool contracts (Pydantic) for tool inputs/outputs, but has no policy for how the LLM produces structured reasoning or non-tool structured outputs — intent classification, session compaction, and review generation are all unaddressed. Schema-Guided Reasoning (SGR) is a candidate philosophy worth evaluating before the orchestrator scaffold is written.

**Constraints:**
- Must work with openai Python SDK targeting Gemini 2.5 Flash (dec-20260320-003)
- Must not add production dependencies without justification
- Must preserve provider-swappability via env-var swap
- Pydantic schemas are already the typed boundary standard — any new approach must compose with them

**Acceptance:** A clear policy on when/whether to use reasoning schemas vs plain tool calls vs free-form text, documented in ARCHITECTURE.md, with refresh trigger set for post-first-implementation review.

## 2. Decision

**Selected:** V3 — Selective SGR (hybrid policy)

V3 is the only non-dominated variant. V1 has no reasoning auditability. V2 had unverified provider compatibility and disproportionate schema overhead. The spike (scripts/spike_sgr_response_format.py) confirmed that Gemini 2.5 Flash fully supports json_schema response_format via the openai-compat endpoint — 21/21 checks passed, including SGR reasoning fields populated with coherent prose at ~1.3s latency. Provider compatibility concern is resolved. V3's weakest link (policy clarity) is addressed by the invariants below.

**Invariants:**
- Tool-call mechanism is the only output path for agent actions — never replaced by response_format
- Non-tool structured outputs (intent resolution, session compaction, review generation) use response_format json_schema with a Pydantic-backed schema
- Every SGR schema must include a 'reasoning' field as the first field so the model commits its logic before the decision fields
- SGR is added to a call type only when auditability or output consistency is a documented need — not by default for every call
- Pydantic models are the single source of truth for both the json_schema payload and the production validation step — no hand-rolled schema dicts
- Provider swap (OPENAI_BASE_URL + OPENAI_API_KEY) must not require schema changes

**Pre-conditions:**
- [ ] openai Python SDK is the only LLM dependency (dec-20260320-003)
- [ ] Pydantic is already the typed boundary standard across all tool contracts
- [ ] Gemini 2.5 Flash json_schema support verified — spike 21/21 PASS (scripts/spike_sgr_response_format.py)
- [ ] Correct model ID for Gemini compat endpoint: models/gemini-2.5-flash

**Post-conditions:**
- [ ] ARCHITECTURE.md documents the SGR policy under Agent Orchestrator
- [ ] First SGR call type (intent resolution) uses a Pydantic schema with a leading reasoning field
- [ ] Tool-call contracts are unchanged
- [ ] Contract test tier (dec-20260320-004) covers at least one SGR call type with a live Gemini assertion

**Admissibility:**
- NOT: Do not use response_format for tool-call outputs — tool calls go through the SDK tool-call mechanism only
- NOT: Do not add a reasoning field to schemas where the call type has no auditability need (e.g., trivial lookups)
- NOT: Do not hand-roll JSON schema dicts — derive from Pydantic models only
- NOT: Do not pin reasoning field length or content — it is free-form prose, not a structured sub-schema

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| V3 — Selective SGR (hybrid policy) | **Selected** | V3 is the only non-dominated variant. V1 has no reasoning... |
| V1 — Tool-calls-only | Rejected | Reasoning auditability is none — no way to inspect why the LLM classified an intent or made a compaction decision. Fails the primary optimization target. |
| V2 — SGR-everywhere | Rejected | Schema design overhead is disproportionate for simple calls (~30-50 LoC per call type). Also conflates tool-call output (already well-handled by the SDK) with non-tool structured output, adding complexity without benefit. Rejected even after provider compatibility was verified. |

**Weakest link:** Policy clarity — two output paradigms (tool-calls + SGR) require discipline to apply consistently; invariants above are the guardrail

**Evidence requirements:**
- Spike passes 21/21 (already met — scripts/spike_sgr_response_format.py)
- First SGR call type implemented and contract-tested against live Gemini endpoint
- Reasoning field content reviewed for >=2 distinct intent examples to confirm auditability quality

## 4. Consequences

**Refresh triggers:**
- Orchestrator scaffold is running and >=2 SGR call types have been implemented in production
- Provider swap (non-Gemini) is attempted — re-verify json_schema support for new provider
- openai SDK major version bump — re-verify response_format API surface
- Reasoning field content is found to be consistently empty or meaningless in practice

**Affected files:** ARCHITECTURE.md, scripts/spike_sgr_response_format.py
