# SDK content blocks — inline image/PDF bytes in query prompt

## 1. Problem Frame

**Signal:** When users send files via Telegram, the agent fails in two distinct ways. PDFs: Read tool returns "File does not exist" — confirmed with "Analyst_Ilya_Nartov (1).pdf" (2026-04-18). Images: agent hallucinates content, claims image is unclear despite high-res source, or fails to extract visible text. Root cause hypothesis: linuz90/claude-telegram-bot does not download attachments from Telegram Bot API and persist them to file paths the Claude Code agent can read. Agent receives message context but not a valid filesystem path to the file. Affects all attachment types (images, PDFs, documents). File pipeline investigation task: Agent_Obsidian_Vault/tasks/bot-improvement/2026-04-17-improve-image-handling.md. Dependency: dec-20260416-003 (linuz90 bot bridge is the affected codebase).

**Constraints:**
- Fix must be in linuz90/claude-telegram-bot TypeScript/Bun codebase — no custom Python runtime (architecture invariant dec-20260414-001)
- File storage must be accessible to Claude Code agent on VM (under /home/assistant/ or /tmp/)
- Must not require bot restart for each attachment type
- Temp files must be cleaned up after session ends to avoid disk accumulation

**Acceptance:** Phase 1 (blocker): Agent can read PDFs sent via Telegram using Read tool; agent can access images sent via Telegram with valid filesystem paths; files persist long enough for agent to process them. Phase 2 (quality): Agent accurately reads text from high-res screenshots 90%+ of the time; agent never fabricates image content; when genuinely uncertain, asks user to clarify rather than hallucinating.

## 2. Decision

**Selected:** SDK content blocks — inline image/PDF bytes in query prompt

**Selection policy:** Maximize attachment_access_reliability and vision_accuracy for single-user personal assistant. Implementation risk and dependency surface are tiebreakers. A variant that eliminates hallucination root cause is preferred over one that merely reduces it.

**Why selected:** Eliminates the Read-tool round-trip entirely: model receives image/PDF bytes directly via Anthropic's native vision and document pipelines. SDK bundle analysis (sdk.mjs streamInput) confirms endInput() is called after async generator exhausts, preserving single-turn semantics. Drops poppler-utils dependency.


**Invariants:**
- Telegram two-way messaging remains operational
- Subscription billing not replaced by API token billing
- No new system dependencies added

**Post-conditions:**
- [ ] Agent reads PDFs sent via Telegram without invoking Read tool or pdftotext
- [ ] Agent describes images accurately without fabricating content
- [ ] Voice and text flows unchanged (string-prompt path preserved)
- [ ] bun run typecheck passes with zero errors

**Admissibility:**
- NOT: NOT: Do not revert to pdftotext-only path without a new decision
- NOT: NOT: Do not pass image paths as plain strings in prompts

## 3. Rationale

**Counterargument:** V_path_fixup (install poppler + forceful prompt) is lower risk since it does not touch the SDK interface. isSingleUserTurn=false behavior with canUseTool permission-keyboard flow is untested.

**Selected variant weakest link:** canUseTool permission flow during isSingleUserTurn=false — if Claude Code triggers a non-allowlisted tool while processing an image/PDF, the bidirectional streamInput path is activated. This path was confirmed in SDK bundle but has not been exercised in this bot.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| SDK content blocks — inline image/PDF bytes in query prompt | **Selected** | Eliminates the Read-tool round-trip entirely: model recei... |
| V_path_fixup | Rejected | Model still chooses whether to call Read. Does not eliminate hallucination at root cause. |
| V_hybrid | Rejected | Keeps poppler system dep and splits architecture without gaining over V_content_blocks on primary targets. |

**Evidence requirements:**
- 3+ diverse PDFs processed successfully
- 5+ test images processed without hallucination including 1 blurry image
- Voice and text flows verified unchanged

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| PDF sent via Telegram is summarized correctly | Agent returns document summary without File does not exist error | 100% on first 3 test PDFs |
| Image sent via Telegram described accurately without hallucination | Agent describes visible content correctly; asks to clarify when blurry | No fabricated descriptions across 5 test images |
| Permission-keyboard flow works during content-block sessions | Non-allowlisted tool during image processing triggers Telegram keyboard; Allow/Deny resolves correctly | At least 1 successful permission round-trip verified |

## 4. Consequences

**Rollback plan:**
Triggers:
- Agent returns errors processing PDFs or images after 3+ consecutive attempts
- Permission keyboard breaks during content-block sessions
Steps:
1. Revert session.ts sendMessageStreaming to string-only
2. Revert photo.ts to path-string prompt
3. Revert document.ts to pdftotext path + install poppler-utils
4. Restart claude-telegram-bot.service
Blast radius: src/session.ts, src/handlers/photo.ts, src/handlers/document.ts on VM

**Refresh triggers:**
- Claude Agent SDK breaking change to AsyncIterable streaming input
- Anthropic API removes PDF document content block support
- Hallucination reports resurface after deployment

**Affected files:** /home/assistant/claude-telegram-bot/src/session.ts, /home/assistant/claude-telegram-bot/src/handlers/photo.ts, /home/assistant/claude-telegram-bot/src/handlers/document.ts, /home/assistant/claude-telegram-bot/src/config.ts
