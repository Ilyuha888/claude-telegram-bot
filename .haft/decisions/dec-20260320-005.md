# V2 — Manifest-only staging + PR on approval

## 1. Problem Frame

**Signal:** ADR-006 specifies a 6-state + 4-transition Git approval state machine for vault changes. This was designed defensively but the platform has exactly one user with no concurrent editors and no merge conflicts possible. The complexity cost (implementation, testing, debugging) is paid upfront against a risk that does not exist at MVP scale.

**Constraints:**
- Must preserve the review-before-commit safeguard (no silent vault mutations)
- Must not require shell/subprocess for Git operations (ADR-005)
- Single-user constraint must remain explicit

## 2. Decision

**Selected:** V2 — Manifest-only staging + PR on approval

Eliminates the pre-approval Dulwich worktree dependency (resolving the Dulwich worktree risk flagged in prob-003), reduces state count from 10 to 6, and cuts implementation surface by ~40%. Conflict detection via replay on latest base is preserved. PR creation kept as a stepping stone. The single unverified assumption (Dulwich in-memory commit without a checkout) is low-risk and verifiable in a short spike before implementation.

**Invariants:**
- review-before-commit: no vault mutation may be committed without explicit user approval
- no silent conflict resolution: if replay on latest base diverges from the reviewed diff, the flow stops and the review request becomes conflicted
- no shell execution: all Git operations go through Dulwich in-process (ADR-005)
- change_manifest in Postgres is the source of truth for in-flight review requests, not the filesystem

**Pre-conditions:**
- [ ] change_manifest and proposed file content blobs stored in Postgres before approval is solicited
- [ ] base_commit recorded at staging time
- [ ] Dulwich in-memory commit path verified via spike (ObjectStore + index manipulation without working tree checkout)

**Post-conditions:**
- [ ] review branch pushed to origin under assistant/review/<workspace_slug>/<review_request_id>-<theme_slug>
- [ ] PR created via forge HTTP adapter
- [ ] local staging data and any temporary objects cleaned up after terminal state
- [ ] review_request status transitions to pr_created or conflicted or failed_recoverable

**Admissibility:**
- NOT: no pre-approval filesystem worktree — all pre-approval state lives in Postgres
- NOT: no auto-rebase of a conflicted review request — must regenerate
- NOT: no direct push to main — all approved changes land via a review branch and PR
- NOT: no storing large binary blobs in the change_manifest without a size cap policy

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| V2 — Manifest-only staging + PR on approval | **Selected** | Eliminates the pre-approval Dulwich worktree dependency (... |
| V1 — Keep ADR-006 as-is (worktrees + PR) | Rejected | Dominated by V2 on dulwich_risk and implementation_surface with no compensating advantage. Dulwich worktree support is unverified and would require a spike regardless; better to design around that dependency. |
| V3 — Manifest-only staging + direct push to main (no PR) | Rejected | Saves ~150 LOC but sacrifices PR as a secondary review surface and stepping stone. For vault changes that span multiple files, the Telegram diff is the only review gate. The marginal implementation cost of V2 over V3 is justified by the safety and auditability benefit of the PR. |

**Weakest link:** Dulwich in-memory commit without a working tree checkout — the entire V2 model depends on this being viable. Must be verified before implementation begins.

## 4. Consequences

**Rollback plan:**
Triggers:
- Dulwich in-memory commit spike proves infeasible
- PR creation forge adapter proves too costly for MVP
Steps:
1. If Dulwich in-memory commit fails: introduce a minimal short-lived worktree checked out only at commit time (seconds, not review duration) — same manifest-in-Postgres model, worktree lifetime collapses to the commit operation
2. If PR creation is too costly: fall back to V3 (direct push to main) — remove forge adapter, remove branch_pushed and pr_created states, no other structural change needed
Blast radius: Vault write path and Change Management bounded context only

**Refresh triggers:**
- Dulwich major version upgrade that changes ObjectStore or index API
- vault repository moves to a forge that does not support PR creation via HTTP API
- multi-user support is unlocked (supersedes dec-20260320-001) — concurrent staging conflicts become possible and the state machine may need expansion
