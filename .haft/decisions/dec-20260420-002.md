# buildMessageContext helper: prepend [Forwarded]/[Replying to]/[Quoting] lines before user text

## 1. Problem Frame

**Signal:** Source audit of the live linuz90 bot (note-20260420-003..006, evid-20260420-428532000) confirmed all 5 delivery scenarios are *safe* but three of them are *UX-incomplete*: forwarded messages lose their source attribution, reply-to drops the original message that gives the reply meaning, and Bot-API 7.0 `message.quote` fragments never reach Claude. The handler reads only `ctx.message.text` / `ctx.message.caption` — all provenance metadata (`forward_origin`, `reply_to_message`, `quote`) is silently discarded. User policy calls: (1) forward → include source + content; (2) edit → keep silent-drop; (3) rapid succession → keep sequentialize; (4) reply-to → include source (original message); (5) quote → include quoted fragment.

**Constraints:**
- No code changes to edit-handling (scenario 2) or sequentialize middleware (scenario 3)
- Reply-to payload must be truncated to bound context size (budget: ~500 chars of original text)
- No new handlers — enrich inside existing text.ts and media handlers only
- Must not break existing plain-text message path — enrichment only fires when metadata present

**Acceptance:** Handler enriches user text with forward_origin source line (when present), reply_to_message truncated snippet (when present), and message.quote fragment (when present), before passing to session.sendMessageStreaming. Verified by: sending a forward, a reply, and a quoted message from Telegram, reading journald to confirm the enriched string reaches the query() call. Edit and rapid-succession behavior unchanged.

## 2. Decision

**Selected:** buildMessageContext helper: prepend [Forwarded]/[Replying to]/[Quoting] lines before user text

**Selection policy:** User pre-declared five policy calls; agent role is to bind them to the minimum-blast implementation. Prefer one helper function over five scattered edits. Keep the enriched prompt human-readable so Claude's existing context parsing just works — no new SDK contract.

**Why selected:** Enrichment is the only open gap; the existing text handler already takes a string, so we wrap the string. A single buildMessageContext(ctx) helper produces a multi-line string: "[Forwarded from @name on date]\n[Replying to: \"snippet\"]\n[Quoting: \"fragment\"]\nuser text". All three enrichments are optional — helper emits only the sections whose metadata is present. Changes remain inside the bot; Claude side needs no awareness.


**Invariants:**
- ctx.message.text plain path (no metadata) is byte-identical to current behavior
- Edit handling unchanged — no bot.on(edited_message) added
- sequentialize middleware untouched
- No reply_to / quote / forward data flows to Claude unless the user-owned message explicitly carries it

**Pre-conditions:**
- [ ] Confirmed grammY Context types include forward_origin, reply_to_message, quote (Bot API 7.0 since grammY >= 1.20)
- [ ] Access to VM for code edits as user assistant

**Post-conditions:**
- [ ] buildMessageContext helper tested manually with each of forward / reply / quote / combined / none
- [ ] Journald shows enriched prompt for each scenario at least once
- [ ] No regression in plain-text path
- [ ] tsc --noEmit passes

**Admissibility:**
- NOT: NOT: Do NOT reintroduce the tg-in idempotency spec
- NOT: NOT: Do NOT include reply_to_message bodies longer than 500 chars verbatim — truncate with ellipsis
- NOT: NOT: Do NOT process edited_message updates — edit policy (note-20260420-006) stands
- NOT: NOT: Do NOT include quote.position or quote.entities in v1 — plain text fragment only

## 3. Rationale

**Counterargument:** Prepending bracketed "[Forwarded from ...]" / "[Replying to: ...]" / "[Quoting: ...]" lines to user text mixes instruction with provenance in a way a prompt-injection-aware Claude could still confuse. A safer design would pass the source metadata in a structured side-channel (e.g., a separate SDKUserMessage content block with an explicit "context" role), but the Claude Agent SDK does not expose a distinct role for untrusted quoted context. The plain-prepend approach is therefore a pragmatic compromise, not a security-strong separation.

**Selected variant weakest link:** forward_origin union-type handling: Bot API 7.0+ defines four discriminated variants (user, hidden_user, chat, channel). Getting the type narrowing wrong on one variant silently drops that source line. Mitigation: TypeScript discriminated union switch with exhaustiveness check.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| buildMessageContext helper: prepend [Forwarded]/[Replying to]/[Quoting] lines before user text | **Selected** | Enrichment is the only open gap; the existing text handle... |
| Per-scenario inline patches in text.ts | Rejected | Three separate if-blocks duplicated across 5 media handlers — rots faster, scatters the context contract. |
| Pass full ctx.message object to Claude as JSON | Rejected | Leaks irrelevant fields (message_id, entities, etc.), bloats prompt, couples Claude to Telegram schema. |
| Add edited_message handler with UX notice | Rejected | User explicitly chose 'keep the way it is' for edits. Out of scope. |

**Evidence requirements:**
- Each of 3 enrichments (forward, reply, quote) observed in journald on live bot
- Plain-text path confirmed unchanged via at least one trivial message after deploy
- tsc --noEmit clean

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| Enriched prompt fits within ~1.5x plain-text prompt size on typical reply-to | Journald prompt length on a reply-to flow | Enriched prompt <= original_text_length + 600 chars added overhead |
| Claude correctly attributes forwarded content when asked about source | Ask bot 'who sent this?' after a forward | Answer references the forwarded sender in 2 of 2 test forwards |
| Plain text path unchanged | journald prompt for message without forward/reply/quote | Prompt equals datePrefix + raw text (first message) or raw text (resumed) |

## 4. Consequences

**Rollback plan:**
Triggers:
- Claude misinterprets the source-line convention and executes a quoted fragment as a command
- Prompt-size inflation blows past Claude context window on ordinary replies
Steps:
1. Revert src/handlers/text.ts to read ctx.message.text directly
2. Leave the buildMessageContext helper in place for future iteration
3. Restart claude-telegram-bot service
Blast radius: Single file (text.ts) if media handlers kept untouched.

**Refresh triggers:**
- grammY upgrades its Context types and forward_origin variants change
- Telegram deprecates forward_origin or adds a new discriminant
- Prompt size from reply-to consistently blows past 2x budget in production logs

**Affected files:** /home/assistant/claude-telegram-bot/src/handlers/text.ts, /home/assistant/claude-telegram-bot/src/utils.ts
