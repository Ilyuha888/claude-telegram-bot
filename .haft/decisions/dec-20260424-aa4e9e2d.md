# V_inprocess — in-process node-cron scheduler + notification system

## 1. Problem Frame

## 2. Decision

**Selected:** V_inprocess — in-process node-cron scheduler + notification system

**Selection policy:** Prefer V_inprocess: only Pareto-front variant combining auto missed-fire recovery with medium effort. Extended with Path A delivery (ephemeral session, no context pollution), soft notification-first UX, and notifications.json persistence. V_cron_cli eliminated on dialogue-aware delivery. V_gcal eliminated on effort and OAuth fragility.

**Why selected:** node-cron runs inside the Bun/grammY process giving direct access to session.isActive, the atomic-write pattern from store.ts, and bot.api for delivery. The notification system (soft delivery + persistent store + Mode-2 Notifications tab) is only clean in-process: [Show]/[Delete]/[Remind later] callbacks need the grammY callback-query router and notifications-store. Path A (ephemeral scheduler session, no SESSION_FILE write) keeps the user's working context clean.


**Invariants:**
- Scheduler fires MUST NOT inject into the user's active ClaudeSession history (Path A — ephemeral session only)
- Ephemeral scheduler session MUST NOT write to SESSION_FILE (/tmp/claude-telegram-session.json)
- notifications.json MUST be written atomically (write-tmp-rename, same pattern as store.ts)
- Full content MUST only be shown on [Show] tap — soft notification sent first always
- [Remind later] MUST create a finite one-shot re-fire (1h); MUST NOT loop indefinitely
- Scheduler MUST NOT send any message while session.isActive=true — wait up to 60s then send soft notification regardless

**Pre-conditions:**
- [ ] node-cron is Bun-compatible (pure JS, no native bindings — confirmed by dry-run)
- [ ] Bot has write access to /home/assistant/bot-data/ (verified 2026-04-23 per dec-20260422-003)
- [ ] TELEGRAM_ALLOWED_USERS env provides the chatId for proactive delivery
- [ ] schedules.json created with 4 routine entries before scheduler init

**Post-conditions:**
- [ ] Daily 09:00 Moscow soft notification delivered within ≤1 min on healthy VM
- [ ] Missed fire on reboot caught and delivered within 5 min of bot restart
- [ ] [Show] tap delivers full digest as new Telegram message
- [ ] [Delete] removes from notifications.json, edits original message
- [ ] [Remind later] creates one-shot 1h re-fire in schedules.json
- [ ] Mode-2 /menu shows Notifications (N) badge when N > 0
- [ ] Ephemeral session produces no sessions.json entry and no SESSION_FILE write

**Admissibility:**
- NOT: NOT: scheduler session MUST NOT call saveSession() or write SESSION_FILE
- NOT: NOT: [Remind later] MUST NOT create a recurring entry — one-shot only
- NOT: NOT: full content MUST NOT be sent without user [Show] tap
- NOT: NOT: subscription billing only — no pay-per-token API key

## 3. Rationale

**Counterargument:** V_cron_cli remains the most operationally resilient choice. It runs entirely outside the bot process: a VM cron job survives bot crashes, OOM kills, and deployment restarts without any catch-up logic. The notification store and Mode-2 UI add significant implementation surface (three new files, new callback namespace, menu revision) that increases the bot's complexity and failure surface. If the bot process itself becomes unstable (see prob-20260420-008 — 15+ restarts/day at peak), in-process scheduling inherits that instability. The counterargument is strongest if bot stability remains unresolved: a scheduler that co-dies with the bot is worse than a cron that doesn't.

**Selected variant weakest link:** Bot-crash correlation at exact fire time — if bot crashes at 08:59:50 and restarts at 09:00:10, atomic write of last_fired before crash determines whether catch-up fires. Residual risk: ≤10s window per daily fire where last_fired write may be incomplete.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| V_inprocess — in-process node-cron scheduler + notification system | **Selected** | node-cron runs inside the Bun/grammY process giving direc... |
| V_cron_cli | Rejected | Cannot access session.isActive or implement notification callbacks from a shell script without IPC. Would require reinventing in-process logic as glue. No path to the notification store or Mode-2 Notifications tab. |
| V_cron_rc | Rejected | Dominated by V_cron_cli on effort and coupling; same missed-fire weakness. Eliminated in Pareto. |
| V_gcal | Rejected | Highest effort, OAuth fragility, ~1-5 min precision vs ≤1 min target. All cadences are fixed — calendar-native management adds no value for fixed schedules. |

**Evidence requirements:**
- 7-day log of daily 09:00 fire timestamps
- One simulated reboot catch-up test with log evidence
- Concurrent-tap corruption test on notifications.json
- journalctl confirming ephemeral session produces no SESSION_FILE write

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| Daily 09:00 fire precision ≤1 min on healthy VM | Log timestamp of each scheduler fire vs Moscow wall clock | ≤1 min deviation on 7 consecutive daily fires |
| Missed-fire catch-up works on reboot | Simulate reboot at 08:58, restart bot at 09:05 | Soft notification delivered within 5 min of bot startup |
| notifications.json not corrupted under concurrent taps | Simultaneous [Show] + [Delete] on same notification | Zero corruption events in 20 rapid-tap test |

## 4. Consequences

**Rollback plan:**
Triggers:
- Duplicate notifications from catch-up logic bug
- notifications.json corruption despite atomic writes
- node-cron incompatibility with Bun update
Steps:
1. Set enabled=false on all schedules in schedules.json — fires stop immediately, no code change
2. Remove Notifications button from mode2/menu.ts and callback registrations from index.ts
3. Delete notifications.json and schedules.json
Blast radius: Bot process and bot-data/ only. Sessions, vault, working sessions untouched.

**Refresh triggers:**
- node-cron breaks on a Bun runtime update
- Notification count exceeds 50 unread — consider auto-expiry
- User requests schedule management from Telegram (/schedule command)
- Bot instability (prob-20260420-008) persists — revisit V_cron_cli as fallback

**Affected files:** src/scheduler.ts, src/session.ts, src/mode2/notifications-store.ts, src/handlers/mode2/notifications.ts, src/handlers/mode2/menu.ts, src/index.ts, /home/assistant/bot-data/schedules.json, /home/assistant/bot-data/notifications.json
