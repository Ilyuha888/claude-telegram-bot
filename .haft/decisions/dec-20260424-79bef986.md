# buildMessageContext hint — surface /tmp paths in user turn

## 1. Problem Frame

**Signal:** When a user sends a photo/document via Telegram, the bot downloads the file to /tmp/telegram-bot/ and inlines its bytes as an SDK content block (dec-20260420-001) so the agent can see it for the current turn. But the agent cannot copy the file INTO the vault: the inbound path is ephemeral, /tmp/telegram-bot/ is outside ALLOWED_PATHS, and no handler persists the bytes alongside a note's files/ directory. Concrete recent miss: user wanted BLS visa payment receipt photos attached to 2026-06-06-check-portugal-visa-status.md — had to manually transcribe the data instead. Distinct from prob-20260422-001 (outbound agent→user) and dec-20260420-001 (inbound-ephemeral); this is inbound persistence. Task spec: Agent_Obsidian_Vault/tasks/2026-04-28-bot-image-file-attachment.md.

**Constraints:**
- Path validation must hold — every write path checked by isPathAllowed() from dec-20260420-003
- Subscription billing untouched (dec-20260416-003 admissibility; dec-20260414-001)
- No new egress — download via existing api.telegram.org allowlist path (dec-20260416-001)
- Credential-protection posture untouched — settings.local.json chmod 600, .env, guard-sensitive-reads.py scope all preserved
- No shell escape / raw subprocess in the runtime LLM path (AGENTS.md §Safety Rules)
- Denylist model preserved — no relaxation of BASH_DENY_RE from dec-20260420-003
- Writable surface scoped — only the vault's files/ subdirectories become writable, not new top-level ALLOWED_PATHS
- Single-user scope (dec-20260320-001) — no multi-recipient fan-out
- Commit-confirm flow for User_Obsidian_Vault writes stays intact — large attachment commits go through the same gate
- Content scope v1: photos (sendPhoto/Telegram photo type) and documents (sendDocument) — audio/video deferred

**Acceptance:** User sends one or more photos/documents via Telegram with an instruction naming a target note (e.g., "attach these receipts to the visa status note"). Bot (a) downloads each file from Telegram to a vault-writable path, (b) places it under the target note's files/ sibling directory using a deterministic stable filename, (c) updates the target note with a markdown image reference or wikilink to each attached file, (d) commits the vault change through the existing commit-confirm flow, (e) confirms in Telegram with the list of saved paths. Multi-file messages (Telegram media groups) handled as a single batch. Zero regressions in inbound content-block flow, text path, or voice path.

## 2. Decision

**Selected:** buildMessageContext hint — surface /tmp paths in user turn

**Selection policy:** Minimum new code: prefer the option that requires no new infrastructure and reuses the agent's existing approved toolset (Read/Write/Bash) over the option that introduces a new MCP subprocess.

**Why selected:** V1 requires zero new infrastructure. The agent already has Read/Write/Edit/Bash auto-approved for vault paths, so surfacing the /tmp path in the turn text is sufficient for the full persistence flow. Vision still works (hint is additive, not a replacement for inline content blocks).


**Invariants:**
- Attachment hint is additive — inline content blocks (vision) must still be present
- Hint only surfaces paths for files that actually landed on disk
- No new MCP tool or subprocess for inbound persistence

**Admissibility:**
- NOT: Vision must still work after attachment hint is prepended
- NOT: Hint must not appear for voice messages or plain text

## 3. Rationale

**Counterargument:** Persistence quality (filename choice, wikilink placement, commit message) depends on agent judgment, not code enforcement. A poorly-instructed agent could save to the wrong location or use a bad filename.

**Selected variant weakest link:** CLAUDE.md contract quality — if the agent ignores or misreads the attachment-persistence instructions, the flow breaks silently.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| buildMessageContext hint — surface /tmp paths in user turn | **Selected** | V1 requires zero new infrastructure. The agent already ha... |
| V2 — dedicated save_attachment_to_vault MCP tool | Rejected | The hardest part of the problem — resolving the user's natural language target note — still requires agent judgment. A new tool doesn't eliminate that gap and adds subprocess surface area to secure. |

**Evidence requirements:**
- Single photo with natural-language note target: files/ dir created, wikilink added, commit landed
- Agent resolves informal note reference without being given the full path
- Inline vision still works after attachment (B.4 regression)

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| Successful vault-attach flows | files/ directory created next to target note, wikilink in note, vault git commit | >= 3 in 2 weeks |
| Natural language note resolution | agent resolves target note from informal description without path hint | >= 2 of 3 attempts |
| No regression in text/voice/inline-bytes paths | plain text and voice messages handled normally, vision still works | = 0 regressions observed |

## 4. Consequences

**Rollback plan:**
Triggers:
- Agent systematically saves to wrong location
- Hint causes context overflow on large media groups
Steps:
1. Remove attachments parameter from buildMessageContext calls in photo.ts and document.ts
2. Revert PendingMediaGroup captionCtx back to eager caption string in types.ts + media-group.ts
3. Restart bot
Blast radius: Inbound persistence only — outbound send_file and inline vision unaffected

**Refresh triggers:**
- CLAUDE.md contract proves insufficient after 5+ real uses
- Agent SDK changes how user-turn text are structured
- Vault structure changes (files/ convention deprecated)

**Affected files:** src/utils.ts, src/handlers/photo.ts, src/handlers/document.ts, src/handlers/media-group.ts, src/types.ts, CLAUDE.md
