# Static allowlist in canUseTool

## 1. Problem Frame

**Signal:** Telegram bot's canUseTool callback routes every tool invocation (Write, Edit, Bash, Read, MultiEdit) to a Telegram Allow/Deny prompt. No allowlist is implemented. Multi-file operations (vault migrations, bulk renames, refactors) generate one prompt per file/operation — a 100-file migration produces ~100 sequential taps. User flow breaks. An earlier dec-20260416-003 measurement described Write/Edit/MultiEdit auto-approval and expanded Bash patterns as "implemented", but the current bot code has no such logic — the claim was never committed.

**Constraints:**
- Security invariants of dec-20260416-001 must hold: reversibility gate (commit/push/destructive rm always gated per-op), no secrets in repo, non-root service
- ALLOWED_PATHS path restriction in security.ts must remain enforced — auto-approval of a tool does not bypass the path check
- Irreversible operations (git push, rm -rf, destructive delete) must NEVER be auto-approved
- No persistent grant state that silently persists across sessions without user-visible audit trail

**Acceptance:** Measured on a ≥50-file vault migration replay: total Telegram prompts ≤ 2. Audit log shows every auto-approved tool call. Zero destructive operations auto-approved over 2-week observation window.

## 2. Decision

**Selected:** Static allowlist in canUseTool

**Selection policy:** Maximize prompt reduction per unit of implementation effort, constrained by destructive_op_safety=3. Prefer stepping-stone variants when effort is proportionate.

**Why selected:** Only non-dominated variant. Achieves ≤2 prompts per 50-file batch at ~30 lines with no new state management. Safety=3 preserved via explicit deny for destructive patterns. Stepping stone for V_batch_mode.


**Invariants:**
- isPathAllowed() called for every auto-approved tool — path restriction never bypassed
- rm, git push, git commit always route to Telegram prompt
- Every auto-approved call logged with toolName + input to audit log

**Pre-conditions:**
- [ ] canUseTool defined and active in src/session.ts
- [ ] isPathAllowed() in security.ts correct and tested
- [ ] Audit log path writable by assistant user

**Post-conditions:**
- [ ] Write(*)/Edit(*)/MultiEdit(*) auto-approved when path within ALLOWED_PATHS
- [ ] Bash(mv|mkdir|cp|find|touch) auto-approved when path within ALLOWED_PATHS
- [ ] rm, git push, git commit still route to Telegram prompt
- [ ] Every auto-approved call in audit log with timestamp+toolName+input
- [ ] tsc --noEmit passes

**Admissibility:**
- NOT: NOT: auto-approve rm under any condition
- NOT: NOT: auto-approve git push or git commit
- NOT: NOT: skip isPathAllowed() for auto-approved tools
- NOT: NOT: remove AskUserQuestion interception

## 3. Rationale

**Counterargument:** Static patterns are brittle — Bash ops outside the allowlist still prompt. Pattern maintenance becomes a tax as workflows grow.

**Selected variant weakest link:** Pattern coverage: Bash commands outside the allowlist still route to Telegram prompt. Gaps require incremental additions post-deployment.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| Static allowlist in canUseTool | **Selected** | Only non-dominated variant. Achieves ≤2 prompts per 50-... |
| V_batch_mode | Rejected | Equal outcomes but 5x effort. Scope-bounded UX is a future upgrade. |
| V_sdk_permissions | Rejected | Constraint-eliminated: SDK safety unverified (safety=2), loses AskUserQuestion interception. |

**Evidence requirements:**
- Run 50-file migration replay, count Telegram prompts
- Grep audit log for rm/push/commit auto-approvals after 2 weeks
- tsc --noEmit passes

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| 50-file migration produces ≤2 Telegram prompts | Count permission keyboard messages during migration replay | ≤2 total |
| Zero destructive ops auto-approved over 2-week window | Audit log rm/git push/commit entries show behavior=prompt | 0 auto-approvals on destructive ops |

## 4. Consequences

**Rollback plan:**
Triggers:
- Auto-approved tool call outside ALLOWED_PATHS in audit log
- Destructive op auto-approved
Steps:
1. Revert allowlist if-block in canUseTool in src/session.ts
2. Restart claude-telegram-bot.service
3. Verify bot prompts for Write/Edit again
Blast radius: Bot-local: src/session.ts only.

**Refresh triggers:**
- Agent SDK ships native tool permission model
- New Bash pattern consistently prompts during normal use
- V_batch_mode use case emerges

**Affected files:** src/session.ts, src/handlers/permission.ts
