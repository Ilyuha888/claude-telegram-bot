# Denylist-only permission model in canUseTool

## 1. Problem Frame

**Signal:** Telegram bot's canUseTool callback routes every tool invocation to a Telegram Allow/Deny prompt. Multi-file ops (vault migrations, bulk renames) generate one prompt per file — a 100-file migration produces ~100 taps. Subagents (spawned via the Agent tool) share the same canUseTool path but can't tap the keyboard — their tool calls time out after 15 min and deny silently. Subagents misreport this as "sandbox blocks subprocess." An allowlist-first approach (static BASH_AUTO_RE) was tried but required constant pattern expansion as new Bash invocations appeared.

**Constraints:**
- Security invariants of dec-20260416-001 must hold: non-root service, no secrets in repo
- git integrity is the primary safety surface — vault has a post-commit auto-push hook, so git push/commit/reset/rebase/clean must remain gated
- Local filesystem changes are recoverable from git — rm and arbitrary Bash on tracked files are acceptable
- AskUserQuestion interception must remain intact
- No persistent grant state without user-visible audit trail

**Acceptance:** ≥50-file vault migration produces 0 Telegram prompts for Bash. git push/commit/reset/rebase/clean still route to Telegram keyboard. tsc --noEmit passes.

## 2. Decision

**Selected:** Denylist-only — auto-approve all Bash except git remote/history ops and sudo

**Why selected:** Allowlist maintenance is a tax that grows with every new workflow. Root insight: the only non-recoverable surface is the git remote (vault auto-pushes on commit). Local filesystem changes — including rm — are reversible via git. Flipping to denylist eliminates the maintenance burden and solves the subagent stall class entirely.

**Invariants:**
- `git push|commit|reset|rebase|clean` always route to Telegram prompt — vault remote is protected
- `sudo` always routes to Telegram prompt — no privilege escalation
- `isPathAllowed()` still gates Write/Edit/MultiEdit/Read — path restriction unchanged
- AskUserQuestion interception unchanged

**Pre-conditions:**
- [x] canUseTool defined and active in src/session.ts
- [x] isPathAllowed() in security.ts correct

**Post-conditions:**
- [x] All Bash auto-approved except BASH_DENY_RE matches
- [x] Write(*)/Edit(*)/MultiEdit(*)/Read auto-approved when path within ALLOWED_PATHS
- [x] git push/commit/reset/rebase/clean + sudo still route to Telegram prompt
- [x] tsc --noEmit passes

**Admissibility:**
- NOT: auto-approve git push or git commit under any condition
- NOT: skip isPathAllowed() for Write/Edit/Read auto-approval
- NOT: remove AskUserQuestion interception

## 3. Rationale

**Counterargument:** Denylist is a broad trust grant — any Bash command not explicitly blocked runs without review. A novel destructive pattern not in BASH_DENY_RE would auto-approve.

**Selected variant weakest link:** Deny list completeness. Mitigation: BASH_DENY_RE covers the git remote surface (the only truly non-recoverable ops); local filesystem mistakes are git-reversible.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| Static allowlist in canUseTool | **Selected** | Only non-dominated variant. Achieves ≤2 prompts per 50-... |
| V_batch_mode | Rejected | Equal outcomes but 5x effort. Scope-bounded UX is a future upgrade. |
| V_sdk_permissions | Rejected | Constraint-eliminated: SDK safety unverified (safety=2), loses AskUserQuestion interception. |

**Evidence requirements:**
- Run 50-file migration replay, count Telegram prompts
- Grep audit log for rm/push/commit auto-approvals after 2 weeks
- tsc --noEmit passes

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| 50-file migration produces 0 Telegram prompts for Bash | Count keyboard messages during migration replay | 0 Bash prompts |
| git push/commit still gated after 2-week window | Observe keyboard on any git push/commit attempt | 100% gated |

## Allowlist Extension (2026-04-20)

Added `git mv` to BASH_AUTO_RE. Rationale: `git mv` is a tracked rename — fully reversible via `git restore --staged` + inverse rename. Does not expose push/commit/destructive paths (deny list unchanged). Extension required because parallel subagents running vault migrations couldn't respond to Telegram keyboard prompts, stalling on any `git mv` call. Commit `ebea71a` on `Ilyuha888/claude-telegram-bot`. tsc --noEmit: 0 errors post-change.

## Allowlist Extension 2 (2026-04-20)

Added `python3`, `sed`, `bash /tmp/`, `cat > /tmp/` to BASH_AUTO_RE.

**Root cause investigation:** canUseTool in the Agent SDK (≥0.1.76) intercepts ALL tool calls — main agent AND subagents — through the same permission flow. Subagents cannot tap the Telegram inline keyboard, so any unmatched Bash call times out after 15 min and denies. Subagents misreport this as "sandbox blocks subprocess." Fixed by expanding the allowlist to cover patterns routinely used in batch migration ops. DENY_BASH_RE unchanged — destructive ops still gate. Commit `cc5ad13`. Layer B (`/batch` toggle via `permissionMode: "acceptEdits"`) deferred to a new haft cycle.

## Permission Model Inversion (2026-04-20)

Dropped BASH_AUTO_RE entirely. Flipped Bash to **denylist-only**: auto-approve all Bash except `sudo` and git remote/history ops (`push|commit|reset|rebase|clean`). Rationale: local filesystem is recoverable from git; the vault's git integrity is the only non-recoverable surface. Blocking `git commit` also guards the vault's post-commit auto-push hook. Commit `f012f76`. tsc --noEmit: 0 errors.
