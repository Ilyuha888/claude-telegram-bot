# linuz90/claude-telegram-bot (Agent SDK, TypeScript/Bun)

## 1. Problem Frame

**Signal:** The official Telegram channel plugin is live and working, but research confirms fundamental limitations that block three of four active UX problems: no message queuing (messages lost when session is closed), no voice transcription, dropped messages under stress, no retry logic, fire-and-forget architecture, no formatting control, single session only. Mature community alternatives (linuz90/claude-telegram-bot 423★, terranc/claude-telegram-bot-bridge 38★, hanxiao/claudecode-telegram 15★) demonstrate the CLI-auth/subscription-compatible bridge pattern with voice, multi-project, streaming, and inline keyboard approval flows. The admissibility constraint in dec-20260414-001 says to use the plugin first and fall back to Agent SDK only if the plugin proves unworkable. Evidence is accumulating that the plugin has a hard ceiling for the target UX quality.

**Constraints:**
- Must use Claude subscription billing (Pro/Max) — not API pay-per-token; ~$10/mo budget for non-Claude API costs (voice transcription etc.)
- dec-20260414-001 admissibility: do not build custom bot unless plugin proves unworkable
- Single-user scope (dec-20260320-001)
- Telegram two-way messaging must remain operational during any transition
- Security posture must improve, not degrade — current setup runs as root

**Acceptance:** A decision record with evidence from either: (a) measured plugin failure rate over 2 weeks of real use, OR (b) a working prototype of the chosen alternative demonstrating the claimed improvements.

## 2. Decision

**Selected:** linuz90/claude-telegram-bot (Agent SDK, TypeScript/Bun)

**Selection policy:** Maximize UX quality and message reliability for single-user personal assistant. Subscription billing is a hard constraint. Implementation effort and community risk are tiebreakers. Stepping-stone value is a secondary preference when variants score similarly on primary dimensions.

**Why selected:** V_linuz90 is the only variant that addresses all four blocking UX problems simultaneously: message queuing (no dropped messages on session restart), voice input via Whisper, MarkdownV2 formatting with streaming, and per-repo multi-session management. It uses the Claude Code SDK's SubprocessCLITransport — the same underlying mechanism as V_terranc — so billing risk is equal across both. It adds a meaningful security layer (intent classification, path validation, rate limiting, audit logging) that no other variant provides. It is the only stepping-stone option: the TypeScript Agent SDK already exposes richer tooling (ask_user MCP → native inline keyboards, send_file) than the Python SDK, and sits on Anthropic's intended programmatic interface boundary. V_terranc's sole advantage — message reliability score 4 vs 3 — is a marginal difference that does not matter for a single-user setup where re-sending a dropped message is always possible.


**Invariants:**
- Telegram two-way messaging remains operational at all times during transition
- Claude subscription billing is not replaced by API pay-per-token billing
- SSH access to VM remains available as admin fallback regardless of bot state
- Vault git repo and commit-confirm flow are unaffected by bridge change

**Pre-conditions:**
- [ ] claude auth login completed on VM — ~/.claude/ credentials present and valid
- [ ] Bun runtime installable on Ubuntu 24.04 (verified: Bun officially supports Linux)
- [ ] OpenAI API key available for Whisper voice transcription (~$0.006/min)
- [ ] Existing Telegram bot token (@ianartov_personal_assistant_bot) available for reuse or new bot registered
- [ ] Telegram channel plugin disabled in /root/.claude/settings.local.json before bot goes live

**Post-conditions:**
- [ ] linuz90 bot running as systemd service, survives reboot
- [ ] Telegram channel plugin removed from Claude Code MCP config
- [ ] Voice input (Whisper) working end-to-end from Telegram voice message
- [ ] MarkdownV2 formatting visible in Telegram responses
- [ ] Message queuing verified: send message while bot is processing, confirm it queues and delivers
- [ ] Multi-session: /new and /resume working for at least the vault repo
- [ ] OpenAI API key cost confirmed within $10/mo budget at actual usage rate
- [ ] Rollback path tested: plugin can be re-enabled within 5 minutes

**Admissibility:**
- NOT: Do NOT switch to ANTHROPIC_API_KEY pay-per-token billing without an explicit new decision
- NOT: Do NOT disable the tmux Claude Code session entirely — keep it available for SSH/admin access and potential future scheduling use
- NOT: Do NOT fork and maintain linuz90 codebase without recording a new decision — if the repo goes stale, evaluate alternatives via /h-explore first

## 3. Rationale

**Counterargument:** The stepping-stone advantage is weaker than framed: both V_linuz90 and V_terranc use Claude Code SDK with SubprocessCLITransport underneath — the same abstraction layer in different languages. V_linuz90's richer tooling (ask_user, send_file) reflects the bot author's implementation quality, not a durable SDK-level moat. V_terranc's Python stack is easier to fork and patch if the solo maintainer goes quiet, and its Python codebase has better operational fit with the existing shell/Python VM environment. The decision ultimately bets on one solo maintainer's TypeScript project over another solo maintainer's Python project — a fragile foundation for either.

**Selected variant weakest link:** Solo maintainer abandonment — linuz90/claude-telegram-bot is a personal project by a single maintainer. If the repo goes stale after a Claude Code breaking change, the VM runs a bot that no longer works and the owner must either fork and patch a TypeScript/Bun codebase or roll back. This risk is more likely to materialize than the OAuth policy risk originally identified.

Lower UX ceiling (no ask_user MCP, no send_file), weaker multi-session (2 vs 3), no security layer, no stepping-stone value. macOS-first codebase with confirmed-partial Ubuntu support adds operational risk. |

| V_plugin (official Telegram channel plugin) | Rejected | Architectural ceiling: closed MCP server with no extension points. Cannot address message reliability, formatting, voice, or multi-session without Anthropic shipping plugin improvements. Retained as explicit rollback target only. |

- At least one /resume session verified working across a VM restart
- No regression in vault read/write operations (commit-confirm flow intact)

## 4. Consequences

**Rollback plan:**
Triggers:
- Bot fails to deliver >5% of messages over any 48-hour window
- SubprocessCLITransport auth breaks and no fix is available within 72 hours
- linuz90 repo goes unmaintained and a Claude Code update breaks the bot with no patch path
Steps:
1. Stop linuz90 systemd service
2. Re-enable Telegram channel plugin in /root/.claude/settings.local.json
3. Restart Claude Code tmux session
4. Verify Telegram two-way messaging restored via plugin
5. File new prob- artifact to re-evaluate bridge options
Blast radius: start-assistant.sh, systemd service unit for linuz90 bot, /root/.claude/settings.local.json

**Refresh triggers:**
- linuz90 repo has no commits for 3+ months while Claude Code has had a breaking release
- CLAUDE_CODE_OAUTH_TOKEN / SubprocessCLITransport auth mechanism changes in a Claude Code release
- Anthropic ships official Telegram channel plugin with queuing, formatting, and voice — re-evaluate V_plugin
- A second community bot reaches feature parity with significantly better maintenance signal (multiple contributors, org backing)

**Affected files:** /root/.claude/settings.local.json, /root/start-assistant.sh, /etc/systemd/system/claude-assistant.service
