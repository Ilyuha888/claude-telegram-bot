# Add [Voice transcript] branch to buildMessageContext — prepend transcription notice when input is voice

## 1. Problem Frame

**Signal:** Telegram bot transcribes voice messages via OpenAI Whisper and passes the resulting text to Claude as a plain user message. No metadata flag indicates the message originated from speech. Agent treats the transcription as carefully-composed text, causing two failure modes: (1) interprets filler words, incomplete sentences, and verbal shortcuts literally; (2) asks for clarification on intent that was clear from speech context. buildMessageContext (dec-20260420-002) already enriches message context for forward/reply/quote provenance — voice transcription is the same pattern but missing. Detailed spec: Agent_Obsidian_Vault/tasks/bot-improvement/2026-04-17-telegram-transcription-handling.md

**Constraints:**
- Voice message path in linuz90/claude-telegram-bot is known: Whisper API call in media handler, result passed to session.sendMessageStreaming
- No new external dependencies — enrichment is text prepend only, same pattern as dec-20260420-002
- Must not affect typed-text messages — conditional on voice origin flag only
- Transcription confidence score should be included if available from Whisper response

**Acceptance:** buildMessageContext prepends a transcription notice when message source is voice (detected by bot, not by Claude). Agent interprets informal/garbled voice input for intent rather than literal wording ≥90% of the time without asking for clarification. Verified by: sending 5 voice messages with deliberate filler words and one STT error; confirming agent responds to intent, not literal text.

## 2. Decision

**Selected:** Add [Voice transcript] branch to buildMessageContext — prepend transcription notice when input is voice

**Selection policy:** Minimum viable enrichment: reuse the existing enrichment site (buildMessageContext) and pattern (text prepend) already chosen in dec-20260420-002. Add only what is necessary — one conditional branch, no new abstraction.

**Why selected:** buildMessageContext (dec-20260420-002) is already the enrichment site for message provenance. Voice transcription is structurally identical: detected at ingest time, prepended before Claude receives text. No new abstraction needed — one additional conditional branch in the same helper. Cost: ~0 tokens overhead per voice message (notice is small). Benefit: agent shifts from literal to intent-mode interpretation for all voice input automatically.


**Invariants:**
- Must fire only when bot confirms Whisper transcription was used (not on typed text)
- Notice wording must instruct intent-interpretation, not assume errors — avoids over-correction
- Must not suppress agent clarification requests when intent is genuinely ambiguous

**Pre-conditions:**
- [ ] dec-20260420-002 buildMessageContext helper exists and is the active enrichment site
- [ ] Bot already invokes Whisper and has a boolean/flag distinguishing voice from typed input

**Post-conditions:**
- [ ] All voice-originated messages include notice before user text
- [ ] Typed messages unchanged

## 3. Rationale

**Counterargument:** If the voice message is unambiguously clear, the notice adds noise. But cost is minimal and the notice instructs 'interpret for intent' not 'assume ambiguity' — clear messages are unaffected.

**Selected variant weakest link:** Relies on the bot accurately flagging voice origin before calling buildMessageContext. If the Whisper path in media.ts doesn't expose an unambiguous boolean to the enrichment call, the branch can't fire without a small refactor to thread the flag through.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| Add [Voice transcript] branch to buildMessageContext — prepend transcription notice when input is voice | **Selected** | buildMessageContext (dec-20260420-002) is already the enr... |
| System prompt instruction (global) | Rejected | Would apply to all messages, not just voice. Risks making Claude over-interpret intentionally ambiguous typed input. Scope too broad. |
| New message envelope field (metadata JSON) | Rejected | Requires structural change to the message passing contract between bot and Claude Code subprocess. Disproportionate for a text prefix. No benefit over simple prepend. |

**Evidence requirements:**
- Before: journalctl log showing voice message reaching Claude as raw transcription text
- After: same scenario showing notice prepended in query() call

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| Agent interprets intent on garbled voice input without asking for clarification | Send 5 test voice messages with filler words and one STT error; count clarification requests | ≤1 clarification request across 5 messages |

## 4. Consequences

**Rollback plan:**
Triggers:
- Agent begins over-interpreting typed messages
- False-positive voice detection fires on typed text
Steps:
1. Remove voice branch from buildMessageContext
2. Restart claude-telegram-bot.service
Blast radius: bot only — no vault or Claude Code config change

**Refresh triggers:**
- Whisper replaced with different STT provider that passes different confidence format
- buildMessageContext refactored into separate module

**Affected files:** claude-telegram-bot/src/handlers/media.ts, claude-telegram-bot/src/text.ts
